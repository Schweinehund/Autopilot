## WINDOWS-CLONE-DEEPNEST-TIMEOUT-01: Chain-Apex Cold-Clone Deep-Nesting Subprocess Cost on Windows (CARRIED FORWARD — WORSE at Phase 88)

**Scope:** Running the chain-apex from a cold fresh-clone on Windows exhibits an O(n²) cascade of cold Node subprocess spawns at the deepest nested guards, approaching the per-subprocess timeout and producing a spurious truncated FAIL. **Carried forward from v1.9; WORSE at Phase 88** because the apex chain is now `[48..87]` (+6 deeper than Phase 82's `[48..81]`).

**Evidence of correctness (not a regression):**
- Windows warm-tree `check-phase-88.mjs` run 1 and run 2: 41 PASS / 1 FAIL / 1 SKIPPED — identical FAIL on V-88-CHAIN-66 (check-phase-66 subprocess output truncated at `[5/28]` boundary).
- `check-phase-66.mjs` standalone on Windows: 28 PASS / 0 FAIL / 0 SKIPPED (exit 0, clean) — confirms NOT a real check-phase-66 failure.
- The apex returns 42 PASS / 0 FAIL / 1 SKIP on Linux GHA (run 28106073384) — no truncation.

**Mitigation adopted (Phase 88, per D-04 MANDATORY):** The authoritative apex count is taken from **Linux GHA** (exclusively authoritative per D-04 binding condition), NOT from a Windows warm-tree or cold-clone apex run. The 6 non-apex validators are taken from fast standalone fresh-clone runs.

**Source:** v1.9 `WINDOWS-CLONE-DEEPNEST-TIMEOUT-01` (Plan 82-03) + `88-03-AUDIT-RESULTS.md` §D-04 Mitigation (apex via Linux GHA; Windows warm-tree exhibits truncation at depth [48..87]).

**Remediation options (v1.11+):** (1) cache-warm clone before the apex run; (2) raise the per-subprocess timeout for cold-clone re-audits; (3) cache chain-validator subprocess results within a single apex invocation to avoid redundant sub-spawns at depth ≥ 2 (the O(n²) mitigation — increasingly attractive as the chain deepens past [48..87]).

**Trigger-to-sweep (v1.11+):** The next terminal re-audit setup (or when chain depth affects warm-tree performance). Note: as the chain grows past `[48..87]`, option (3) becomes the durable fix.

**Estimated effort:** 30min (cache-warm) / 15min (timeout config) / 2-4h (O(n²) subprocess caching).

**Status:** CARRIED FORWARD from v1.9 (still unresolved; chain now deeper [48..87], +6). DEFERRED to v1.11+. Does NOT affect harness correctness or cross-OS parity.
