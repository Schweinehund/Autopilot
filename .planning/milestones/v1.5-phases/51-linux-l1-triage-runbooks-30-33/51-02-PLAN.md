**6. Cause A: Package Install Failure (paste verbatim, includes `> **Say to the user:**` and read-only commands):**
```
## Cause A: Package Install Failure {#cause-a-package-install}

> See [intune-portal (package)](../_glossary-linux.md#intune-portal-package) for the package definition; [/var/log/dpkg.log](../_glossary-linux.md#varlogdpkglog) for the install-event log path; [APT repository](../_glossary-linux.md#apt-repository) for the packages.microsoft.com source.

**Entry condition:** `apt list --installed | grep intune-portal` returns empty OR `dpkg -l intune-portal` shows a status other than `ii` (e.g., `un`, `iU`, `iF`).

### Symptom

- User reports they tried to install Intune on their Linux device but the install did not complete
- `apt list --installed | grep intune-portal` returns empty (no installed entry)
- `dpkg -l intune-portal` shows package in non-`ii` state — typical: `un` (unknown / not installed), `iU` (install pending), `iF` (install failed during configure)

### L1 Triage Steps

1. > **Say to the user:** "Let me check whether the Intune client package is installed on your device. Please open Terminal and type: `apt list --installed | grep intune-portal`. Read me the output."
2. If the output is empty, the package is not installed — ask the user to also run `cat /var/log/dpkg.log | grep intune` (or `tail -50 /var/log/dpkg.log`) to identify what happened on the last install attempt.
3. If the output shows non-`ii` state, ask the user to read me the full line from `dpkg -l intune-portal`.
4. Cross-reference [/var/log/dpkg.log](../_glossary-linux.md#varlogdpkglog) for the install-event log path; LOW-MEDIUM confidence per Phase 49 attestation.
5. Confirm the user followed the official enrollment walkthrough — see [end-user enrollment steps](../end-user-guides/linux-intune-portal-enrollment.md#enroll-your-device).

### Admin Action Required

**Ask the admin to:**

- Reinstall the `intune-portal` package via `sudo apt install intune-portal` after confirming the packages.microsoft.com APT repository is reachable from the device's network.
- If the device is behind a proxy or firewall, verify that `https://packages.microsoft.com` is reachable.

**Verify:**

- After reinstall: `apt list --installed | grep intune-portal` returns a single line `intune-portal/jammy,now 2.0.X amd64 [installed]` (Ubuntu 22.04) or `intune-portal/noble,now 2.0.X amd64 [installed]` (Ubuntu 24.04).

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

### Escalation (within Cause A)

- Reinstall succeeds but enrollment still does not complete (Cause B or C may apply next)
- packages.microsoft.com unreachable from the device's network despite the user being on the corporate network — escalate to Network team for proxy/firewall review

---
```

```
## Cause B: Sign-In Failure (Microsoft Identity Broker) {#cause-b-sign-in-failure}

> See [microsoft-identity-broker](../_glossary-linux.md#microsoft-identity-broker) for the systemd service definition and [Identity Broker](../_glossary-linux.md#identity-broker) for the architectural role; cross-reference [Phase 50 admin pitfall (Identity Broker v2.0.2+ re-enrollment)](../admin-setup-linux/01-intune-linux-agent.md) for the known v2.0.2+ broker upgrade behavior.

**Entry condition:** Microsoft Identity Broker authentication did not complete — user sees sign-in prompt repeatedly, auth dialog times out, or no enrollment record appears in Intune > Devices > All devices despite the user attempting sign-in.

### Symptom

- User sees the Intune Portal sign-in prompt loop, or the sign-in dialog closes without completing
- `intune-portal` deb is installed (`apt list --installed | grep intune-portal` returns the installed line) but the device never appears in Intune
- `journalctl -u microsoft-identity-broker --since "1 hour ago"` shows authentication errors, network failures, or timeouts

### L1 Triage Steps

1. > **Say to the user:** "Let me check the Microsoft Identity Broker authentication log. Please open Terminal and type: `journalctl -u microsoft-identity-broker --since \"1 hour ago\"`. Read me the output."

   If the output is empty or shows "no entries," ask the user to instead run: `sudo journalctl -u microsoft-identity-broker --since "1 hour ago"` — `microsoft-identity-broker` is a system-scope service per [Phase 50 admin agent guide](../admin-setup-linux/01-intune-linux-agent.md), and on some Ubuntu installs the journal entries are root-readable only.

2. > **Say to the user:** "Now please type `journalctl --user | grep -iE \"broker|signin|sign-in\"` and read me any matching lines from the last hour."
3. Look for: `authentication failed`, `network unreachable`, `token request timed out`, or `client request denied` — escalate the literal error string with the packet.
4. Confirm `intune-portal` is still installed: `apt list --installed | grep intune-portal` (read-only — no sudo).
5. Reference [Phase 50 LIN-05 Known admin pitfall](../admin-setup-linux/01-intune-linux-agent.md) — Identity Broker v2.0.2+ upgrades trigger automatic re-enrollment with new device IDs; check whether this device's broker version recently changed.

### Admin Action Required

**Ask the admin to:**

- Restart the system-scope broker service: `sudo systemctl restart microsoft-identity-broker`
- Verify network connectivity to `https://login.microsoftonline.com` from the device
- Check Intune admin center > Devices > Enrollment > Linux: confirm Linux enrollment is enabled and the user is in the assigned group

**Verify:**

- After admin actions: ask the user to retry sign-in via the Intune Portal app. The device should appear in Intune > Devices > All devices within 5 minutes.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

### Escalation (within Cause B)

- Broker restart succeeds but sign-in still fails after retry
- Conditional Access blocking sign-in (route to [Runbook 32](32-linux-ca-blocking-web-access.md) if applicable)
- Multiple users on the same tenant report identical broker auth failure (likely tenant-side issue)

---
```

**8. Cause C: Enrollment Timeout (paste — `intune-agent.timer` is USER-scope, NEVER sudo):**
```
## Cause C: Enrollment Timeout (`intune-agent.timer`) {#cause-c-enrollment-timeout}

> See [intune-agent.timer](../_glossary-linux.md#intune-agenttimer) for the user-scope systemd timer definition; [journalctl](../_glossary-linux.md#journalctl) for log retrieval semantics.

**Entry condition:** `intune-agent.timer` did not check in within the expected window. `systemctl --user list-timers intune-agent.timer` shows `-` for NEXT, or the last activation timestamp is > 1 hour ago.

### Symptom

- `intune-portal` deb is installed AND user sign-in completed AND device shows up in Intune > Devices > All devices, but the device never moves out of "Enrolling" state
- `systemctl --user is-active intune-agent.timer` returns `inactive` or `failed`
- `systemctl --user list-timers intune-agent.timer` shows `-` for the NEXT column

### L1 Triage Steps

1. > **Say to the user:** "Let me check whether the Intune agent timer is running on your device. Please open Terminal and type: `systemctl --user status intune-agent.timer`. Read me the output."
2. > **Say to the user:** "Now please type `systemctl --user list-timers intune-agent.timer` and read me the NEXT column."
3. If NEXT shows `-`, the timer has no scheduled next-run. Ask the user to run `journalctl --user -u intune-agent.timer --since "1 hour ago"` and read me the output. Look for: `Failed to start`, `Unit not found`, or absence of expected check-in entries.
4. > **Say to the user:** "Please type `systemctl --user is-enabled intune-agent.timer` and read me the output."
5. Confirm `intune-portal` deb is installed: `apt list --installed | grep intune-portal` (read-only — no sudo).

### Admin Action Required

**Ask the admin to:**

- Restart the user-scope timer: `systemctl --user start intune-agent.timer` (NO sudo — `--user` units do not take sudo)
- If the timer is in `failed` state: `systemctl --user reset-failed intune-agent.timer` then `systemctl --user start intune-agent.timer`
- If reinstall is needed: `sudo apt install --reinstall intune-portal` (system-scope reinstall — sudo correct here)

**Verify:**

- After admin actions: `systemctl --user is-active intune-agent.timer` returns `active`. Within approximately 30 minutes the device's last-check-in timestamp in Intune admin center updates.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

### Escalation (within Cause C)

- Timer restarts but immediately re-enters `failed` state within minutes
- Timer is `active (waiting)` but no check-in occurs within 60 minutes (route to [Runbook 33: Linux Agent Service Failure](33-linux-agent-service-failure.md) if symptom evolves)
- Reinstall succeeds but timer is still inactive after reinstall

---
```
