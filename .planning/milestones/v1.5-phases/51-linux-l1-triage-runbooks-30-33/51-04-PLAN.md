**3. H1 + intro + cause enumeration:**
```
# Linux CA Blocking Web-App Access

L1 runbook for Linux endpoints (Ubuntu 22.04/24.04 LTS) where Conditional Access (CA) is blocking the user from accessing Microsoft 365 web apps in Microsoft Edge for Linux. Three distinct observable causes are diagnosed independently:

- **Cause A:** Device is not enrolled in Intune (Linux device serial absent from inventory)
- **Cause B:** Device is enrolled but reporting `Not compliant` (CA grant denies access while non-compliant)
- **Cause C:** Device is enrolled + compliant but Edge for Linux is not signed in with the user's work account

Routed here from the [Linux Triage Decision Tree](../decision-trees/09-linux-triage.md) LINR32 branch via the LINCA disambiguation node.
```

```
> ⚠️ **Architecture: Linux is web-app CA only.** Device-level CA (the grant tied to compliance state) is not supported on Linux — only web-app sign-in via Edge for Linux 102.x+ is enforceable. See [Linux Capability Matrix — Conditional Access](../reference/linux-capability-matrix.md#conditional-access) for the full architectural detail.
```

**7. Cause A: Not Enrolled (portal-only; hand-off to Runbook 30):**
```
## Cause A: Not Enrolled {#cause-a-not-enrolled}

> See [MS Edge for Linux](../_glossary-linux.md#ms-edge-for-linux) for browser definition; [Identity Broker](../_glossary-linux.md#identity-broker) for sign-in surface architecture.

**Entry condition:** Device serial does not appear in Intune > Devices > All devices (Linux platform filter). The user is blocked because the device has not enrolled in Intune at all — there is no compliance state to evaluate.

### Symptom

- User reports they are blocked accessing Microsoft 365 web apps in Edge for Linux
- P-INV (Intune > Devices > All devices, Linux platform filter) does NOT show the user's device serial
- Edge browser may show a Conditional Access blocked page citing "device not registered" or "device not compliant"

### L1 Triage Steps

1. Open Intune admin center > Devices > All devices, filter by **Platform = Linux**, search for the user's device serial number.
2. Confirm the device serial does NOT appear in the inventory.
3. > **Say to the user:** "It looks like your device hasn't completed Intune enrollment yet. Have you installed the Intune Portal app and signed in?"
4. If the user reports they have not enrolled, route to [Runbook 30: Linux Enrollment Failed](30-linux-enrollment-failed.md) for the enrollment-side L1 walkthrough.

### Hand-Off

This cause is the **enrollment-failure entry pattern** — primary remediation lives in [Runbook 30](30-linux-enrollment-failed.md). Document the CA-blocked symptom in the ticket and continue investigation in Runbook 30 (Cause A package install / Cause B sign-in failure / Cause C enrollment timeout).

### Escalation (within Cause A)

- User insists they completed enrollment but device serial still absent from P-INV — collect enrollment-attempt timestamp + user UPN and route to Runbook 30 first.

---
```

**9. Cause C: Edge Not Signed In (the WEB-APP-CA-specific failure mode):**
```
## Cause C: Edge Not Signed In {#cause-c-edge-not-signed-in}

> See [web-app CA](../_glossary-linux.md#web-app-ca) for the Linux-specific CA enforcement surface; [MS Edge for Linux](../_glossary-linux.md#ms-edge-for-linux) for browser-side requirements.

**Entry condition:** Device is enrolled (Cause A ruled out) AND device is `Compliant` in P-09 (Cause B ruled out), but Edge for Linux is not signed in with the user's work account. The web-app CA check has no signed-in identity to evaluate, so access is blocked at the browser-identity layer.

### Symptom

- P-INV shows the device serial; P-09 shows compliance state `Compliant`
- User reports Microsoft 365 web apps redirect them to a sign-in page repeatedly, or Edge blocks access citing "no work account signed in"
- User reports they have multiple Edge profiles or use a personal account profile

### L1 Triage Steps

1. > **Say to the user:** "Let me check your Edge browser sign-in. Open Edge for Linux and click your profile icon at the top right. Is your work account signed in (the email matching your organization)?"
2. If the user's profile shows only personal accounts, the work account is not signed into Edge — this is the cause.
3. > **Say to the user:** "Please open Edge Settings > Profiles and confirm whether your work account profile exists. If it does not, click 'Add profile' and sign in with your work email."
4. Confirm Edge for Linux version is 102.x+ (web-app CA requires Edge 102+ per [Phase 50 admin CA guide](../admin-setup-linux/05-conditional-access.md)). User can check via Edge > Settings > About Microsoft Edge.

### Admin Action Required

**Ask the admin to:**

- Verify the user's CA policy assignment in P-CA — the Linux-targeted CA policy should require web-app sign-in but not device-level compliance grant (per [Phase 50 admin CA guide](../admin-setup-linux/05-conditional-access.md) and [Linux Capability Matrix — Conditional Access](../reference/linux-capability-matrix.md#conditional-access)).
- If the assigned CA policy targets a device-level grant, the policy is misconfigured for Linux — only web-app CA is enforceable on Linux.

**Verify:**

- After user signs into Edge with work account: user retries the M365 web app and reaches it without redirect.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

### Escalation (within Cause C)

- User signs into Edge with work account but is still blocked — escalate with full Edge profile + sign-in dialog screenshot
- Multiple users report identical symptom — likely tenant-side CA policy misconfiguration; escalate to identity admin team

---
```
