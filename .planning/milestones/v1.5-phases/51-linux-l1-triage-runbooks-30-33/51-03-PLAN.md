```
# Linux Compliance Non-Compliant

L1 runbook for Linux endpoints (Ubuntu 22.04/24.04 LTS) where compliance evaluation is reporting `Not compliant` in Intune. Four distinct causes are diagnosed independently:

- **Cause A:** Distro version out of supported range (not Ubuntu 22.04 LTS or 24.04 LTS)
- **Cause B:** Disk not encrypted (LUKS / dm-crypt absent)
- **Cause C:** Password policy not met (passwd not set or complexity insufficient)
- **Cause D:** Custom-compliance Bash discovery script reporting non-compliant

Routed here from the [Linux Triage Decision Tree](../decision-trees/09-linux-triage.md) LINR31 branch.

> **Routing discipline (PITFALL-2):** This runbook covers compliance-policy-driven non-compliance only — i.e. the device shows `Not compliant` in Intune > Devices > All devices. If the user is BLOCKED accessing Microsoft 365 in Edge for Linux specifically, that is a web-app CA workflow — route to [Runbook 32: Linux CA Blocking Web-App Access](32-linux-ca-blocking-web-access.md). Linux does NOT support device-level Conditional Access; only web-app CA via Edge for Linux is enforceable.
```

**6. Cause A: Distro Version Out of Range (portal-first; user-side optional):**
```
## Cause A: Distro Version Out of Range {#cause-a-distro-version-out-of-range}

> See [Ubuntu LTS](../_glossary-linux.md#ubuntu-lts), [GA kernel](../_glossary-linux.md#ga-kernel), [HWE kernel](../_glossary-linux.md#hwe-kernel) for supported-distro definitions.

**Entry condition:** P-09 failing-setting shows "Operating system version" or "Allowed distributions." Ubuntu Intune support is scoped to Ubuntu 22.04 LTS and 24.04 LTS only (Ubuntu 20.04 dropped from Intune 2508 in August 2025).

### Symptom

- P-09 (Intune > Devices > All devices > [device] > Device compliance) shows compliance state "Not compliant" with failing-setting label "Operating system version" or "Allowed distributions"
- User running an older Ubuntu (20.04, 18.04) or non-Ubuntu distro (Debian, Mint, Fedora, RHEL) — out of v1.5 supported scope

### L1 Triage Steps

1. > **Say to the user:** "Let me check which Linux distribution and version your device is running. Please open Terminal and type: `lsb_release -a`. Read me the Distributor and Release lines."
2. If output shows distributor != Ubuntu OR release != 22.04 / 24.04, the device is out of supported scope.
3. Optionally ask the user to also run `uname -r` to identify the kernel — distinguishes [GA kernel](../_glossary-linux.md#ga-kernel) vs [HWE kernel](../_glossary-linux.md#hwe-kernel) tracks (admin guidance for Phase 50 LIN-04 Allowed Distributions configuration).
4. Confirm the failing-setting in P-09 — share the full failing-setting label with the admin packet.

### Admin Action Required

**Ask the admin to:**

- Verify the Linux compliance policy P-COMP "Allowed Distributions" setting matches the device's distro+version. If the device is on a supported Ubuntu LTS, the failing-setting should self-resolve at the next compliance evaluation (~30 minutes).
- If the device is on an unsupported distro, the user must either upgrade to Ubuntu 22.04 / 24.04 LTS OR be moved out of the Linux-managed group (per organization policy).

**Verify:**

- After admin actions or user upgrade: P-09 compliance state moves to `Compliant` within ~30 minutes of the next compliance evaluation.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

---
```

**7. Cause B: Disk Not Encrypted (portal-first; informational user-side `lsblk -f`):**
```
## Cause B: Disk Not Encrypted {#cause-b-disk-not-encrypted}

> See [LUKS](../_glossary-linux.md#luks) and [dm-crypt](../_glossary-linux.md#dm-crypt) for the Linux disk-encryption stack used by Ubuntu installer's "Encrypt the new Ubuntu installation" option.

**Entry condition:** P-09 failing-setting shows "Device encryption" or "Encryption of data storage." Ubuntu installer LUKS + dm-crypt encryption was NOT selected during initial install (cannot be retroactively enabled without re-imaging).

### Symptom

- P-09 shows compliance state "Not compliant" with failing-setting label "Device encryption" or "Encryption of data storage"
- User reports they did not enable disk encryption during Ubuntu install

### L1 Triage Steps

1. Confirm the failing-setting in P-09 by sharing the full failing-setting label with the admin packet.
2. > **Say to the user:** "Let me check whether disk encryption is currently active on your device. Please open Terminal and type: `lsblk -f`. Read me the FSTYPE column for your root and home partitions."
3. If FSTYPE for root/home shows `crypto_LUKS`, encryption IS active — the failing-setting may be a stale evaluation; ask the admin to trigger compliance re-evaluation.
4. If FSTYPE shows `ext4` / `btrfs` / `xfs` only (no `crypto_LUKS`), encryption is NOT active.
5. Note: this user-side `lsblk -f` is informational diagnostic only — the authoritative source is the admin-portal compliance evaluation (per D-15 portal-first).

### Admin Action Required

**Ask the admin to:**

- Trigger compliance re-evaluation for the device (Intune > Devices > All devices > [device] > Sync). Allow ~30 minutes.
- If FSTYPE confirms encryption is NOT active: the user must re-install Ubuntu with the "Encrypt the new Ubuntu installation" option selected (Ubuntu LUKS encryption cannot be retroactively enabled without re-imaging).
- If FSTYPE confirms encryption IS active but compliance still fails: escalate to L2 for compliance-evaluation troubleshooting.

**Verify:**

- After admin actions: P-09 compliance state moves to `Compliant` within ~30 minutes of the next compliance evaluation.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

---
```

**8. Cause C: Password Policy Not Met (portal-first; informational user-side `passwd --status`):**
```
## Cause C: Password Policy Not Met {#cause-c-password-policy-not-met}

**Entry condition:** P-09 failing-setting shows "Password" or "Password policy." User has no password set, or the password complexity / expiry / minimum-length does not meet the P-COMP password policy.

### Symptom

- P-09 shows compliance state "Not compliant" with failing-setting label "Password" / "Password policy" / "Password complexity"
- User reports they log in via SSH-key only or a shared/no password account

### L1 Triage Steps

1. Confirm the failing-setting in P-09 — share the full failing-setting label with the admin packet.
2. > **Say to the user:** "Let me check the current password status for your account. Please open Terminal and type: `passwd --status`. Read me the output."
3. The output is one line: `username P 04/27/2026 0 99999 7 -1` where field 2 is the password status — `P` = password set, `NP` = not set, `L` = locked.
4. If field 2 = `NP` or `L`, the user has no usable password; this maps to the P-COMP password policy failure.
5. Note: this user-side `passwd --status` is informational diagnostic only — the authoritative source is the admin-portal compliance evaluation.

### Admin Action Required

**Ask the admin to:**

- Verify the P-COMP password policy settings (minimum length, expiry, complexity).
- If the user has `NP` / `L` status: the user must set a password matching the policy via `passwd` (interactive — user will be prompted for new password).
- Trigger compliance re-evaluation after the user sets the password.

**Verify:**

- After user sets password: `passwd --status` field 2 returns `P`. P-09 compliance state moves to `Compliant` within ~30 minutes of the next compliance evaluation.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

---
```

**9. Cause D: Custom-Compliance Failure (terminal walkthrough — Bash exit codes):**
```
## Cause D: Custom-Compliance Failure {#cause-d-custom-compliance-failure}

> See [Linux compliance settings](../_glossary-linux.md#linux-compliance-settings), [/var/log/intune-update.log](../_glossary-linux.md#varlogintune-updatelog), [journalctl](../_glossary-linux.md#journalctl) for the custom-compliance evaluation surface.

**Entry condition:** P-09 failing-setting shows the custom-compliance Bash script name (matches a script registered in P-CC). The Bash discovery script is reporting non-compliant exit code.

### Symptom

- P-09 shows compliance state "Not compliant" with failing-setting label matching the registered Bash script name
- `/var/log/intune-update.log` may contain non-zero exit codes from the script execution

### L1 Triage Steps

1. > **Say to the user:** "Let me check the Intune update log for any custom-compliance script failures. Please open Terminal and type: `cat /var/log/intune-update.log | tail -50`. Read me any lines mentioning custom compliance or non-zero exit codes."
2. > **Say to the user:** "Now please type `journalctl --user | grep intune-update` and read me any matching lines from the last 24 hours."
3. Identify the script exit code in the log output. A non-zero exit code from the discovery script means the device is reporting non-compliant on that custom-compliance criterion.
4. Cross-reference [/var/log/intune-update.log](../_glossary-linux.md#varlogintune-updatelog) — LOW-MEDIUM confidence per Phase 49 attestation.

### Admin Action Required

**Ask the admin to:**

- Look up the registered Bash discovery script in P-CC (`Endpoint security > Device compliance > [policy] > Custom compliance`) — identify what compliance criterion it checks.
- Determine whether the device's actual state matches the criterion. If the script logic has a defect, the admin updates the script in P-CC.
- For full Bash discovery-script authoring patterns, see [v1.5.1 LIN-DEFER-01 Linux Bash custom-compliance deep-dive] (forthcoming) and [Phase 50 admin compliance policy guide](../admin-setup-linux/03-compliance-policy.md).

**Verify:**

- After admin actions: re-run the script via the next compliance evaluation cycle. P-09 compliance state moves to `Compliant` within ~30 minutes.

**If the admin confirms none of the above applies:**

- Proceed to [Escalation Criteria](#escalation-criteria).

---
```
