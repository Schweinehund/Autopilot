## CHAIN_SKIP {48, 51, 58, 60, 61} Resolution via CI-Linux Job

**Scope:** Pre-existing CRLF + archived-path validator failures documented at `scripts/validation/check-phase-64.mjs:65-73`. All 5 entries remain suppressed-as-justified across `check-phase-62.mjs` through `check-phase-66.mjs` CHAIN_SKIP sets — non-fatal on the Windows host where v1.6 development occurred.

**Root causes** (per `check-phase-64.mjs:65-73`):
- **(a) Archive drift:** `.planning/phases/` partially archived to `.planning/milestones/v1.5-phases/` after v1.5 close. `check-phase-48.mjs` hardcodes the pre-archival path for `48-VERIFICATION-broken-links.md`. Also, `regenerate-supervision-pins.mjs --self-test` fails due to `_glossary-android.md` `+1` line-number shift from Phase 62-06/62-07 banner additions; `v1.5-audit-allowlist.json` needs line-number rebase (tracked as out-of-scope for v1.6 maintenance).
- **(b) Windows CRLF-LF mismatch:**
  - `check-phase-51.mjs` Mermaid regex uses literal `\n` but `docs/decision-trees/09-linux-triage.md` has CRLF line endings on this Windows worktree.
  - `check-phase-58.mjs` frontmatter parse uses `\n` but `docs/reference/4-platform-capability-comparison.md` has CRLF.
- **(c) Cascading failures:** `check-phase-60.mjs` + `check-phase-61.mjs` cascade from (a) + (b) + v1.5 harness C7/C9 line-number mismatch.

**Current state at v1.6 close:** All 5 entries remain in CHAIN_SKIP across `check-phase-62.mjs` → `check-phase-66.mjs`. The Phase 66 terminal re-audit per D-03 runs from a fresh `git clone --no-hardlinks` into `$env:TEMP\v1.6-audit-<rand>` — BUT it remains on the same Windows host, so CRLF behavior is unchanged. CHAIN_SKIP {48, 51, 58, 60, 61} continues to suppress as expected.

**v1.6 Phase 66 advisor finding (new):** `.planning/phases/65-apple-business-l1-l2-hub-navigation-integration/65-VERIFICATION.md:439` Section K originally aspired that "Phase 66's fresh Linux worktree is expected to resolve" the CRLF failures. Phase 66 D-03 re-audit revealed that the user's standing constraint `.planning/config.json:7` `use_worktrees:false` (durable per memory file `project_execphase_sequential.md`) precludes worktree-based execution; the Phase 66 fresh-clone alternative satisfies D-22 INTENT at STRICTER physical isolation but does NOT change the host OS. Therefore the Section K aspirational expectation is amended here to: **"v1.7 CI-Linux job is expected to resolve"** (matching the GitHub Actions runner OS where line endings are LF).

**Trigger-to-sweep:** v1.7 introduces a CI-Linux job (in `audit-harness-v1.7-integrity.yml` or equivalent) that executes `check-phase-{48..66}.mjs` on `ubuntu-latest` runners. When ubuntu-latest output confirms PASS on phases 48, 51, 58, 60, 61, the v1.7 close-gate executor removes the 5 entries from `CHAIN_SKIP` in `check-phase-62..66.mjs` (or, equivalently, marks them resolved in the new v1.7 chain validator).

**Estimated effort:** 1-2h for v1.7 entry-phase planning (CI-Linux job authoring + smoke test + CHAIN_SKIP removal). Resolution of root cause (a) archive drift is independent of CRLF and may surface as a separate v1.7 deliverable.
