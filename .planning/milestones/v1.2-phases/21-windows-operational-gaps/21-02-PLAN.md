**win32-app-packaging.md (WINF-04):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "Win32 App Packaging Best Practices for ESP Reliability"
- Version gate: "This guide covers Win32 app packaging for both APv1 and APv2 deployments. For ESP configuration, see [ESP Policy Configuration](../admin-setup-apv1/03-esp-policy.md)."
- ## Detection Rule Priority Order (section with numbered list and explanation):
  1. **MSI product code** — Deterministic for MSI installs. Intune auto-detects MSI product code during .intunewin upload. Most reliable, no maintenance.
     > **What breaks if misconfigured:** Wrong MSI code = app shows as "not installed" even after successful install. ESP never completes because it waits for a detection that never succeeds.
  2. **File version check** — For non-MSI apps that update in place. Specify: file path, version, and comparison operator (>=).
     > **What breaks if misconfigured:** Using "file exists" instead of "version >=" means old app versions pass detection. Updates silently skip. Use version check for apps that update in-place.
  3. **Registry key/value** — When no reliable file path exists. Look for uninstall registry key under `HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\`.
     > **What breaks if misconfigured:** 32-bit apps write to Wow6432Node. Specifying wrong registry hive = detection fails on 64-bit Windows.
  4. **PowerShell custom script** — Only when above methods are insufficient. Exit code 0 = detected. Performance cost: each script adds 5-15 seconds to detection cycle.
     > **What breaks if misconfigured:** Script returns non-zero unexpectedly (e.g., command not found). App re-installs on every sync check.
- ## ESP Tracking Requirements:
  - Required + device group = Device phase tracking
  - Required + user group = User phase tracking
  - Available assignment = NEVER tracked by ESP, NEVER blocks desktop
  - "Selected" mode: up to 100 apps in blocking list
  - "All" mode: every Required app blocks desktop access
  - Table showing assignment type vs ESP phase
- ## Install Order and Dependencies:
  - Win32 apps support dependency declarations in Intune (App > Dependencies)
  - App A as dependency of App B → App A installs first automatically
  - Warning: circular dependencies fail silently; check IME logs (`C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\IntuneManagementExtension.log`) for conflict errors
  - > **What breaks if sequenced incorrectly:** App B depends on App A's runtime. If App A is not declared as dependency, install order is non-deterministic. App B may install first and fail. ESP retries, eventually times out.
- ## Anti-Pattern Warning: "Do NOT mix Win32 (.intunewin) and LOB (MSI) as Required in the same ESP deployment — TrustedInstaller conflict causes ESP to hang. See [ESP Policy Configuration](../admin-setup-apv1/03-esp-policy.md) for details."

**esp-timeout-tuning.md (WINF-05):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "ESP Timeout Tuning Guide"
- Version gate: "This guide extends the ESP configuration in [ESP Policy Configuration](../admin-setup-apv1/03-esp-policy.md) with scenario-specific timeout recommendations. For ESP setting definitions and 'What breaks' callouts, see that guide first."
- This doc extends, not duplicates, `admin-setup-apv1/03-esp-policy.md`. That file documents what each setting does. This file adds the scenario-based tuning matrix.
- ## Recommended Timeout Values by Scenario:
  | Scenario | Recommended Timeout | Default | Reasoning |
  |----------|---------------------|---------|-----------|
  | Standard Entra join (cloud-only) | 60 minutes | 60 min | Baseline — sufficient for 5-10 Win32 apps |
  | Hybrid Entra join | 100 minutes | 60 min | ODJ connector adds ~40 min to configured timeout; 60 min default often insufficient |
  | Pre-provisioning (technician phase) | 60 minutes | 60 min | Device phase only; user phase happens separately at user sign-in |
  | Large app payload (5+ GB total) | 90-120 minutes | 60 min | Delivery Optimization download time varies by network speed |
  | Windows quality updates enabled during ESP | Add 20-40 minutes to base | N/A | Quality update setting adds to base timeout budget; cumulative with app install time |
  | APv2 (Device Preparation) | N/A — fixed 60 min | 60 min | APv2 does not expose configurable ESP timeout; fixed at 60 minutes |
  Each row has a "What breaks" callout:
  - Hybrid at 60 min: "ODJ connector takes 40+ minutes. Remaining 20 minutes insufficient for app installs. ESP times out. Device appears 'not configured' to user."
  - Quality updates enabled: "Timeout budget consumed by update download + install. No time remaining for app installs. ESP times out after update completes."
- ## Timeout Calculation Formula:
  Recommended timeout = (Hybrid ODJ overhead: 0 or 40 min) + (Estimated app install time) + (Quality update buffer: 0 or 20-40 min) + (15 min safety margin)
- ## Misconfiguration Consequences table (extends the 03-esp-policy.md table):
  | Timeout Setting | Too Low | Too High | Sweet Spot |
  |----------------|---------|----------|------------|
  | Device phase timeout | Apps don't finish; ESP fails | User waits unnecessarily | Measured app install time + 15 min buffer |
  | User phase timeout | User apps don't install | User stares at ESP screen | Measured user app time + 10 min buffer |
  | Overall timeout | Entire deployment marked failed | Masks actual install failures | Sum of phase timeouts + 10 min |
- ## See Also: ../admin-setup-apv1/03-esp-policy.md, win32-app-packaging.md, ../lifecycle/04-esp.md
