## WINDOWS-CLONE-DEEPNEST-TIMEOUT-01: Chain-Apex Cold-Clone Deep-Nesting Subprocess Cost on Windows (CARRIED FORWARD — WORSE at Phase 95 + D-03 CORRECTED)

**Scope:** Running ANY chain validator from a cold fresh-clone on Windows exhibits an O(n²) cascade of cold Node subprocess spawns at the deepest nested guards, approaching the per-subprocess timeout and producing spurious failures or hangs. **Carried forward from v1.11; WORSE at Phase 95** because the apex chain is now `[48..94]` (+2 deeper than Phase 93's `[48..92]`, +7 deeper than Phase 88's `[48..87]`). **D-03 OS-SPLIT CORRECTED at v1.12:** BOTH chain validators cascade — not only the apex, but also the continuity validator `check-phase-93.mjs` (chain [48..92]).

**Evidence of correctness (not a regression):**
- Phase 88 (v1.10): Windows warm-tree `check-phase-88.mjs` runs 1 and 2: 41 PASS / 1 FAIL / 1 SKIPPED — identical FAIL on V-88-CHAIN-66 (check-phase-66 subprocess output truncated). `check-phase-66.mjs` standalone: 28/0/0 — confirms NOT a real failure.
- Phase 93 (v1.11): Windows cold-clone `check-phase-88.mjs` (continuity, [48..87]) did not return within 3+ minutes during the Axis-1 re-audit (12+ live `node` processes); empirically confirmed same cascade on cold clone.
- Phase 95 (v1.12): BOTH `check-phase-93.mjs` (continuity, [48..92]) AND `check-phase-95.mjs` (apex, [48..94]) confirmed as chain validators that cascade. Neither was run on Windows cold-clone Axis-1. Linux GHA apex (49/0/1) and continuity (47/0/1) are exclusively authoritative.
- Linux GHA apex (check-phase-95, [48..94]): 49/0/1 — no truncation. Linux GHA is exclusively authoritative for BOTH chain validators.

**D-03 OS-split correction (v1.12 — corrects v1.11 "continuity = fast non-apex" error):**
- `check-phase-95.mjs` (v1.12 apex, CHAIN `[48..94]`, 47 phases) → Linux GHA **sole-authoritative**: **49/0/1**.
- `check-phase-93.mjs` (v1.11 continuity, CHAIN `[48..92]`, 45 phases) → Linux GHA **sole-authoritative**: **47/0/1** (confirmed exits 0 nested in v1.12 chain run). The v1.11 plan labeled this validator "fast non-apex" — corrected: it is a chain validator that spawns children with `CHECK_PHASE_NESTED=1` and produces the same O(n²) cascade on cold Windows.

**Source:** v1.9 `WINDOWS-CLONE-DEEPNEST-TIMEOUT-01` (Plan 82-03) → v1.10 carry → v1.11 carry (`93-03-AUDIT-RESULTS.md`) → v1.12 carry (`95-03-AUDIT-RESULTS.md`). Depth now **[48..94]** (D-01 corrected from [48..92]).

**Mitigation adopted (Phase 95, per D-03 MANDATORY — corrected):** The authoritative chain validator counts are taken from **Linux GHA** (exclusively authoritative per D-03 corrected binding condition), NOT from a Windows warm-tree or cold-clone chain-validator run. The 2 genuine LEAF validators (v1.12-audit + check-phase-94) are taken from fast standalone fresh-clone runs and confirmed cross-OS EXACT MATCH.

**Remediation options (v1.13+):** (1) cache-warm clone before the apex run; (2) raise the per-subprocess timeout for cold-clone re-audits; (3) cache chain-validator subprocess results within a single apex invocation to avoid redundant sub-spawns at depth ≥ 2 (the O(n²) mitigation — increasingly attractive as the chain deepens past [48..94]).

**Trigger-to-sweep (v1.13+):** The next terminal re-audit setup (or when chain depth affects warm-tree performance). Note: as the chain grows past `[48..94]`, option (3) becomes the durable fix.

**Estimated effort:** 30min (cache-warm) / 15min (timeout config) / 2-4h (O(n²) subprocess caching).

**Status:** CARRIED FORWARD from v1.11 (still unresolved; chain now deeper [48..94], +2 from v1.11's [48..92]; D-03 corrected: BOTH chain validators Linux-authoritative). DEFERRED to v1.13+. Does NOT affect harness correctness or cross-OS parity.
