## FETCH-DEPTH-01: Shallow-clone broke v1.5-frozen-aware validators on GHA ubuntu-latest (CLOSED 2026-05-28)

**Scope:** Plan 68-03 Task 1 (commit `d7d7d5f`) introduced `readRequirementsAtV15Close()` in `scripts/validation/check-phase-61.mjs`: a helper that reads REQUIREMENTS.md state at the frozen v1.5-close SHA `ba2cbc0` via `execFileSync('git', ['show', 'ba2cbc0:.planning/REQUIREMENTS.md'])`. **Source-of-truth:** Plan 69-02 SUMMARY §B.1 iter 1 evidence + `69-VERIFICATION.md` §Section E Discovery #1 (FETCH-DEPTH-01).

**Root cause:** `actions/checkout@v4` defaults to `fetch-depth: 1` (shallow clone — only HEAD commit + tree present in runner's git object database). **Evidence:**
- B.1 iter 1 run `26513528485` (https://github.com/Schweinehund/Autopilot/actions/runs/26513528485): FAIL `Result: 23 PASS, 5 FAIL, 0 SKIPPED` with `fatal: invalid object name 'ba2cbc0'` as proximate cause
- B.1 iter 3 run `26576405590` (https://github.com/Schweinehund/Autopilot/actions/runs/26576405590): SUCCESS `Result: 28 PASS, 0 FAIL, 0 SKIPPED` post-fix

**Resolution (Plan 69-02 Fix-1 commit `85521bb`):** Added `with: { fetch-depth: 0 }` to the `actions/checkout@v4` step in the `linux-chain-ubuntu-latest` job of `.github/workflows/audit-harness-v1.7-integrity.yml`. **Risk surface:** LOW. `fetch-depth: 0` is the standard GHA pattern for any workflow needing git history. v1.6 + v1.5 predecessor workflows unaffected (they don't reach check-phase-61 — those validators are Phase 61+ deliverables pinned to earlier validator surfaces).

## CHAIN-WRAPPER-01: check-phase-66.mjs:313 chain-apex wrapper stderr-only capture (DEFERRED to v1.8+)

**Scope:** `scripts/validation/check-phase-66.mjs` chain-apex wrapper at line 313 captures only `err.stderr` when a spawned validator fails:

```javascript
try {
  execFileSync('node', [path], { stdio: 'pipe', timeout: 300000 });
  return { pass: true, detail: 'exit 0' };
} catch (err) {
  return { pass: false, detail: `check-phase-${phaseNum} FAIL: ${(err.stderr || '').toString().trim()}` };
}
```

check-phase-61 (and most chain validators) writes per-validator FAIL detail to **stdout**, not stderr. The chain-66 wrapper truncates the diagnostic to empty stderr noise, masking the real failure cause in chain-apex output.

**Source-of-truth:** Plan 69-02 SUMMARY §SCOPE-GAP-61 forensic narrative + `69-VERIFICATION.md` §Section E Discovery #4 (CHAIN-WRAPPER-01).

**Impact:** Masked SCOPE-GAP-61 on Windows local for 2 weeks. Plan 69-01's tracking-update commit `6e12a75` was authored 2026-05-27; SCOPE-GAP-61 should have surfaced immediately on the next chain-66 run, but the stderr-only wrapper hid the V-61-05 detail. Only the GHA workflow_dispatch run B.1 iter 2 made the failure visible (because the GHA log shows the full job output, including chain-66's empty-stderr noise + the underlying check-phase-61's standalone diagnostic). On Windows local, the chain-66 output looked like a generic chain failure rather than a specific V-61-05 failure.

**Recommendation (v1.8+):** Fix wrapper to capture both stderr AND stdout, preserving diagnostic clarity in chain-apex output:

```javascript
} catch (err) {
  const out = ((err.stderr || '').toString() + (err.stdout || '').toString()).trim();
  return { pass: false, detail: `check-phase-${phaseNum} FAIL: ${out}` };
}
```

**Trigger-to-sweep:** Any future chain-apex masking incident OR v1.8+ chain-wrapper architecture review. Combined with SCOPE-GAP-61 retrospective audit for V-NN-NN HEAD-coupled assertion class — both belong to the same v1.8+ chain-resilience pickup.

**Status:** DEFERRED to v1.8+ per scope discipline. CILINUX-01 scope is workflow YAML, not chain wrapper. Surfaced as Phase 69 forensic discovery for v1.8+ pickup.

**Estimated effort:** 30min surgical wrapper edit + chain-apex regression sweep to confirm no false positives.
