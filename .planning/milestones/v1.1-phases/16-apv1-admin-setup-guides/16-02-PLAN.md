    Title: `# Dynamic Device Groups for Autopilot`

    ## Prerequisites:
    - Global Administrator or Groups Administrator role in Entra ID
    - Azure AD Premium P1 or P2 license (required for dynamic group membership — without it, dynamic membership does not work)
    - At least one device registered in Windows Autopilot

    ## Steps:

    ### Step 1: Create dynamic device group
    Navigate to **Azure portal** > **Microsoft Entra ID** > **Groups** > **New group**.

    Settings:
    - Group type: Security
    - Membership type: Dynamic Device
    - Add dynamic query

    **ZTDId membership rule** (matches ALL Autopilot devices in tenant):
    ```
    (device.devicePhysicalIDs -any (_ -startsWith "[ZTDid]"))
    ```

    What-breaks: Using wrong attribute name (e.g., `devicePhysicalIds` with wrong casing — note the `IDs` capitalization varies between docs; both work in practice but use the canonical form above). Admin sees empty group. End user sees standard OOBE. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    ### Step 2: Group tag targeting (optional)
    For targeting a subset of Autopilot devices by group tag:
    ```
    (device.devicePhysicalIds -any (_ -eq "[OrderID]:YOUR_GROUP_TAG"))
    ```
    Replace `YOUR_GROUP_TAG` with the actual tag value assigned during hardware hash import.

    What-breaks: Tag value is case-sensitive and must match exactly. Mismatched tag = device not in group = no profile. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    ### Step 3: Wait for group evaluation
    **Sync delay expectations** (prominent callout):
    - Simple rules in small tenants: 5-15 minutes
    - Complex rules or large tenants (10,000+ devices): up to 24 hours for initial evaluation
    - **Do NOT power on devices until group membership is confirmed** — device reaching OOBE before group evaluates means no Autopilot profile, standard Windows setup runs instead

    What-breaks: Admin sees device registered in Autopilot but group shows 0 members. End user sees standard Windows OOBE instead of Autopilot experience. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    ### Step 4: Verify group membership
    Navigate to Azure portal > Microsoft Entra ID > Groups > [group name] > Members. Confirm target devices appear. Cross-check with Intune admin center > Devices > Windows Autopilot > Devices — Profile Status should show "Assigning" or "Assigned" for devices in the group.

    **Profile conflict resolution callout:** Multiple profiles targeting same device via different groups: oldest created profile wins (not most specific, not highest priority). Audit profile creation dates when troubleshooting unexpected assignments. Avoid broad "All Autopilot Devices" profile if targeted profiles also exist. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    ## Verification:
    - [ ] Dynamic device group exists with correct membership rule
    - [ ] Group shows expected devices as members (may take 5-15 min)
    - [ ] Profile Status on target devices shows "Assigning" or "Assigned"
    - [ ] No conflicting broad profiles exist (check Deployment Profiles list for "All Devices" assignments)

    ## Configuration-Caused Failures table:
    | Misconfiguration | Symptom | Runbook |
    Entries for: wrong membership rule syntax, Azure AD Premium missing (dynamic membership silently fails), group tag case mismatch, device imaged before group evaluates, conflicting broad profile overriding targeted profile (oldest wins). All link to `../l1-runbooks/03-profile-not-assigned.md`.

    Intro: APv1 supports three deployment modes. Choose the mode that matches your provisioning workflow, then follow the mode-specific guide for detailed configuration.

    ## Mode Comparison Table (per D-08):

    | Feature | User-Driven | Pre-Provisioning | Self-Deploying |
    |---------|-------------|------------------|----------------|
    | **TPM 2.0 required** | No | **Yes** | **Yes** |
    | **Wired ethernet required** | No | **Yes (technician flow)** | **Yes** |
    | **User credentials at OOBE** | Required | First by technician, then user | None |
    | **Hybrid join support** | Yes (Intune Connector) | Yes (Intune Connector) | **No** (no user affinity) |
    | **User phase of ESP** | Yes | Yes (after reseal) | **No** |
    | **Trigger key** | N/A | **Win+F12** at first OOBE screen | N/A (automatic) |
    | **Technician reseals device** | No | Yes | No |
    | **Best for** | Standard office deployments | Factory/staging facility | Kiosks, shared devices, signage |

    ## Mode Selection Guidance:
    - **Start with User-Driven** if: standard office deployment, no staging facility, hybrid join needed without complexity.
    - **Use Pre-Provisioning** if: IT staging facility available, want device ready before shipping to user, TPM 2.0 and wired ethernet available in staging.
    - **Use Self-Deploying** if: shared device or kiosk scenario, no user affinity needed, device has TPM 2.0 and wired ethernet.

    ## Common OOBE Profile Settings (shared across all modes):
    These settings apply to all deployment modes and are configured in the deployment profile (see [Deployment Profile](02-deployment-profile.md)):
    - Microsoft Software License Terms (EULA): Show or Hide
    - Privacy settings: Show or Hide
    - Language/Region selection
    - Automatically configure keyboard: Yes or No
    - Apply device name template: Yes or No (Entra join only, <=15 chars)

    ## Mode-Specific Configuration Guides:
    - [User-Driven Mode](06-user-driven.md) — Standard deployment where end user authenticates during OOBE
    - [Pre-Provisioning Mode](07-pre-provisioning.md) — Technician prepares device before user receives it
    - [Self-Deploying Mode](08-self-deploying.md) — Fully automated, no user interaction required

    Title: `# User-Driven Mode Configuration`

    Intro: Most common APv1 deployment mode. End user authenticates during OOBE and the device completes enrollment with user affinity.

    ## Prerequisites (per D-09 — mode-specific prerequisites prominent):
    - Deployment profile created with Deployment mode = User-Driven (see [Deployment Profile](02-deployment-profile.md))
    - ESP policy configured and assigned (see [ESP Policy](03-esp-policy.md))
    - Dynamic device group configured and verified (see [Dynamic Groups](04-dynamic-groups.md))
    - For Entra join: no additional prerequisites beyond the above
    - For Hybrid Entra join: Intune Connector for Active Directory installed and active (see [Intune Connector](09-intune-connector-ad.md))

    ## Steps:

    ### Step 1: Verify deployment profile settings for user-driven
    Navigate to Intune admin center > Devices > Windows > Enrollment > Windows Autopilot > Deployment Profiles > [profile name].
    Confirm:
    - Deployment mode: **User-Driven**
    - Join type: **Microsoft Entra joined** OR **Hybrid Microsoft Entra joined**

    What-breaks: Wrong deployment mode = user sees different OOBE flow than expected. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    ### Step 2: Configure join type
    **Entra join path:**
    - No additional configuration beyond deployment profile and ESP
    - Device joins directly to Entra ID during OOBE

    **Hybrid Entra join path:**
    - Intune Connector for Active Directory must be installed, active, and version >= 6.2501.2000.5 (see [Intune Connector](09-intune-connector-ad.md))
    - OU for computer object creation must be configured in connector settings
    - ESP timeout must account for +40 minutes connector processing delay (see [ESP Policy](03-esp-policy.md))

    What-breaks for hybrid: Connector not installed or version too old = hybrid join fails silently. Admin sees device enrolled in Intune but not joined to AD. End user sees ESP timeout. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    3. "Let's set things up for work or school" prompt appears (Autopilot branding)
    4. User enters their organizational email and password
    5. Device phase of ESP begins (device setup, apps, policies)
    6. User phase of ESP begins (user-specific apps, policies)
    7. Desktop reached

    What-breaks: If step 3 shows standard Windows OOBE instead of Autopilot branding = profile not assigned. Check dynamic group membership and profile status. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    Title: `# Pre-Provisioning Mode Configuration`

    Intro: Two-phase deployment — technician completes device setup in a staging facility, then reseals for user. Also known historically as "white glove."

    ## Prerequisites (per D-09 — mode-specific prerequisites PROMINENT):
    - **Physical device required** (VMs not supported for pre-provisioning)
    - **TPM 2.0** with attestation capability (fTPM devices must reach manufacturer EK certificate endpoints)
    - **Wired ethernet** — Wi-Fi is NOT supported during pre-provisioning technician flow
    - Deployment profile with "Allow pre-provisioned deployment" set to **Yes** (see [Deployment Profile](02-deployment-profile.md))
    - User-driven mode working successfully in the tenant first (pre-provisioning builds on user-driven)
    - ESP policy configured with adequate timeout (recommend 90+ minutes for technician flow with large app sets)
    - For Hybrid Entra join: Intune Connector for Active Directory installed and active (see [Intune Connector](09-intune-connector-ad.md))

    ## Steps:

    ### Step 1: Verify deployment profile settings
    Confirm "Allow pre-provisioned deployment" = **Yes** in the deployment profile.

    What-breaks: Setting to No still allows Win+F12 keypress but deployment fails with error **0x80180005**. Admin sees error in Autopilot diagnostics. End user / technician sees provisioning failure screen. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    ### Step 2: Technician flow (staging facility)
    Document the technician OOBE flow:
    1. Connect device to **wired ethernet** (mandatory — Wi-Fi will NOT work)
    2. Power on device
    3. At the first OOBE screen (region selection), press **Win+F12** (or 5 times on the Windows button within 10 seconds as alternative)
    4. Pre-provisioning screen appears — "Windows Autopilot provisioning"
    5. Select **Provision device** (or it starts automatically depending on profile)
    6. Device completes Entra join, device phase of ESP (apps, policies, certificates)
    7. Screen shows green "Device provisioned successfully" or red error screen
    8. Technician selects **Reseal** — device shuts down for shipping to end user

    What-breaks for step 1: Using Wi-Fi instead of ethernet = provisioning fails at TPM attestation or network stage. Runbook: [Network Connectivity](../l1-runbooks/04-network-connectivity.md)

    What-breaks for step 3: Win+F12 not working = keyboard issue or BIOS intercepting F12. Try 5x Windows button alternative.

    ### Step 3: User flow (after reseal)
    End user receives device, powers on:
    1. Device boots to user sign-in (region/keyboard already configured by technician)
    2. User enters credentials
    3. User phase of ESP runs (user-specific apps, policies)
    4. Desktop reached

    ### Step 4: Hybrid join considerations
    Same as user-driven: Intune Connector required, OU configuration, +40 minute ESP buffer. Cross-reference [Intune Connector](09-intune-connector-ad.md).

    ## Verification:
    - [ ] Win+F12 triggers pre-provisioning screen (not standard OOBE)
    - [ ] Technician flow completes with green success screen
    - [ ] Device reseals and shuts down
    - [ ] End user receives device and completes user phase ESP
    - [ ] Device appears in Intune with correct user affinity after user sign-in

    ## Configuration-Caused Failures table (per D-15):
    | Misconfiguration | Symptom | Runbook |
    Entries for: "Allow pre-provisioned deployment" = No (error 0x80180005), Wi-Fi instead of ethernet, TPM 2.0 not available, VM used (not supported), connector too old for hybrid, ESP timeout insufficient. Link to `../l1-runbooks/05-oobe-failure.md` and `../l1-runbooks/04-network-connectivity.md`.

    **L2 supplementary** (per D-16 — `<details>` block): Link to [L2: TPM Attestation](../l2-runbooks/03-tpm-attestation.md) for TPM investigation.

    6. No user phase — any user signing in gets standard sign-in experience
