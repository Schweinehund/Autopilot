Context section: State the gather-first principle (same as APv1 01-log-collection.md). Then add the CRITICAL callout (per D-01 and Pitfall 1 from research):

```
> **CRITICAL: MDM Diagnostic Tool does NOT apply to APv2.**
> `mdmdiagnosticstool.exe -area Autopilot -cab` is designed for APv1 (classic Autopilot) and the ESP flow. It does not capture BootstrapperAgent data. Do not use it for APv2 investigations. APv2 diagnostics are collected from: (1) the BootstrapperAgent event log, (2) the IME log folder, and (3) the Intune deployment report in the admin center.
```

Investigation section with these steps:

**Step 1: Verify deployment state in Intune portal**
Navigate to Intune admin center > Devices > Monitor > Windows Autopilot device preparation deployments. Select the device. Record: Deployment status, Phase, Serial number, UPN, Deployment time. Note any error details.

**Step 2: Export BootstrapperAgent event log**
Build naming prefix using established convention from 01-log-collection.md:
```powershell
$serial = (Get-WmiObject -Class Win32_BIOS).SerialNumber
$date = Get-Date -Format "yyyy-MM-dd"
$prefix = "$date`_$serial"
```

Discovery command (include as Step 2a — makes guide self-correcting per research recommendation):
```powershell
# Discover the exact registered log name on this device
wevtutil el | Where-Object { $_ -match "Bootstrapper|IntuneManagement" }
```

Export command:
```powershell
# Export BootstrapperAgent event log (replace log name if discovery returned a different name)
wevtutil epl "Microsoft-Windows-IntuneManagementExtension/BootstrapperAgent" "$prefix`_bootstrapper.evtx"
```

Note: "If the log name is not found, the IME may not be installed yet on this device. This is expected for failures that occur before Step 4 (IME installation) of the deployment flow. Proceed to Step 3 for IME log collection."

**Step 3: Copy IME log folder**
Path: `C:\ProgramData\Microsoft\IntuneManagementExtension\Logs` (Official, HIGH confidence).
Copy commands for the 3 primary APv2 investigation logs:
```powershell
$src = "C:\ProgramData\Microsoft\IntuneManagementExtension\Logs"
$dest = "C:\Temp\APv2Diag"
New-Item -ItemType Directory -Path $dest -Force | Out-Null
Copy-Item "$src\IntuneManagementExtension.log" "$dest\$prefix`_ime-main.log"
Copy-Item "$src\AppWorkload.log" "$dest\$prefix`_appworkload.log"
Copy-Item "$src\AgentExecutor.log" "$dest\$prefix`_agentexecutor.log"
```

Include table of what each log contains:
| Log File | Primary Content |
|----------|----------------|

| AgentExecutor.log | PowerShell script execution tracking -- primary source for script failure investigation |

**Bottom section — Investigation Paths by Failure Type (per D-07):**

### Entra Join Failed (Phase: Policy installation)
Indicators: Deployment status Failed, Phase shows Policy installation, no Apps/Scripts tab entries.
Investigation:
1. Check Entra device settings: Entra admin center > Devices > Device settings > "Users may join devices to Microsoft Entra" — verify user is included
2. Check ETG group ownership: verify Intune Provisioning Client (AppID: f1346770-5b25-470b-88bd-d5744ab7952c) is owner of the ETG group
3. Check MDM auto-enrollment scope: Entra admin center > Mobility > Microsoft Intune — verify scope includes user
Resolution: Fix the identified configuration gap and retry deployment.

### Enrollment Failed (Phase: Policy installation)
Indicators: Deployment status Failed, Phase shows Policy installation, device appears in Entra but not in Intune.
Investigation:
1. Check MDM scope: Entra admin center > Mobility > Microsoft Intune — MDM user scope must include the enrolling user
2. Check license: Microsoft 365 admin center > Active users > select user > Licenses — verify Intune-capable license assigned
3. Check enrollment restrictions: Intune admin center > Devices > Enrollment restrictions — verify no blocking restriction
Resolution: Assign missing license or correct MDM scope, then retry.

### App Install Failed (Phase: App installation or Policy installation)
Indicators: One or more apps show Failed in Apps tab.
Investigation:
1. Note the failed app name and status
2. Collect AppWorkload.log per [APv2 Log Collection](06-apv2-log-collection.md)
3. Search AppWorkload.log for the app name — look for exit codes, error messages, detection rule failures
4. Check app installation context (System vs User) in Intune admin center > Apps > select app > Properties
5. If context is User: this is the cause — APv2 OOBE requires System context
Resolution: Change app install context to System, fix detection rules, or repackage.

### Script Failed (Phase: Script installation)
Indicators: One or more scripts show Failed in Scripts tab.
Investigation:
1. Note the failed script name
2. Collect AgentExecutor.log per [APv2 Log Collection](06-apv2-log-collection.md)
3. Search AgentExecutor.log for the script name — look for exit codes and error output
4. Check script run-as-account (System vs User) in Intune admin center > Devices > Scripts
Resolution: Fix script error, change context to System, or adjust exit code handling.

### Deployment Timeout
Indicators: Deployment status Failed, deployment time matches or exceeds configured timeout.
Investigation:
1. Check timeout configuration: Intune admin center > Devices > Windows > Enrollment > Device preparation policies > select policy > "Minutes allowed before device preparation fails"
2. Count total apps + scripts assigned to the policy
3. Review Apps and Scripts tabs for In progress items at time of timeout
4. For Windows 365 devices: check if deployment predates February 2026 fix for hardcoded 60-minute timeout
Resolution: Increase timeout, reduce app count, optimize app packaging, or split across policies.
