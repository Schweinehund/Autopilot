**Title:** `# APv2 Setup Step 3: Device Preparation Policy`

**Introduction:** The Device Preparation policy is the central APv2 configuration object. It defines deployment settings, OOBE experience, and the apps/scripts to install during enrollment. This guide walks through creating the policy step by step with troubleshooting guidance for each configurable setting.

**Prerequisites section:**
- Step 1 complete: [Prerequisites and RBAC Role](01-prerequisites-rbac.md)
- Step 2 complete: [ETG Device Group](02-etg-device-group.md) with apps/scripts assigned
- Custom RBAC role or Global Administrator permissions

**Steps section:**

### Step 1: Create a new Device Preparation policy

1. Navigate to **Intune admin center** > **Devices** > **Windows** > **Windows enrollment** > **Device preparation policies**.
2. Select **Create**.
3. Enter a policy name (e.g., "APv2 User-Driven Entra Join") and description.

### Step 2: Configure deployment settings

1. **Deployment mode**: Select **User-driven**.

2. **Join type**: Select **Microsoft Entra joined**.

3. **User account type**: Select **Standard User** or **Administrator**.

   > **Active known issue (April 2026):** The User account type setting interacts with the Entra ID Local administrator settings. Not all combinations produce the expected result. The following table documents the valid combinations:
   >
   > **For Standard User result** (user is non-admin on device):
   >
   > | Entra ID Local Admin Setting | Device Prep Policy User Account Type | Result |
   > |------------------------------|--------------------------------------|--------|
   > | None | Administrator | Standard User |
   > | Selected (non-admin users excluded) | Administrator | Standard User |
   > | All | Standard User | Standard User |
   >
   > **For Administrator result** (user is admin on device):
   >
   > | Entra ID Local Admin Setting | Device Prep Policy User Account Type | Result |
   > |------------------------------|--------------------------------------|--------|
   > | All | Administrator | Administrator |
   > | Selected (user is selected) | Administrator | Administrator |
   >
   > **Entra ID path:** Azure portal > Microsoft Entra ID > Manage | Devices > Manage | Device settings > Local administrator settings

   > **What breaks if misconfigured:** **Admin sees:** Deployment report shows success, but user account type does not match intent.
   > **End user sees:** If Entra ID Local Admin = Selected or None AND Device Prep policy = Standard User: provisioning is skipped entirely, user reaches the desktop WITHOUT expected apps installed, no error shown during OOBE.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 3: Configure device group

1. Select **Add device group**.
2. Choose the ETG device group created in Step 2 of the setup sequence ([ETG Device Group](02-etg-device-group.md)).

   > **Warning:** Select the **device** group here, not a user group. The Device group page requires the ETG security group. The Assignments page (later) requires the user group. Selecting the wrong group type is a common misconfiguration (see Pitfall 7 in the failure table below).

   > **What breaks if misconfigured:** **Admin sees:** Policy shows "0 devices" or wrong group in the Device group field.
   > **End user sees:** Deployment never launches -- no matching policy found for the enrolling device.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 4: Configure out-of-box experience settings

1. **Minutes allowed before showing installation error**: Enter a value between **15** and **720** (recommended: **60** for standard deployments, increase for large app sets).

   > **What breaks if misconfigured:** **Admin sees:** Deployment report shows "Deployment timed out" status.
   > **End user sees:** Error screen appears during OOBE after the timeout elapses, with Retry and (optionally) Continue anyway buttons.
   > **Runbook:** [Deployment Timeout](../l1-runbooks/09-apv2-deployment-timeout.md)

2. **Custom error message**: Enter an optional message shown to end users on failure (e.g., "Contact IT Help Desk at ext. 1234").

3. **Allow users to skip setup after multiple attempts**: Select **Yes** or **No**.
   - **Yes**: Adds a "Continue anyway" button alongside Retry on the error screen (allows user to proceed to desktop with incomplete setup).
   - **No**: User can only Retry or contact support.

   > **What breaks if misconfigured:** **Admin sees:** No failure -- this is a policy choice, not a misconfiguration.
   > **End user sees:** If set to Yes: user may skip setup and work with missing apps/policies. If set to No: user is blocked until deployment succeeds or IT intervenes.
   > **Runbook:** [Deployment Timeout](../l1-runbooks/09-apv2-deployment-timeout.md)

4. **Show link to diagnostics**: Select **Yes** or **No**.
   - **Yes**: Shows a diagnostic log export link on the failure page (useful for L2 troubleshooting).
   - **No**: No diagnostic link shown.

### Step 5: Configure apps

1. Select **Add apps** to choose up to **25** managed applications to install during OOBE.
2. Supported app types: LOB (line-of-business), Win32, Microsoft Store (WinGet only), Microsoft 365, Enterprise App Catalog.

   > **Requirement:** Every app selected here MUST also be assigned to the ETG device group with **Required** assignment intent and **Device (System)** install context. The policy selection and the app assignment are two separate steps -- both are required. See [ETG Device Group Step 3](02-etg-device-group.md) for the assignment procedure.

   > **What breaks if misconfigured:** **Admin sees:** Apps show "Skipped" status in the APv2 deployment report (if not assigned to ETG group) or "Failed" status (if install context is User instead of Device).
   > **End user sees:** Device Preparation experience completes but apps are missing from the desktop.
   > **Runbook:** [Apps Not Installed](../l1-runbooks/07-apv2-apps-not-installed.md)

### Step 6: Configure scripts

1. Select **Add scripts** to choose up to **10** PowerShell scripts to run during OOBE.

   > **Requirement:** Every script selected here MUST also be assigned to the ETG device group. The script setting "Run this script using the logged on credentials" MUST be set to **No** (runs as System context). There is no signed-in user during OOBE.

   > **What breaks if misconfigured:** **Admin sees:** Scripts do not appear in the deployment report or show "Failed."
   > **End user sees:** Expected configurations from scripts are not applied.
   > **Runbook:** [Apps Not Installed](../l1-runbooks/07-apv2-apps-not-installed.md)

### Step 7: Configure assignments (user group)

1. Under **Assignments**, select **Add group**.
2. Choose the **user security group** containing users who should receive this APv2 deployment.

   > **Warning:** Select the **user** group here, not the device group. The Assignments page targets the user group. The Device group page (Step 3) targets the ETG device group. Mixing these up prevents policy matching at enrollment time.

   > **What breaks if misconfigured:** **Admin sees:** Policy exists but no users are targeted. Deployment report shows no entries.
   > **End user sees:** Deployment never launches during OOBE.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 8: Review and create

1. Review all settings on the summary page.
2. Select **Create**.

**Policy priority note:** If multiple Device Preparation policies target the same user group, the highest priority policy (lowest priority number) wins. Manage priority by drag-and-drop in the Device preparation policies list at **Intune admin center** > **Devices** > **Windows** > **Windows enrollment** > **Device preparation policies**.

**Verification section:**
- [ ] Device Preparation policy appears in the Device preparation policies list
- [ ] Policy shows the correct ETG device group in the Device group field
- [ ] Policy shows the correct user group in the Assignments field
- [ ] Timeout value is set (recommended: 60 minutes for standard, higher for large app sets)
- [ ] All required apps appear in the policy's Apps section
- [ ] Entra ID Local administrator settings match the valid combinations table above for the desired User account type result

**Title:** `# APv2 Setup Step 4: Corporate Identifiers`

**Introduction:** Corporate identifiers identify devices as corporate-owned during enrollment. This step is **conditional** -- corporate identifiers are only required if your organization has enrollment restrictions that block personal device enrollments. If no enrollment restrictions exist, or if restrictions allow personal device enrollment, this step can be skipped.

**When Corporate Identifiers Are Required:** A brief subsection explaining:
- Corporate identifiers are needed when: enrollment restrictions are configured to block personal Windows device enrollment AND devices will enroll via APv2
- Corporate identifiers are NOT needed when: no enrollment restrictions exist, or restrictions allow personal device enrollment
- Without corporate identifiers, a device enrolling via APv2 is treated as a personal device at enrollment time. If a personal enrollment restriction blocks it, the Device Preparation experience never launches.

**Prerequisites section:**
- Step 1-3 complete: [Prerequisites and RBAC](01-prerequisites-rbac.md), [ETG Device Group](02-etg-device-group.md), [Device Preparation Policy](03-device-preparation-policy.md)
- Device manufacturer, model, and serial number information for target devices
- Minimum Windows version: Windows 11 22H2 with KB5035942 (OS Build 22621.3374) or later

**Steps section:**

### Step 1: Prepare the CSV file

1. Create a CSV file with the following format (no header row):
   ```
   manufacturer,model,serial_number
   ```
   Example:
   ```
   Microsoft,Surface Pro 9,01234567890123
   Dell,Latitude 5540,ABC1234567
   ```

2. **Windows-supported identifier type:** Manufacturer, model, and serial number (combined). The following identifier types are NOT supported for Windows:
   - Serial number only (supported for Android/iOS only)
   - IMEI (not supported for Windows)

3. **Limits:** Maximum 5,000 rows or 5 MB per CSV file, whichever comes first. Maximum 10 CSV files for Windows identifiers. For larger volumes, use PowerShell or Microsoft Graph API.

   > **What breaks if misconfigured:** **Admin sees:** Upload fails or identifier does not match the enrolling device.
   > **End user sees:** Deployment never launches if enrollment restrictions block personal devices.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 2: Upload corporate identifiers

1. Navigate to **Intune admin center** > **Devices** > **Enrollment** > **Corporate device identifiers** tab.
2. Select **Add** > **Upload CSV file**.
3. For Identifier type, select **Manufacturer, model, and serial number (Windows only)**.
4. Browse to the CSV file created in Step 1 and upload.
5. Review the import results for any errors or duplicate entries.

   > **What breaks if misconfigured:** **Admin sees:** Import errors in the upload results, or identifiers uploaded with wrong type selection.
   > **End user sees:** Device not recognized as corporate-owned at enrollment time. If enrollment restriction blocks personal devices, deployment never launches.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 3: Verify uploaded identifiers

1. In the Corporate device identifiers list, confirm the uploaded identifiers appear.
2. Verify the Identifier type column shows "Manufacturer, model, and serial number."
3. Spot-check several entries against the source data for accuracy.

**Enrollment Restriction Interaction subsection** (per D-10 -- substantive, not just a link):

### Enrollment Restriction Interaction with APv2

This section explains how enrollment restrictions interact with APv2 (Device Preparation) deployments. Per D-11, this covers APv2-relevant interactions only.

**Key principle:** Enrollment restrictions are evaluated BEFORE the Device Preparation policy. A device blocked by an enrollment restriction will never reach the Device Preparation experience.

**APv2 Enrollment Ownership Behavior:**

| Enrollment Scenario | Without Corporate Identifiers | With Corporate Identifiers |
|---------------------|-------------------------------|---------------------------|
| APv2 Device Preparation enrollment | Treated as **personal** device at enrollment time. **Blocked** by personal enrollment restriction. | Treated as **corporate-owned** at enrollment time. Allowed by personal enrollment restriction. |

**Conflict Precedence Rules:**
1. Enrollment restrictions are evaluated first, before any Device Preparation policy processing.
2. If the device is blocked by an enrollment restriction, the Device Preparation experience never launches. The user sees a generic enrollment error, not an APv2-specific error.
3. Corporate identifiers determine ownership ONLY at enrollment time. After enrollment, the device's ownership state follows standard Intune rules.
4. If multiple enrollment restrictions target the same user, the highest priority restriction (excluding the Default restriction) applies. The Default restriction applies to any user not targeted by a higher-priority restriction.

**Diagnosis path:** If a device's APv2 deployment does not launch and the admin suspects an enrollment restriction issue, follow [Deployment Not Launched Runbook](../l1-runbooks/06-apv2-deployment-not-launched.md) -- step 13 checks corporate identifiers as a root cause.

   > **What breaks if misconfigured:** **Admin sees:** Device enrolls into Intune but Device Preparation experience does not launch. Deployment report shows no entry for the device.
   > **End user sees:** OOBE completes without the APv2 setup experience. Device reaches the desktop without expected apps/policies.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

**Verification section:**
- [ ] Corporate identifiers appear in Intune admin center > Devices > Enrollment > Corporate device identifiers
- [ ] Identifier type shows "Manufacturer, model, and serial number" for all Windows entries
- [ ] If enrollment restrictions block personal devices: target devices' manufacturer/model/serial match uploaded identifiers
- [ ] Enrollment restrictions page shows the expected restriction priority order
