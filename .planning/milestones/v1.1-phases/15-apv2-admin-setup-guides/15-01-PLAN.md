**Title:** `# APv2 Setup Step 1: Prerequisites and RBAC Role`

**Prerequisites section:**
Per D-02, APv1 deregistration is the FIRST prerequisite. Structure as a numbered checklist:

0. **APv1 Deregistration** -- If any target devices are currently registered with APv1 (classic Autopilot), they must be deregistered BEFORE configuring APv2. APv1 silently takes precedence when both registrations exist -- the device will enter the APv1 ESP flow instead of the APv2 Device Preparation experience with no error shown. See [APv1 Registration Conflict Runbook](../l1-runbooks/08-apv2-apv1-conflict.md) for diagnosis steps.

1. **Licensing** -- Microsoft Intune Plan 1 (included in Microsoft 365 E3/E5, EMS E3/E5, Intune standalone). Device Preparation requires Intune enrollment authority.

2. **Intune auto-enrollment** -- MDM auto-enrollment must be configured in Entra ID: Azure portal > Microsoft Entra ID > Mobility (MDM and WIP) > Microsoft Intune. MDM user scope must cover targeted users. Without this, devices will not auto-enroll into Intune during OOBE.

3. **Entra ID device join permissions** -- Targeted users must be allowed to join devices to Microsoft Entra ID: Azure portal > Microsoft Entra ID > Devices > Device settings > Users may join devices to Microsoft Entra ID. Without this, the join step during OOBE will fail.

4. **Device-level prerequisites** -- For OS version, networking, and hardware requirements, see [APv2 Prerequisites Checklist](../lifecycle-apv2/01-prerequisites.md). This guide covers admin configuration prerequisites only.

**Steps section:**

### Step 1: Verify APv1 deregistration status

1. Navigate to **Intune admin center** > **Devices** > **Windows** > **Windows enrollment** > **Devices** (under Windows Autopilot Deployment Program).
2. Search for target device serial numbers.
3. If devices appear in this list, they have APv1 registrations that must be removed before APv2 configuration.

   > **What breaks if misconfigured:** **Admin sees:** Device appears to enroll normally in Intune.
   > **End user sees:** ESP (Enrollment Status Page) appears during OOBE instead of the Device Preparation experience -- the wrong deployment flow runs silently.
   > **Runbook:** [APv1 Registration Conflict](../l1-runbooks/08-apv2-apv1-conflict.md)

### Step 2: Create custom RBAC role for APv2 administration

Per ASET-04, the RBAC role must be created as a prerequisite before any setup step.

1. Navigate to **Intune admin center** > **Tenant administration** > **Roles** > **Create**.
2. Select **Intune role**.
3. Enter a name (e.g., "APv2 Device Preparation Administrator").
4. Enter a description (e.g., "Custom role for Windows Autopilot Device Preparation policy management").
5. Configure the following 5 permission categories:

   **Category 1: Device configurations**
   - Read: Yes
   - Delete: Yes
   - Assign: Yes
   - Create: Yes
   - Update: Yes
   - View Reports: No (leave at default)

   **Category 2: Enrollment programs**
   - Enrollment time device membership assignment: Yes
   - All other permissions: No (leave at defaults)

   **Category 3: Managed apps**
   - Read: Yes
   - All other permissions: No

   **Category 4: Mobile apps**
   - Read: Yes
   - All other permissions: No

   **Category 5: Organization**
   - Read: Yes
   - All other permissions: No

6. Select **Next** to proceed to Scope tags (configure as needed for your organization).
7. Select **Next** to proceed to Assignments. Add the security group containing APv2 administrators.
8. Select **Create** to save the role.

   > **What breaks if misconfigured:** **Admin sees:** "Access denied" or "Insufficient permissions" errors when creating Device Preparation policies or assigning ETG groups.
   > **End user sees:** Deployment never launches because the policy was not created or not assigned.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

**Title:** `# APv2 Setup Step 2: Enrollment Time Grouping Device Group`

**Introduction:** Brief paragraph explaining that the Enrollment Time Grouping (ETG) device group is the core mechanism for APv2. At enrollment time, the Intune Provisioning Client service principal automatically adds the device to this group, triggering app and policy delivery. Link to [ETG explanation](../lifecycle-apv2/00-overview.md#enrollment-time-grouping----the-core-mechanism) for the full model description.

**Prerequisites section:**
- Step 1 complete: [Prerequisites and RBAC Role](01-prerequisites-rbac.md)
- Microsoft Graph PowerShell modules installed (Microsoft.Graph.Authentication and Microsoft.Graph.Applications) for service principal setup
- Global Administrator or Application Administrator role for the PowerShell consent step

**Steps section:**

### Step 1: Create the ETG security group

1. Navigate to **Azure portal** > **Microsoft Entra ID** > **Groups** > **New group**.
2. Configure the group with the following 4 required settings:

   **ETG Device Group Checklist** (all 4 items required per D-05):

   | Setting | Required Value | Notes |
   |---------|---------------|-------|
   | Group type | Security | NOT Microsoft 365 |
   | Membership type | Assigned | NOT Dynamic -- ETG requires assigned membership |
   | Microsoft Entra roles can be assigned to the group | No | Role-assignable groups have different lifecycle |
   | Owner | Intune Provisioning Client (AppID: `f1346770-5b25-470b-88bd-d5744ab7952c`) | Added in Step 2 below |

3. Enter a group name (e.g., "APv2-ETG-Devices") and description.
4. Select **Create**.

   > **What breaks if misconfigured:** **Admin sees:** Group appears in Entra ID but devices are never added to it during enrollment.
   > **End user sees:** Device Preparation experience never launches during OOBE, or apps/scripts are not delivered.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 2: Add the Intune Provisioning Client as group owner

The Intune Provisioning Client service principal (AppID: `f1346770-5b25-470b-88bd-d5744ab7952c`) MUST be the owner of the ETG device group. Without this ownership, the service principal cannot add devices to the group at enrollment time.

**Option A: Add via Entra ID portal**

1. Navigate to **Azure portal** > **Microsoft Entra ID** > **Groups** > select the ETG group created in Step 1.
2. Select **Owners** > **Add owners**.
3. Search for "Intune Provisioning Client" or "Intune Autopilot ConfidentialClient" (both names refer to the same service principal with AppID `f1346770-5b25-470b-88bd-d5744ab7952c`).
4. Select the service principal and confirm.

   > **Note:** If the service principal does not appear in the search results, it may not exist in your tenant yet. Use Option B (PowerShell) to create it.

**Option B: Add via PowerShell** (per D-04 -- inline, not in `<details>` block)

Use this procedure if the service principal does not appear in the Entra ID portal, or to automate the setup.

```powershell
# Source: https://learn.microsoft.com/en-us/autopilot/device-preparation/tutorial/user-driven/entra-join-device-group
# Required modules
Install-Module Microsoft.Graph.Authentication -Scope CurrentUser
Install-Module Microsoft.Graph.Applications -Scope CurrentUser

# Connect with required scope
Connect-MgGraph -Scopes "Application.ReadWrite.All"
# Accept the "Permissions requested" consent window
# Check "Consent on behalf of your organization" if prompted

# Create the Intune Provisioning Client service principal in your tenant
New-MgServicePrincipal -AppId f1346770-5b25-470b-88bd-d5744ab7952c
```

**Expected results:**
- **Success:** The command returns a service principal object with the AppId and DisplayName.
- **`Status: 409 (Conflict)` with `Request_MultipleObjectsWithSameKeyValue`:** The service principal already exists in your tenant. No action needed -- proceed to add it as group owner.
- **`Status: 403 (Forbidden)` with `Authorization_RequestDenied`:** Insufficient permissions. Verify the account has Global Administrator or Application Administrator role and that the consent prompt was accepted.

After the service principal exists, add it as the ETG group owner:

```powershell
# Get the service principal object ID
$sp = Get-MgServicePrincipal -Filter "AppId eq 'f1346770-5b25-470b-88bd-d5744ab7952c'"

# Get the ETG group (replace with your group name)
$group = Get-MgGroup -Filter "DisplayName eq 'APv2-ETG-Devices'"

# Add the service principal as group owner
New-MgGroupOwnerByRef -GroupId $group.Id -BodyParameter @{
    "@odata.id" = "https://graph.microsoft.com/v1.0/serviceprincipals/$($sp.Id)"
}
```

   > **What breaks if misconfigured:** **Admin sees:** Device Preparation policy is created and assigned, but the deployment report shows "Not started" for all enrolling devices.
   > **End user sees:** OOBE completes but the Device Preparation experience never launches -- device goes directly to the desktop without apps or policies.
   > **Runbook:** [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md)

### Step 3: Assign apps and scripts to the ETG device group

Apps and PowerShell scripts that should be delivered during APv2 enrollment must be assigned to the ETG device group with **Required** assignment and **Device (System)** install context.

1. Navigate to **Intune admin center** > **Apps** > **All apps**.
2. Select each app to include in the APv2 deployment.
3. Select **Properties** > **Edit** (next to Assignments).
4. Under **Required**, select **Add group** and choose the ETG device group.
5. Set **Install context** to **Device** (System).
6. Repeat for all apps (up to 25 apps supported in the Device Preparation policy).

For PowerShell scripts:
1. Navigate to **Intune admin center** > **Devices** > **Scripts and remediations** > **Platform scripts**.
2. Select each script to include.
3. Under Assignments, add the ETG device group.
4. Ensure **Run this script using the logged on credentials** is set to **No** (runs as System context).

   > **What breaks if misconfigured:** **Admin sees:** Apps show "Skipped" status in the APv2 deployment report. Scripts do not appear in the deployment report at all.
   > **End user sees:** Device Preparation experience completes but expected apps are not installed. Desktop appears without required applications.
   > **Runbook:** [Apps Not Installed](../l1-runbooks/07-apv2-apps-not-installed.md)

**Verification section:**
- [ ] ETG security group exists in Entra ID with Group type = Security, Membership type = Assigned
- [ ] Intune Provisioning Client (AppID `f1346770-5b25-470b-88bd-d5744ab7952c`) appears in the group's Owners list
- [ ] Required apps are assigned to the ETG device group with Required assignment intent and Device (System) context
- [ ] PowerShell scripts assigned to the ETG device group have "Run this script using the logged on credentials" set to No

|------------------|---------|---------|
| Group type set to Microsoft 365 instead of Security | ETG membership assignment fails silently | [Deployment Not Launched](../l1-runbooks/06-apv2-deployment-not-launched.md) |
