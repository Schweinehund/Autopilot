# Requirements: macOS MDM-Migration Verification Closure (v1.12)

**Defined:** 2026-06-26
**Core Value:** IT teams can independently provision, troubleshoot, and manage macOS devices through Intune — including migrating fleets from Kandji/Iru — without escalating to engineering, trusting that every post-migration claim in the docs is either source-verified or honestly confidence-labelled.

**Scope basis:** Closes the three Phase-90 post-migration verification gaps routed forward at the v1.11 close-gate (`.planning/milestones/v1.11-DEFERRED-CLEANUP.md` Part A: INTUNE-PROFILE-ENROLLMENT-01, IRU-CONSOLE-DELETE-01, SUPERVISION-STATUS-POST-MIGRATION-01). All three edit the single existing file `docs/macos-lifecycle/02-mdm-migration-psso.md` (already navigation-wired in v1.11 — no new files, no navigation-last phase).

**Scope locked via `/adversarial-review`** (Finder/Adversary/Referee scoring, per durable preference `feedback_adversarial_review_preference`), 2026-06-26 — coherent triple **1-A / 2-A / 3-iii**:
- **Supervision (1-A / 3-iii):** ship the best-available inference as an explicit **MEDIUM-confidence callout + "pilot recommended"** pointer — NOT a flat assertion (would be assertion-on-silence), NOT an author-unrunnable pilot procedure (authors cannot verify their own steps).
- **Close ritual (2-A):** full **Path-A** harness lineage bump + 3-axis terminal re-audit — the Referee ruled the lineage a machine-validated, retroactively-costly invariant (held 9 milestones; the re-audit empirically caught FETCH-DEPTH-01 / SCOPE-GAP-61), not skippable ceremony.
- **Do NOT bundle** the CI-3 Managed-Apple-ID→Account 45-occurrence rename or any tooling refactor — both review agents concurred it is a byte-unchanged-invariant hazard and motivated scope-padding.

**Firm facts carried from v1.11 (do not re-litigate):** PSSO re-registration is ALWAYS required post-migration (Apple authoritative); OS-26 in-place migration is a genuine unenroll+reenroll with a reissued ACME cert; the Intune-side result is profile-based enrollment; same-tenant PSSO key survival is LOW-confidence and must not be relied on. The supervision question and the Intune-profile-config question are the *open* deltas these requirements close.

- [x] **MIGV-02**: The Kandji/Iru source-side steps section of guide `02` reflects the **verified current Iru (post-Kandji-rebrand, Oct 2025) console device-deletion UI path**, checked against live vendor documentation (`support.iru.io` / current Iru support portal + admin-console docs) — confirming the path is unchanged or correcting it — and confirms whether the delete operation still requires the documented secret-retrieval pre-flight (FileVault recovery key / Activation Lock bypass code). Both **Kandji** and **Iru** names remain surfaced for searchability.

- [x] **MIGV-03**: Guide `02` documents the best-available inference on whether **device supervision is preserved after an OS-26 in-place ABM "Assign Device Management" migration** to Intune, framed as an explicit **MEDIUM-confidence callout** that (a) states the most-likely behavior with sources, (b) **recommends a pilot-device test** (`profiles status` / `profiles list` before-and-after) without shipping author-unverified step-by-step procedure as if validated, and (c) makes **no claim that the PSSO key survives** (Apple authoritative: re-registration always required).
