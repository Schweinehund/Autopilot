11 sections in order:

1. **Context**: Stage 4 of 5. The ESP is the progress screen that tracks policy and app installation before unlocking the desktop. Two sequential phases: device phase (before user login) and user phase (after user login). Depends on: OOBE (Stage 3). Feeds into: Post-Enrollment Verification (Stage 5).

2. **What the User Sees**: The ESP screen showing "Identifying...", "Device setup", "Account setup" progress bars. Describe the visual progression without screenshots. Note that in self-deploying mode, there is no user phase display.

3. **What Happens**: Two subsections with numbered steps:

   **Device Phase** (runs before user login):
   1. MDM enrollment completes -- device receives enrollment GUID
   2. Device policies applied (Configuration profiles, Compliance, Device Restrictions)
   3. Device-targeted apps install (Win32 via Intune Management Extension, LOB via MDM pipeline)
   4. Certificates and SCEP/PKCS profiles applied
   5. Device-context PowerShell scripts run
   6. `FirstSync\IsServerProvisioningDone` set to true -- device phase complete

   **User Phase** (runs after user logs in):
   1. User enters credentials on ESP login screen
   2. User-targeted policies applied
   3. User-targeted apps installed
   4. User-targeted scripts and certificates applied
   5. Desktop unlocked on completion

   Include the **app type tracking table** (critical per CONTEXT.md):

   | App Type | Tracked by ESP | Required vs Available | Blocks Desktop |
   |----------|----------------|----------------------|----------------|
   | Win32 (required) | Yes | Required | Yes |
   | Win32 (available) | No | Available | No |
   | LOB/MSI (required) | Yes | Required | Yes |
   | LOB/MSI (available) | No | Available | No |
   | Microsoft Store (required) | Yes | Required | Yes |
   | Microsoft Store (available) | No | Available | No |
   | PowerShell scripts (device) | Yes | Required | Yes |
   | Certificates (device) | Yes | Required | Yes |

   Add a bold clarification after the table: "Available apps do NOT block the ESP. Only required apps configured to track in ESP block the desktop. This is the most frequent source of confusion."

   Include Mermaid diagram (LR direction, sequential subgraph per RESEARCH.md Pattern 7):
   ```
   subgraph Device Phase
     D1[MDM enrollment] --> D2[Device policies] --> D3[Device apps] --> D4[Device phase complete]
   end
   subgraph User Phase
     U1[User logs in] --> U2[User policies] --> U3[User apps] --> U4[Desktop unlocked]
   end
   D4 --> U1
   ```

4. **Behind the Scenes (L2 Note)**: Blockquote with 4-5 bullets covering:
   - `HKLM:\SOFTWARE\Microsoft\Enrollments\{GUID}\FirstSync` checkpoint. Link to `../reference/registry-paths.md`.
   - `HKLM:\SOFTWARE\Microsoft\Windows\Autopilot\EnrollmentStatusTracking` root. Link to registry-paths.md.
   - Sidecar/IME process for Win32 app installation tracking.
   - ExpectedPolicies and Sidecar diagnostic subkeys.

5. **Success Indicators**: Both phases complete without error, desktop unlocks, all required apps show installed, no timeout triggered.

6. **Failure Indicators**: ESP stuck at percentage, timeout after 60 min default, app installation failure, policy application error. Forward-links: "see [Phase 3 error codes](../error-codes/) *(available after Phase 3)*, [L1 ESP runbook](../l1-runbooks/) *(available after Phase 5)*, [L2 ESP deep-dive](../l2-runbooks/) *(available after Phase 6)*"

7. **Typical Timeline**: Device phase: 15-45 minutes depending on app count. User phase: 5-15 minutes. Total: 20-60 minutes typical. Default timeout: 60 minutes (configurable up to 120).

8. **Watch Out For**:
   - Too many required apps causing timeout
   - Available apps misconfigured as required (blocks ESP unnecessarily)
   - Co-management app tracking differences (brief note per CONTEXT.md)
   - ESP context disambiguation: OOBE ESP vs post-enrollment ESP (brief note per CONTEXT.md)

9. **Tool References**: `Get-AutopilotDeviceStatus`, `Restart-EnrollmentStatusPage` (forward-link: "remediation function -- see Phase 6"), `Get-AutopilotLogs`. Link all to `../reference/powershell-ref.md`. Further Reading URLs.

2. **What the Admin Sees**: Intune portal device view showing enrollment status, compliance state, app installation status. Also local device state via PowerShell.

3. **What Happens**: 3 numbered steps:
   1. User arrives at Windows desktop -- deployment complete from user perspective.
   2. Admin verifies deployment through checklist of observable states (see below).
   3. Device is handed off for productive use.

   Include the **verification checklist** (required per CONTEXT.md):
   - [ ] Device appears in Intune with "MDM" enrollment type
   - [ ] Azure AD join state shows "Azure AD joined" (or "Hybrid Azure AD joined" for hybrid)
   - [ ] Assigned Autopilot profile shows as "Assigned" in Intune device view
   - [ ] Compliance policy evaluation = Compliant
   - [ ] Required apps show installation state = Installed (Intune > Device > Apps)
   - [ ] Device serial number matches Autopilot registration record
   - [ ] No critical error events in MDM and Provisioning event log channels
   - [ ] `Get-AutopilotDeviceStatus` returns `RegistrationState.IsRegistered = $true`

4. **Behind the Scenes (L2 Note)**: Compliance evaluation timing, Graph API device state sync, Intune device inventory refresh interval. Link relevant endpoints.

5. **Success Indicators**: All checklist items pass, device is compliant, all apps installed, no error events.

6. **Failure Indicators**: Device shows non-compliant, apps missing, profile not assigned, error events present. Forward-links with Phase annotations.

7. **Typical Timeline**: Compliance evaluation: 15-30 minutes after enrollment. Full Intune sync: up to 8 hours for inventory refresh.

8. **Watch Out For**:
   - Compliance policy evaluation delay (device may show "Not evaluated" initially)
   - App installation state lag in Intune portal vs actual device state
   - Serial number mismatch if device was re-imaged without cleanup

9. **Tool References**: `Get-AutopilotDeviceStatus` (comprehensive snapshot), `Get-AutopilotRegistrationState` (ZTDID + tenant verification), `Get-AutopilotProfileAssignment` (profile confirmation). Link all to `../reference/powershell-ref.md`. Further Reading URLs.
