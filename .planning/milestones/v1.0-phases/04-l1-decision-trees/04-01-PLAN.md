**File 1: `docs/decision-trees/02-profile-assignment.md`**

Title: "# Profile Assignment Failure Decision Tree"
Intro: Helps L1 agents triage cases where no Autopilot profile is assigned or the wrong profile is applied.

Mermaid `graph TD` with PR-prefixed node IDs. Locked branch logic:

Validation gate: PRD1 "Is the device registered in the Autopilot portal?" — No: `([Return to Initial Triage](00-initial-triage.md))` (triage handles "not in portal" separately). Yes: continue.

Branches (per CONTEXT.md — assumes device IS in portal):
- PRD2: "Is an Autopilot profile assigned to the device?" (check Intune portal)
  - No: PRD3 "Is the device in the correct group for profile assignment?"
    - No: PRA1 "Add device to the correct group" → PRD4 "Wait 30 minutes — is profile now assigned?" → Yes: PRR1 Resolved / No: PRE1 Escalate L2 (profile still not assigned after group fix)
    - Yes (in correct group but no profile): PRE2 Escalate L2 (group membership correct but profile not assigning)
    - Don't know: PRE3 Escalate L2 (collect group info for investigation)
  - Yes: PRD5 "Is it the correct profile for this device type?"
    - No: PRE4 Escalate L2 (wrong profile assigned — requires admin correction)
    - Yes: PRD6 "Has the profile been applied to the device?" (check device status in portal)
      - No: PRA2 "Reboot device and wait for sync" → PRD7 "Is profile now applied?" → Yes: PRR2 Resolved / No: PRE5 Escalate L2 (profile assigned but not applying)
      - Yes: PRR3 Resolved: Profile correctly assigned and applied
    - Don't know: PRE6 Escalate L2 (collect profile name and device details)

classDef declarations at bottom. All terminals use appropriate classes.

How to Check table entries:
- PRD1: "Intune admin center > Devices > Windows > Windows devices > search by serial number — device should appear"
- PRD2: "Same device page > Overview > Autopilot profile — check if a profile name is shown"
- PRD3: "Intune admin center > Groups > search for Autopilot group — check if device is a member"
- PRD5: "Compare profile name on device page to the expected profile for this device type. If unsure what the correct profile should be, choose Don't know."
- PRD6: "Device page > Overview > check profile status shows Applied, not Pending"

Escalation Data table: PRE1-PRE6 collect device serial, profile name (if any), group membership status, deployment mode, timestamp. See Also: `[L2 Profile Investigation](../l2-runbooks/) (available after Phase 6)`.

Resolution & Next Steps table: PRR1-PRR3 with forward-link `[L1 Profile Runbook](../l1-runbooks/) (available after Phase 5)`.

**File 2: `docs/decision-trees/03-tpm-attestation.md`**

Title: "# TPM Attestation Failure Decision Tree"
Intro: Helps L1 agents triage TPM-related failures during pre-provisioning and self-deploying mode provisioning.

Mermaid `graph TD` with TP-prefixed node IDs. Locked branch logic:

Validation gate: TPD1 "Did the device fail during pre-provisioning or self-deploying mode with a TPM-related error?" — No: `([Return to Initial Triage](00-initial-triage.md))`. Yes: continue.

Branches (per CONTEXT.md):
- TPD2: "Is TPM enabled in BIOS?" (L1 can check BIOS settings)
  - No: TPA1 "Enable TPM in BIOS" → TPD3 "Retry provisioning — did it succeed?" → Yes: TPR1 Resolved: TPM enabled, provisioning complete / No: TPE1 Escalate L2: TPM issue persists after enable
  - Yes: TPD4 "Is TPM version 2.0?" (visible in BIOS)
    - No: TPE2 Escalate: Hardware (TPM version too old, hardware replacement needed) — use escalateL2 class
    - Yes: TPD5 "Does the error screen show an error code?"
      - Yes: TPA2 "Look up error code in TPM error table" linking to `../error-codes/02-tpm-attestation.md` → TPR2 "Resolved: Follow L1 Action column in error table" / TPE3 "Escalate L2: Error code not found in table"
      - No: TPA3 "Power off device, wait 30 seconds, retry provisioning" → TPD6 "Did provisioning succeed after retry?" → Yes: TPR3 Resolved: Provisioning succeeded on retry / No: TPE4 Escalate L2: TPM attestation fails without error code
  - Don't know (cannot access BIOS): TPE5 Escalate L2: Cannot verify BIOS TPM settings

classDef declarations at bottom. All terminals use appropriate classes. Note: TPE2 uses escalateL2 (not escalateInfra) since hardware replacement is an L2/procurement issue.

How to Check table entries:
- TPD1: "Check error message on screen — look for keywords: TPM, attestation, provisioning failed, 0x800705xx series codes"
- TPD2: "Restart device, enter BIOS/UEFI setup (usually F2, F10, Del, or Esc at boot). Navigate to Security > TPM — check if TPM is Enabled"
- TPD4: "In BIOS TPM settings, check version number. Must show 2.0. Version 1.2 is not supported."
- TPD5: "Look at the error screen on the device — is there a hex code (0x...) displayed?"

Escalation Data table: TPE1-TPE5 collect device serial, make/model, TPM status (enabled/disabled/version), error code if visible, deployment mode (pre-provisioning or self-deploying), timestamp, screenshot. See Also: `[L2 TPM Investigation](../l2-runbooks/) (available after Phase 6)`.

Resolution & Next Steps table: TPR1-TPR3 with forward-link `[L1 TPM Runbook](../l1-runbooks/) (available after Phase 5)`.
