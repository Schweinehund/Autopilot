## WINDOWS-CLONE-DEEPNEST-TIMEOUT-01: Chain-Apex Cold-Clone Deep-Nesting Subprocess Cost on Windows (CARRIED FORWARD — WORSE at Phase 93)

**Scope:** Running the chain-apex from a cold fresh-clone on Windows exhibits an O(n²) cascade of cold Node subprocess spawns at the deepest nested guards, approaching the per-subprocess timeout and producing spurious failures or hangs. **Carried forward from v1.10; WORSE at Phase 93** because the apex chain is now `[48..92]` (+5 deeper than Phase 88's `[48..87]`, +11 deeper than Phase 82's `[48..81]`).

**Evidence of correctness (not a regression):**
- Phase 88 (v1.10): Windows warm-tree `check-phase-88.mjs` runs 1 and 2: 41 PASS / 1 FAIL / 1 SKIPPED — identical FAIL on V-88-CHAIN-66 (check-phase-66 subprocess output truncated at `[5/28]` boundary). `check-phase-66.mjs` standalone: 28/0/0 — confirms NOT a real failure.
- Phase 93 (v1.11): Windows cold-clone `check-phase-88.mjs` (continuity, [48..87]) did not return within 3+ minutes during the Axis-1 re-audit (12+ live `node` processes); empirically confirmed same cascade on cold clone.
- Linux GHA apex (check-phase-93, [48..92]): 47/0/1 — no truncation. Linux GHA apex is exclusively authoritative.

**Mitigation adopted (Phase 93, per D-03 MANDATORY):** The authoritative apex count is taken from **Linux GHA** (exclusively authoritative per D-03 binding condition), NOT from a Windows warm-tree or cold-clone apex run. The 4 non-chain leaf validators (89/90/91/92) are taken from fast standalone fresh-clone runs and confirmed cross-OS EXACT MATCH.

**Source:** v1.9 `WINDOWS-CLONE-DEEPNEST-TIMEOUT-01` (Plan 82-03) → v1.10 carry (`88-03-AUDIT-RESULTS.md` §D-04 Mitigation) → v1.11 carry (`93-03-AUDIT-RESULTS.md` §WINDOWS-CLONE-DEEPNEST-TIMEOUT-01). Depth now **[48..92]**.

**Remediation options (v1.12+):** (1) cache-warm clone before the apex run; (2) raise the per-subprocess timeout for cold-clone re-audits; (3) cache chain-validator subprocess results within a single apex invocation to avoid redundant sub-spawns at depth ≥ 2 (the O(n²) mitigation — increasingly attractive as the chain deepens past [48..92]).

**Trigger-to-sweep (v1.12+):** The next terminal re-audit setup (or when chain depth affects warm-tree performance). Note: as the chain grows past `[48..92]`, option (3) becomes the durable fix.

**Estimated effort:** 30min (cache-warm) / 15min (timeout config) / 2-4h (O(n²) subprocess caching).

**Status:** CARRIED FORWARD from v1.10 (still unresolved; chain now deeper [48..92], +5 from v1.10's [48..87]). DEFERRED to v1.12+. Does NOT affect harness correctness or cross-OS parity.
