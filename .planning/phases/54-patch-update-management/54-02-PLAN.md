```markdown
# Windows WUfB Rings + Hotpatch + Driver/Firmware

This guide is the Windows-specific patch management reference. It covers Windows Update for
Business (WUfB) deployment ring topology with deferral periods + deadline enforcement, Windows
Autopatch ring service-managed cohorts and disambiguation from WUfB deployment rings (PITFALL-9
mutual-exclusion is explicit), the Hotpatch servicing model defaulting on for Windows 11 Enterprise
24H2+ from May 2026, and the separate driver and firmware update policy surface.

For the cross-platform comparison and Ring Terminology hub, see

```

**Section 4 — WUfB Deployment Rings H2 (PATCH-01; PITFALL-9 first half — qualified ring usage everywhere):**

```markdown
## WUfB Deployment Rings {#wufb-deployment-rings}

A **WUfB deployment ring** is a Windows Update for Business policy in Intune (Intune > Devices >
Update rings) that configures deferral periods, deadlines, and restart behavior for quality and
feature updates. Tenants assign devices to a WUfB deployment ring via Azure AD group targeting,
and devices on that WUfB deployment ring receive Microsoft updates per the ring's configured
deferral cadence.

| WUfB Deployment Ring | Deferral Period | Deadline (days) | Pilot Cohort | Validate Before Promotion |
|----------------------|-----------------|-----------------|--------------|---------------------------|
| Pilot WUfB deployment ring | 0 days quality / 0 days feature | 2 days | 50 representative devices | Reporting healthy in Intune > Devices > Update rings |
| Broad WUfB deployment ring | 7 days quality / 14 days feature | 7 days | Production fleet | All Pilot devices reporting Compliant; no driver regressions |

After the deadline, the WUfB deployment ring policy can force-install + reboot. Configure the
restart-behavior options (auto-restart, snooze, deferral) per WUfB deployment ring to balance
user experience against compliance.
```

**Section 5 — Autopatch Disambiguation H2 (PATCH-01 second half; PITFALL-9 mutual-exclusion narrative):**

```markdown
## Windows Autopatch Rings (Disambiguation) {#autopatch-disambiguation}

A **Windows Autopatch ring** is a service-managed device cohort (Test, First, Fast, Broad
Autopatch rings) that Windows Autopatch automatically rotates and gates. Microsoft owns the
Autopatch ring rotation cadence; admins do not configure deferral periods or deadlines per
Autopatch ring (those are service-managed).

> **PITFALL-9 mutual exclusion:** WUfB deployment rings and Autopatch rings are **mutually
> exclusive** — they cannot coexist on the same device. When a tenant enables Windows Autopatch,
> Autopatch detaches enrolled devices from any pre-existing WUfB deployment ring assignment and
> takes ownership of the Windows Update workload for those devices. A single device is on EITHER
> a WUfB deployment ring OR an Autopatch ring — never both.

This mutual-exclusion is the load-bearing PITFALL-9 disambiguation. Mixing the two systems on the
same device causes update flapping, deadline conflicts, and compliance-report drift. Plan one
system per device fleet.

When migrating from WUfB deployment rings to Autopatch:

1. Validate the Windows Update workload is at Pilot Intune or Intune in co-management (per
   [Workload Slider Migration](../co-management/02-windows-workload-sliders.md)).
2. Enable Windows Autopatch in Intune > Tenant administration.
3. Confirm Autopatch ring assignments via Intune > Devices > Windows Autopatch > Devices.
4. Decommission obsolete WUfB deployment ring policies once all devices have moved to Autopatch
   rings.
```

**Section 6 — Hotpatch H2 (PATCH-02; V-54-12 — H2 string `## Hotpatch` exact match; all 4 tokens `default`/`May 2026`/`VBS`/`opt-out` (or `April 2026`) within file body; paste verbatim from 54-PATTERNS.md lines 165-181):**

```markdown
## Hotpatch {#hotpatch}

Windows 11 Enterprise 24H2+ ships with **Hotpatch as the default servicing model from May 2026**
onwards. Hotpatch eliminates reboot-on-most-updates by patching kernel and OS-mode binaries
in-memory, reducing reboots from monthly to quarterly (a baseline reboot is still required for
the quarterly cumulative update).

**Prerequisites:**

- Windows 11 Enterprise edition (24H2 or later)
- VBS (Virtualization-Based Security) enabled at firmware + OS level — Hotpatch requires VBS to
  validate kernel-binary patches in memory
- Eligible processor (verify via Intune compliance report or `msinfo32`)
- Device assigned to a WUfB deployment ring with Hotpatch enabled (or to an Autopatch ring on a
  Hotpatch-eligible servicing tier)

**Opt-out toggle (April 2026 Intune admin center addition):** Tenants can opt out of default-on
Hotpatch via Intune > Devices > Update rings > [select WUfB deployment ring] > Hotpatch toggle.
Setting the toggle to off restores classic monthly cumulative-update + reboot behavior for devices
on that WUfB deployment ring.

**Compliance reporting impact:** Hotpatch reduces reboot frequency, which changes the "reboot
pending" compliance signal cadence. Existing reboot-based alerting that fires on a monthly cadence
will need re-baselining (signal expected quarterly post-Hotpatch). Update health and pending-update
compliance signals continue to fire monthly per Microsoft's Hotpatch + LCU servicing model.
```

**Section 7 — Driver and Firmware Update Policy H2 (PATCH-03; V-54-13 — driver/firmware H2 + `dual-scan` token; paste structure from 54-PATTERNS.md lines 184-194):**

```markdown
## Driver and Firmware Update Policy {#driver-firmware-policy}

Driver and firmware updates are configured **separately** from quality/feature WUfB deployment
rings. The configuration surface is **Intune > Devices > Windows > Driver and firmware updates**
(or Intune > Devices > Update policies for Windows 10 and later > Driver and firmware updates,
depending on Intune blade version).

**This is NOT a ring** — neither a WUfB deployment ring nor an Autopatch ring. It is a discrete
update policy surface with its own approval cadence (manual driver approval; automatic firmware
delivery for OEM-published catalogs). Treating driver/firmware policy as a WUfB deployment ring
or Autopatch ring leads to misconfiguration.

**Dual-scan source conflict pitfall:** When SCCM co-management still controls the Windows Update
workload (workload slider not yet at Pilot Intune or Intune; see
[Workload Slider Migration](../co-management/02-windows-workload-sliders.md)), `dual-scan` source
conflict can cause WUfB driver/firmware updates to flap. Specifically:

- The Windows Update Agent receives both the SCCM-WSUS scan source AND the WUfB-cloud scan source
- For drivers/firmware specifically, the agent oscillates between the two source verdicts
- Symptom: drivers reported as "Available" then "Not applicable" then "Available" in successive
  scan cycles; user-visible flicker in Update history

**Mitigation:** Move the Windows Update workload to Pilot Intune or Intune (so SCCM no longer
controls scan source); OR set the WUfB driver/firmware policy to "Block automatic driver
delivery" until co-management workload migration completes; OR use Group Policy to disable
dual-scan during the transition (`HKLM\SOFTWARE\Policies\Microsoft\Windows\WindowsUpdate
\DisableDualScan = 1`) — note this disables ALL WUfB cloud scan, not just driver/firmware.
```
