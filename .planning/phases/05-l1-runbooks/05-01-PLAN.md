Title: `# Device Not Registered in Autopilot`

Prerequisites section (per D-01, D-02, D-03):
- Access to Intune admin center (https://intune.microsoft.com) with at least read permissions to Windows Autopilot devices
- Device serial number from the physical device label, BIOS screen, or user report
- Verified: the device is physically powered on and connected to a network (navigate to any website from the device browser to confirm)

Steps section (per D-05 linear only, D-09 full click-paths, D-10 user scripts, D-11 timing):
1. Open the Intune admin center (https://intune.microsoft.com)
2. Navigate to **Devices** > **Windows** > **Enrollment** > **Windows Autopilot devices**
3. In the search box, enter the full device serial number exactly as it appears on the device label. Do not use partial serial numbers.
4. If the device appears: it IS registered — this runbook does not apply. Return to the [initial triage decision tree](../decision-trees/00-initial-triage.md) to identify the correct failure scenario.
5. If no results appear: confirm the serial number. Ask the user to read the serial number from Settings > System > About on the device (if accessible) or from the device label. Re-search with the confirmed serial number.
6. (User script blockquote): "I'm checking your device registration in our management system. This may take a few minutes while I verify the details."
7. Check whether a hardware hash import was previously initiated: Navigate to **Devices** > **Windows** > **Enrollment** > **Windows Autopilot devices** > **Import** and check the import history for the device serial number.
8. If the device was expected to be OEM-registered (purchased from an authorized OEM with Autopilot registration): confirm with the procurement team or the device order record whether Autopilot registration was included in the purchase.
9. (User script blockquote at conclusion): "Your device is not yet registered in our Autopilot system. I'm going to escalate this to our technical team who can complete the registration. They will need [time estimate based on org SLA]. I'll update your ticket when registration is complete."

Escalation Criteria section (per D-04, data from TRE3):
Escalate to L2 if:
- Device serial number is confirmed correct but does not appear in Autopilot devices portal
- Hardware hash was never imported (or import status is unknown)
- Device was OEM-registered but does not appear (possible tenant mismatch)

Before escalating, collect (verbatim from TRE3 escalation data):
- Device serial number
- Device make and model
- Deployment mode (user-driven, pre-provisioning, or self-deploying)
- Whether hardware hash was previously imported (yes / no / unknown)
- Timestamp of the search
- Screenshot of the Autopilot devices search page showing no results

L2 forward-link: `See [L2 Runbooks](../l2-runbooks/) (available after Phase 6) for hardware hash re-import procedures.`

Cross-reference link to error codes: `For related error codes, see [MDM Enrollment Errors](../error-codes/01-mdm-enrollment.md).`

Title: `# Autopilot Profile Not Assigned`

Prerequisites section (per D-01, D-02):
- Access to Intune admin center (https://intune.microsoft.com) with permissions to view Autopilot devices and Azure AD groups
- Device serial number confirmed
- Verified: device IS registered in the Autopilot portal (search returned a result in **Devices** > **Windows** > **Enrollment** > **Windows Autopilot devices**)
- Verified: network connectivity is working (device can reach https://login.microsoftonline.com)

Steps section (per D-05, D-09, D-10, D-11):
1. Open the Intune admin center (https://intune.microsoft.com)
2. Navigate to **Devices** > **Windows** > **Enrollment** > **Windows Autopilot devices**
3. Search for the device serial number. Click on the device entry.
4. Check the **Profile** column. Note the current profile assignment status:
   - "Assigned" — profile is assigned. Skip to step 8.
   - "Not assigned" — profile is NOT assigned. Continue to step 5.
   - "Pending" — profile assignment is in progress. Skip to step 10.
5. Determine which Azure AD group the deployment profile targets. Navigate to **Devices** > **Enrollment** > **Windows Autopilot** > **Deployment Profiles** > select the expected profile > **Assignments**. Note the target group name.
6. Check if the device is a member of the target group. Navigate to **Groups** > search for the group name > **Members**. Search for the device.
7. If the device is NOT in the group: add it. For static groups, use **Members** > **Add members** and search by device name or serial. (User script blockquote): "I've added your device to the correct group. The profile should apply within a few minutes. I'll monitor this and let you know when it's ready."
8. If the device IS in the group but profile shows "Not assigned": this is beyond L1 scope — proceed to Escalation Criteria.
9. After adding the device to the group, wait for the profile to assign. Refresh the Autopilot devices page every 5 minutes. Static group assignment typically takes 1-5 minutes. Dynamic group evaluation takes 5-15 minutes for simple rules, and up to 24 hours for complex rules in large tenants (per [Profile Assignment lifecycle guide](../lifecycle/02-profile-assignment.md)). If status has not changed after 30 minutes, proceed to Escalation Criteria.
10. If status is "Assigned": trigger a sync on the device. Navigate to the device record in **Devices** > **Windows** > find the device > **Sync**. (User script blockquote): "Your device has a profile assigned. I'm triggering a sync now. Please restart the device and try the setup process again. This typically takes 5-10 minutes."
11. After sync and reboot, ask the user to retry OOBE. If provisioning proceeds normally, the issue is resolved.
12. If provisioning still fails after sync and reboot, proceed to Escalation Criteria.

Escalation Criteria (data from PRE1):
Escalate to L2 if:
- Device is in the correct group but profile still shows "Not assigned" after 30 minutes
- The wrong profile is assigned (requires admin correction — not L1 scope)
- Profile shows "Assigned" but device fails to apply it after sync and reboot
- You cannot determine which group the deployment profile targets

Before escalating, collect (verbatim from PRE1):
- Device serial number
- Group name the device was added to (or the group it was already in)
- Timestamp when group change was made (if applicable)
- Deployment mode (user-driven, pre-provisioning, or self-deploying)
- Current profile status shown in the Autopilot device record (screenshot if possible)
- Any error messages visible in the portal

L2 forward-link: `See [L2 Runbooks](../l2-runbooks/) (available after Phase 6) for assignment filter and Graph API investigation.`

First-mention glossary links per file for: [Autopilot](../_glossary.md#autopilot), [deployment profile](../_glossary.md#deployment-profile), [Azure AD](../_glossary.md#azure-ad), [OOBE](../_glossary.md#oobe), and other glossary terms.

Title: `# Network Connectivity Failure`

Prerequisites section (per D-01, D-02):
- Physical access to or screen-sharing with the affected device
- Device is powered on and at the OOBE screen (or can open a browser)
- Knowledge of the physical location and network connection type (Wi-Fi or Ethernet)

Steps section (per D-05 linear, D-09 full click-paths, D-10 user scripts, D-11 timing). This runbook is browser-only — no PowerShell Test-AutopilotConnectivity.

1. (User script blockquote): "I need to check a few websites from your device to determine if it can reach the services we need. I'll walk you through opening a browser and visiting some addresses."
2. Open a web browser on the device. If at the OOBE screen, press Shift+F10 to open a command prompt (if available), then type `start msedge` to launch Edge. If Shift+F10 is disabled by policy, ask the user to connect the device to a known-working network with internet access and retry OOBE.
   NOTE: Shift+F10 command prompt is an L1 exception — this is not PowerShell execution. It is the only way to open a browser at OOBE. If Shift+F10 is blocked, skip browser checks and proceed directly to step 8.
3. Navigate to **https://login.microsoftonline.com** in the browser. Expected result: the Microsoft sign-in page loads. If the page does not load, note the exact error message (connection refused, certificate error, timeout, proxy authentication required).
4. Navigate to **https://ztd.dds.microsoft.com** in the browser. Expected result: the page loads without a connection error (a blank page or JSON response is normal). If connection fails, note the error.
5. Navigate to **https://graph.microsoft.com** in the browser. Expected result: page loads or returns a JSON error (both confirm reachability). If connection fails, note the error.
6. Check the device clock: look at the time displayed on the device. If the OS clock is incorrect by more than 5 minutes, Azure AD token validation will fail. The device should sync to time.windows.com automatically when connected to the internet. If the clock is wrong and no internet is available, note this as a contributing factor.
7. Ask the user: "Are you connected via Wi-Fi or Ethernet? Is a VPN running? Are you on a corporate network, guest network, or personal network?" Record the answers.
8. If only Wi-Fi was tested and connectivity checks failed: ask the user to try a wired Ethernet connection if available. Some networks restrict Wi-Fi access differently than wired.
9. (User script blockquote at conclusion — if connectivity checks fail): "Your device cannot reach the services needed for automatic setup. This is a network configuration issue that our infrastructure team will need to resolve. I'm going to transfer this to them with the details I've collected."
10. For the full list of endpoints that must be reachable, see [Autopilot Network Endpoints](../reference/endpoints.md). Do NOT paste or list individual URLs in this runbook — the endpoints reference page is the canonical source.

Escalation Criteria:
NOTE: Network connectivity failures escalate to the **Infrastructure/Network team**, NOT to L2 engineering (per TRE1/TRE2 routing).

Escalate to Infrastructure/Network if:
- Device cannot reach https://login.microsoftonline.com (no internet or authentication proxy blocking)
- Device can reach general internet but not Autopilot-specific endpoints (firewall/proxy rule blocking)
- Device clock is incorrect and cannot sync (NTP blocked)
- All browser checks pass but OOBE still fails to connect (possible TLS inspection or certificate issue)

Before escalating, collect (verbatim from TRE1/TRE2):
- Device IP address and subnet (visible in Settings > Network if accessible, or from `ipconfig` at command prompt)
- Whether Wi-Fi or Ethernet is in use
- Proxy configured (yes / no / unknown)
- Browser error message shown for each failed endpoint
- Physical location of device (building, floor, network zone)
- Which specific endpoint(s) failed
- Whether the device clock is correct

L2 forward-link: `For endpoint-specific firewall rule requirements, see [Autopilot Network Endpoints](../reference/endpoints.md).`
