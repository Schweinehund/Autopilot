Version gate banner (standard).

Title: `# ESP Stuck or Failed`

Prerequisites section (per D-01, D-02):
- Access to Intune admin center (https://intune.microsoft.com) with permissions to view device enrollment status
- Device serial number from the device label or user report
- Verified: device IS registered in the Autopilot portal
- Verified: network connectivity is working (device reached the sign-in screen, which confirms basic connectivity)
- Know which ESP phase the device is in: "Setting up your device..." = device phase; "Setting up for [username]..." = user phase (per Phase 4 ESP tree ESD4 — screen text identification only, no registry)

After Prerequisites, add a section:

## How to Use This Runbook

"This runbook has three sections based on the ESP symptom. Go directly to the section that matches your situation:
- [Device Phase Steps](#device-phase-steps) — ESP shows 'Setting up your device...' and appears stuck
- [User Phase Steps](#user-phase-steps) — ESP shows 'Setting up for [username]...' and appears stuck
- [Error Code Steps](#error-code-steps) — ESP displayed an error code"

Then three sub-sections, each with its own numbered step sequence:

---

**## Device Phase Steps {#device-phase-steps}**

Entry condition: ESP shows "Setting up your device..." and has been on this screen for more than 30 minutes (per ESD5 threshold).

1. (User script blockquote): "Your device is in the initial setup phase. This process installs required security settings and applications. It normally takes 15 to 45 minutes. Since it has been over 30 minutes, I'm going to try a restart to see if that helps it continue."
2. Note the current time and what the ESP screen shows (which app or step name is displayed, if any).
3. Restart the device: press and hold the power button for 10 seconds, wait 30 seconds, then power on.
4. Wait for the device to boot back to the ESP screen. This typically takes 3-5 minutes.
5. Check if ESP has progressed past the point where it was previously stuck. Look at the step name or app name shown on the ESP screen.
6. If ESP has progressed: (User script blockquote): "The restart helped. Your device is continuing the setup process. I'll check in with you in 10 minutes to make sure everything completes. Please don't turn off or restart the device during this time." Set a 10-minute callback. If ESP completes and the user reaches the desktop, the issue is resolved.
7. If ESP is stuck at the same point after the reboot: do NOT reboot again. Proceed to Escalation Criteria.
8. Important: Only "required" apps block the ESP. "Available" apps do not. If the user reports the ESP is showing an app name, check Intune to see if that app is assigned as Required or Available. Navigate to **Devices** > **Windows** > find device > **Detected apps** or check the app assignment in **Apps** > **Windows** > [app name] > **Assignments**. If the blocking app is Available (not Required), this may indicate a misconfiguration — note this for L2.

---

**## User Phase Steps {#user-phase-steps}**

Entry condition: ESP shows "Setting up for [username]..." and has been on this screen for more than 60 minutes (per ESD7 threshold).

1. (User script blockquote): "Your device finished the initial setup and is now installing applications specific to your user account. This phase can take 5 to 15 minutes for most users, but can be longer if you have many applications assigned. Since it has been over 60 minutes, I'm going to try a restart."
2. Note the current time, the username shown, and which app or step name is displayed.
3. Restart the device: press and hold the power button for 10 seconds, wait 30 seconds, then power on.
4. Wait for the device to boot back and the user to sign in again. This typically takes 3-5 minutes plus sign-in time.
5. Check if the user phase has progressed.
6. If progressed: (User script blockquote): "The restart helped. Your setup is continuing. I'll check in with you in 15 minutes. Please don't restart the device." Set a 15-minute callback.
7. If stuck at the same point: proceed to Escalation Criteria.

---

**## Error Code Steps {#error-code-steps}**

Entry condition: ESP displayed an error code (typically a hex code like 0x8018xxxx or a descriptive message).

1. Note the exact error code and any message text shown on the ESP screen.
2. Look up the error code in the [ESP Error Code Table](../error-codes/03-esp-enrollment.md). Find the row matching your error code.
3. Follow the L1 Action column for that error code. The error table provides specific instructions for each code.
4. If the error code is not found in the table: note the exact code and proceed to Escalation Criteria.
5. (User script blockquote): "Your device encountered an error during setup. I've identified the error and I'm working on the resolution now." (Or, if escalating: "Your device encountered an error during setup that requires our technical team to investigate. I'm collecting the information they need right now.")

---

Escalation Criteria (shared across all sub-sections):

Escalate to L2 if:
- Device phase ESP is stuck at the same point after one reboot and 30+ minutes total wait
- User phase ESP is stuck at the same point after one reboot and 60+ minutes total wait
- Error code is not found in the ESP error code table
- Error code L1 action did not resolve the issue
- A "Required" app appears to be blocking ESP but the app assignment may be misconfigured

Before escalating, collect (from ESE4/ESE5):
- Device serial number
- Deployment mode (user-driven or pre-provisioning)
- Timestamp when ESP was first observed stuck
- ESP phase (device or user)
- Time spent on ESP before the reboot
- Time spent on ESP after the reboot
- Username used during enrollment (user phase only)
- App name shown on ESP screen at time of failure (if visible)
- Error code (if displayed)
- Screenshot of the ESP screen

L2 forward-link: `See [L2 Runbooks](../l2-runbooks/) (available after Phase 6) for ESP registry analysis and app conflict investigation.`

Cross-reference: `For ESP phase details, see [Enrollment Status Page Lifecycle Guide](../lifecycle/04-esp.md).`

Title: `# OOBE Fails Immediately`

A brief intro paragraph (1-2 sentences): "Use this runbook when the device fails during the initial Out-of-Box Experience before reaching the Enrollment Status Page. This covers crashes, freezes, BSODs, reboots to recovery, and OOBE completing standard Windows setup without Autopilot customization."

Prerequisites section (per D-01, D-02):
- Access to Intune admin center (https://intune.microsoft.com)
- Device serial number from the device label or user report
- A description of what the user saw on screen (exact error messages, screens shown, sequence of events)

Steps section (per D-05 linear, D-09 full click-paths, D-10 user scripts):

1. (User script blockquote): "I understand your device ran into a problem during the initial setup. I'm going to check a few things and then try one restart. If that doesn't resolve it, I'll escalate to our technical team with all the details."
2. Confirm the device is registered in the Autopilot portal. Open the Intune admin center (https://intune.microsoft.com). Navigate to **Devices** > **Windows** > **Enrollment** > **Windows Autopilot devices**. Search for the device serial number. If the device is NOT registered, this is a different issue — follow the [Device Not Registered runbook](01-device-not-registered.md) instead.
3. Confirm network connectivity. Ask the user if the device was connected to a network when the failure occurred. If network connectivity was not confirmed, follow the [Network Connectivity Failure runbook](04-network-connectivity.md) first.
4. Confirm the expected deployment mode. Ask the user or check the Autopilot device record: was this device supposed to use user-driven, pre-provisioning, or self-deploying mode? Note the expected mode for escalation data.
5. Document exactly what appeared on screen. Ask the user to describe the sequence of events:
   - What was the last screen they saw before the failure?
   - Did they see company branding or the standard Windows setup?
   - Was there an error code or message? (Write it down exactly.)
   - Did the device reboot, freeze, or display a blue screen?
6. Perform a single power cycle. Ask the user to: power off the device completely (hold power button 10 seconds), wait 30 seconds, then power on. Let the device boot back to OOBE.
7. If OOBE proceeds normally after the power cycle: monitor until the user reaches the desktop or ESP. The issue may have been transient. (User script blockquote): "The restart appears to have resolved the issue. Your device is continuing setup. Please let me know if you encounter any further problems."
8. If OOBE fails again in the same way: proceed to Escalation Criteria.
9. If OOBE now presents standard Windows setup (no company branding, no Azure AD sign-in prompt): this means the Autopilot profile did not load. Confirm the profile is assigned — follow the [Profile Not Assigned runbook](03-profile-not-assigned.md). This is likely a misroute, not an OOBE failure.

Escalation Criteria:
Escalate to L2 if:
- OOBE fails a second time after power cycle with the same symptoms
- Device displays a BSOD or enters recovery mode
- OOBE completes standard Windows setup despite having a confirmed Autopilot profile (profile loaded but OOBE did not apply it)
- Any failure pattern not covered by the ESP, Profile, or Network runbooks

Before escalating, collect (from TRE5):
- Device serial number
- Deployment mode (user-driven, pre-provisioning, or self-deploying)
- Timestamp of the failure
- Detailed description of what appeared on screen (in order)
- Sequence of events leading to the failure
- Whether the failure repeated after one power cycle
- Screenshot of the failure screen (if available)
- Whether company branding was displayed at any point during OOBE

L2 forward-link: `See [L2 Runbooks](../l2-runbooks/) (available after Phase 6) for OOBE log analysis and provisioning investigation.`
