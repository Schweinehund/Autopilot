**File: `docs/lifecycle/03-oobe.md`** (LIFE-04, 600-900 words -- longer due to 3 deployment modes)

Same frontmatter pattern and version gate banner as Tasks 1.

This is the most complex stage guide because it covers three deployment mode paths in one file (per CONTEXT.md locked decision). Include a Mermaid diagram (LR direction, diverge/reconverge pattern per RESEARCH.md Pattern 6).

11 sections:
1. **Context**: Stage 3 of 5. Where deployment mode selection happens. Depends on: Profile Assignment. Feeds into: ESP (Stage 4). Device powers on, OOBE screens begin, profile detected, mode-specific path begins.
2. **What the User / Technician Sees**: Three subsections:
   - **User-Driven**: Customized OOBE screens (company logo, custom text), Azure AD sign-in prompt.
   - **Pre-Provisioning**: First OOBE screen with Win+F12 option, blue "Windows Autopilot provisioning" screen.
   - **Self-Deploying**: Minimal screens, no credential prompt, automatic progression.
3. **What Happens**: Common path (5 steps: power on, regional settings, network, ZTD profile check, mode branch), then three subsections:
   - **User-Driven path**: User enters AAD creds, device joins AAD, MDM enrollment begins, ESP launches.
   - **Pre-Provisioning path**: Technician Win+F12, device-side ESP runs (policies, apps, certs), TPM attestation required, "Success" screen, reseal, ship to user, user completes user-driven OOBE.
   - **Self-Deploying path**: TPM attestation (no user cred), device joins AAD as device (no user affinity), MDM enrolls, ESP device phase runs, no user phase.

   Include TPM requirements table (3 rows: mode, TPM required yes/no, why). Per RESEARCH.md.

   Include Mermaid diagram (LR direction) showing the diverge/reconverge pattern per RESEARCH.md Pattern 6:
   ```
   START --> M{Deployment Mode?}
   M --> UD[User enters AAD creds]
   M --> PP[Technician Win+F12]
   M --> SD[TPM attestation]
   UD --> ESP4
   PP --> PPEND[Device ESP completes] --> RESEAL --> SHIP[Ship to user -> re-enters User-Driven]
   SD --> ESP4
   ```

4. **Behind the Scenes (L2 Note)**: `ztd.dds.microsoft.com` profile lookup, `login.microsoftonline.com` for AAD auth, `*.microsoftaik.azure.net` for TPM attestation. Link endpoints to `../reference/endpoints.md`.
5. **Success Indicators**: Profile detected at OOBE, correct mode activates, AAD join completes (user-driven/self-deploy), reseal succeeds (pre-provisioning).
6. **Failure Indicators**: No profile at OOBE, TPM attestation fails (pre-prov/self-deploy), network unreachable, wrong deployment mode activates. Forward-links.
7. **Typical Timeline**: OOBE to mode detection: 1-3 minutes. Pre-provisioning device phase: 15-60 min depending on apps. Self-deploying: similar without user phase.
8. **Watch Out For**: (a) TPM not ready for pre-prov/self-deploy, (b) Network/proxy blocking required endpoints, (c) Hybrid join requiring DC reachability.
   - Brief Autopilot Reset note per CONTEXT.md: "Autopilot Reset re-enters at Stage 3, skipping Stages 1-2."
9. **Tool References**: `Get-AutopilotRegistrationState`, `Get-TPMStatus`, `Test-AutopilotConnectivity`. Further Reading URLs.
10. **Navigation**: "Previous: [Stage 2: Profile Assignment](02-profile-assignment.md) | Next: [Stage 4: ESP](04-esp.md)"
11. **Version History**: 2026-03-14 | Initial version.

APv2 Note callout: APv2 has no pre-provisioning mode. No Win+F12. Technician flow handled via assignment policy.

Edge cases per CONTEXT.md: Hybrid join brief note in this stage.
