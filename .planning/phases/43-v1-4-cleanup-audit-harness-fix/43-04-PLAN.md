    Helper for maintaining `scripts/validation/v1.4-audit-allowlist.json` and `v1.4.1-audit-allowlist.json` supervision_exemptions[] pins.

    ## Modes

    ### `--report` (advisory)
    Prints a one-screen report of: pinned / un-pinned Tier-1 / un-pinned Tier-2 / stale-pin counts. No writes.

    ### `--emit-stubs`
    Emits JSON fragments for un-pinned Tier-1 occurrences to stdout (for human reason-filling before merging into sidecar). Side-channel writes un-pinned Tier-2 occurrences to `scripts/validation/suspected-regressions.txt` for human review.

    ### `--self-test`
    Dogfood mode — asserts classifier reproduces Phase 43's hand-authored Tier-1 pin set. Used as CI gate after pin authoring.

    ## Two-Tier Discrimination (D-11)

    - **Tier 1 (stub-eligible):** occurrence line OR two preceding lines contain `\b(iOS|Apple|ADE|macOS|MDM|cross-platform)\b` (case-insensitive), OR occurrence is inside an HTML-comment block.
    - **Tier 2 (suspected regression):** bare occurrences failing Tier 1. Emitted to `suspected-regressions.txt`. **Human must explicitly promote to a pin with written justification** — the helper NEVER auto-pins Tier 2.

    ## Why the Helper Never Auto-Pins (D-11)

    Auto-pinning Tier-2 occurrences would mask real content regressions (bare "supervision" in Android docs is a Phase 34 D-03 content bug — not a pin gap). Masking regressions is a worse failure than mild pin-drift because the silent mask accretes over many commits before surfacing.

    ## Install-Time Invocation

    ```bash
    node scripts/validation/regenerate-supervision-pins.mjs --report
    node scripts/validation/regenerate-supervision-pins.mjs --emit-stubs
    node scripts/validation/regenerate-supervision-pins.mjs --self-test
    ```

    ## CI Integration

    See `.github/workflows/audit-harness-integrity.yml` `pin-helper-advisory` job (advisory only per D-14/D-15 — never fails build).

    ## Relation to `v1.4.1-audit-allowlist.json`
