**01-autopilot-reset.md (WDLC-01):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: APv1`, `audience: both`, `platform: Windows` (per D-21: audience both since L1 handles basic resets)
- Version gate: "This guide covers Windows Autopilot Reset (APv1 only). Autopilot Reset is NOT available for APv2 (Device Preparation). For APv2, use Wipe or Retire. See [APv1 vs APv2](../apv1-vs-apv2.md)."
- Two main sections: "Local Autopilot Reset" and "Remote Autopilot Reset"
- Local reset trigger: Lock screen → CTRL + WIN + R → sign in with local admin account
- Remote reset path: Intune admin center > Devices > All devices > select device > ... > Autopilot Reset
- Prerequisites: WinRE enabled (`reagentc.exe /enable`), `DisableAutomaticReDeploymentCredentials` policy set to Allow for local, MDM managed + Entra joined for remote, Intune Service Administrator role for remote
- Data preservation table: Wi-Fi credentials = Preserved, Provisioning packages = Preserved, Entra device membership = Preserved, MDM enrollment = Preserved, SCEP certificates = Preserved, Personal files/apps/settings = REMOVED, User profile = REMOVED
- Explicit callout: "Autopilot Reset is NOT supported for Microsoft Entra hybrid joined devices or Surface Hub devices. A full Wipe is required for hybrid devices."
- Timing difference: "Local reset happens immediately. Remote reset initiates on the next Intune check-in, not immediately."
- Post-reset behavior: "Device goes to lock screen (NOT OOBE). User signs in. No Autopilot setup re-runs."
- Remote reset note: "Primary user and Entra device owner are removed on remote reset. Next sign-in sets new primary user. Local reset does NOT update primary user automatically."
- L1 Action callout: "> **L1 Action:** Confirm with user the device is plugged in and connected to network before initiating remote reset. Remote reset fires at next Intune check-in — instruct user to leave device powered on."
- L2 Note callout: "> **L2 Note:** If reset fails, check WinRE status with `reagentc.exe /info`. If WinRE is disabled, enable with `reagentc.exe /enable` and retry."
- See Also links to: 02-retire-wipe.md, ../decision-trees/05-device-lifecycle.md, ../lifecycle/04-esp.md, ../apv1-vs-apv2.md

**02-retire-wipe.md (WDLC-02):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: both`, `platform: Windows`
- Title: "Device Retirement, Wipe, and Removal Actions"
- Five-action comparison table with columns: Action, What It Does, What Is Preserved, Enrollment State After, When to Use
  - Autopilot Reset: Removes personal files re-applies org settings | Wi-Fi SCEP certs Entra membership MDM enrollment | Stays enrolled | Ownership transfer within same org (APv1 only)
  - Retire: Removes org data apps policies | Personal data user files | Unenrolled from Intune | BYOD offboarding employee departure
  - Wipe: Factory reset | Nothing (optionally Wi-Fi state) | Unenrolled | Lost/stolen full decommission hybrid devices needing reset
  - Fresh Start: Removes OEM bloatware + apps | User documents (Home folder) | Stays Entra joined re-enrolls MDM | Cleaning up corporate device without losing user files
  - Delete (Deregister): Removes Intune record only | Nothing (physical device unchanged) | Unenrolled + Autopilot record removed | Stale records cleanup pre-tenant-migration
- Link to decision tree: "Not sure which action to use? See [Device Lifecycle Decision Tree](../decision-trees/05-device-lifecycle.md)"
- Step-by-step for each action with Intune portal navigation paths
- Retire path: Intune admin center > Devices > All devices > select device > Retire
- Wipe path: Intune admin center > Devices > All devices > select device > Wipe (option to retain enrollment state and user account)
- Fresh Start path: Intune admin center > Devices > All devices > select device > Fresh Start
- Delete path: Intune admin center > Devices > All devices > select device > Delete
- L1 Action callout: "> **L1 Action:** For BYOD offboarding, use Retire (preserves personal data). NEVER use Wipe on a personal device — this erases everything including personal photos and documents."
- L2 Note callout: "> **L2 Note:** If Retire fails to remove org data, check that the device has an active network connection and has checked in recently. Stale devices (no check-in for 30+ days) may need manual org data removal."

**03-re-provisioning.md (WDLC-03):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: both`, `platform: Windows`
- Title: "Device Re-Provisioning: Ownership Transfer and Re-Enrollment"
- Ownership transfer scenario walkthrough: Device assigned to departing employee → Retire (removes org data) → Wipe if clean slate required → re-import hardware hash if APv1 → assign to new user → user goes through enrollment
- Re-enrollment prerequisites checklist (checkboxes):
  - [ ] Old user's Entra account no longer the primary user
  - [ ] Device removed from old user's group membership (if targeted by group)
  - [ ] Autopilot profile assigned to the correct group for new user's context
  - [ ] Device limit per user not exceeded for new user (Entra ID > Devices > Device settings)
  - [ ] For APv1: verify hardware hash still current (no hardware changes since last import)
  - [ ] For APv2: verify device meets Windows 11 22H2+ requirement
- Steps section: Step-by-step for each re-provisioning scenario (same-org transfer, device refresh, hardware replacement)
- L1 Action callout: "> **L1 Action:** Before re-provisioning, confirm the new user's department and location — this determines which Autopilot profile and app assignments apply. Escalate to L2 if the device is hybrid Entra joined."
- See Also: 01-autopilot-reset.md, 02-retire-wipe.md, ../admin-setup-apv1/04-dynamic-groups.md

**04-tenant-migration.md (WDLC-04):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows` (per D-23: audience admin — project-level operation)
- Title: "Tenant-to-Tenant Device Migration"
- Version gate: "Tenant migration is not a formally supported Intune scenario — this is a best-effort process documented from official Microsoft support guidance."
- Two methods documented:
  - Online migration (tight timeline, days): Collect hashes → delete from Tenant A → import to Tenant B → Reset this PC (NOT Autopilot Reset)
  - Offline migration (large batches, weeks): Deregister in batches → create JSON profile for Tenant B → deploy to `%windir%\provisioning\autopilot\AutoPilotConfigurationFile.json` → full wipe
- Critical warnings box: "Do NOT use Autopilot Reset during tenant migration — it leaves the device managed by the old tenant. Use 'Reset this PC' in Windows Settings or the Intune Wipe action." and "500-device batch limit for hash upload" and "No undo after deletion from Tenant A" and "Leaving too much time between removal from Tenant A and import to Tenant B can result in unmanaged devices"
- Hardware hash collection: `Get-WindowsAutoPilotInfo -online` or via Configuration Manager
- See Also: 02-retire-wipe.md, ../reference/endpoints.md, ../admin-setup-apv1/01-hardware-hash-upload.md
