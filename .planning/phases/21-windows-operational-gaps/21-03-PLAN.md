- Title: "Conditional Access Enrollment Timing: The Compliance Chicken-and-Egg Problem"
- Version gate: "This guide covers the Windows enrollment scenario. macOS Conditional Access enrollment timing will be documented in Phase 23. See also: [Compliance Policy Timing](compliance-timing.md)."

- ## The Problem (narrative explanation):
  Explain the chicken-and-egg scenario step by step:
  1. Admin creates Conditional Access policy: "Require compliant device" for all cloud apps
  2. New device starts Autopilot enrollment → device is not yet evaluated for compliance
  3. CA policy evaluates device → "Not compliant" (because not yet evaluated) → blocks access
  4. Enrollment requires access to Intune service → blocked by CA → enrollment fails
  5. Without enrollment, device can NEVER become compliant → permanent loop

  Include a Mermaid sequence diagram:
  ```mermaid
  sequenceDiagram
    participant Device
    participant Entra as Entra ID
    participant CA as Conditional Access
    participant Intune
    Device->>Entra: Join request
    Entra->>Device: Joined
    Device->>Intune: MDM enrollment request
    Intune->>CA: Check Conditional Access policies
    CA->>CA: Evaluate: Is device compliant?
    CA-->>Intune: BLOCKED (device not compliant)
    Intune-->>Device: Enrollment failed
    Note over Device: Cannot become compliant without enrollment
  ```

- ## Built-In Mitigations:
  - "Microsoft Intune Enrollment" cloud app is excluded by default from CA compliance requirement policies
  - Company Portal app has built-in exclusions from certain CA grant controls
  - These exclusions solve the enrollment loop for the COMMON case — no admin action needed IF defaults are intact

- ## How Admins Break It (3 scenarios):
  1. Creating a CA policy that EXPLICITLY includes "Microsoft Intune Enrollment" or "Microsoft Intune" cloud app in the assignment and requires device compliance
     > **What breaks if misconfigured:** Every new device fails enrollment. Existing devices unaffected. Error appears as authentication failure during OOBE, not as a compliance error — making root cause non-obvious.
  2. Creating an "All cloud apps" policy with "Require compliant device" and no exclusion list
     > **What breaks if misconfigured:** Same enrollment loop as scenario 1. The "All cloud apps" target implicitly includes Intune Enrollment unless explicitly excluded.
  3. Manually removing the default Intune Enrollment exclusion from an existing CA policy
     > **What breaks if misconfigured:** Breaks enrollment for all new devices immediately. No warning in the CA policy editor.

- ## Resolution Patterns:
  1. **Exclude Intune Enrollment app from compliance CA policies:** In CA policy > Cloud apps > Exclude > select "Microsoft Intune Enrollment" and "Microsoft Intune"
  2. **Use Report-only mode:** When rolling out new CA policies, set to Report-only first. Monitor Sign-in logs for "What if" results before switching to On.
  3. **Stage compliance rollout:** Set compliance policy action to "Mark device as noncompliant" with a grace period (minimum 0.25 days / 6 hours) rather than immediately. This gives the device time to enroll and evaluate before enforcement.
  4. **Use device filters:** Target CA policies to specific device groups that are already enrolled, excluding the "all devices" scope during enrollment.

- ## Verification:
  - [ ] CA policy shows "Microsoft Intune Enrollment" in Exclude list
  - [ ] New test device completes enrollment without auth errors
  - [ ] CA Sign-in logs show "Not applied" for Microsoft Intune Enrollment app during device enrollment

- ## Configuration-Caused Failures table:
  | Misconfiguration | Symptom | Runbook |
  |------------------|---------|---------|
  | Intune Enrollment included in compliance CA policy | New devices fail enrollment with auth error at OOBE | Remove Intune Enrollment from CA policy scope |
  | All cloud apps + require compliance, no exclusions | Same as above | Add Intune Enrollment to exclusion list |
  | Grace period set to 0 days | Compliance enforcement immediate; devices blocked before first evaluation completes | Set minimum 0.25 days grace period |

**security-baseline-conflicts.md (WSEC-02):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "Security Baseline Interactions with Autopilot Provisioning"
- Version gate: "This guide covers security baseline conflicts that affect both APv1 and APv2 Autopilot deployments."
- ## Known Conflict Areas (table):
  | Security Feature | Conflict with Autopilot | Timing Issue | Mitigation |
  |-----------------|------------------------|--------------|-----------|
  | BitLocker CSP | BitLocker policy "kicks in too late" — may not apply during ESP device phase | Policy received but encryption starts after ESP completes | Use a separate BitLocker configuration profile targeted to device group, not the security baseline. This ensures BitLocker is configured independently of baseline application order. |
  | LAPS (Local Admin Password Solution) | LAPS policy not applied until user phase begins | Device phase completes without LAPS-managed local admin password | Accept timing gap OR use pre-provisioning mode where technician phase applies LAPS during device phase |
  | AppLocker / WDAC (Windows Defender Application Control) | Can block Intune Management Extension (IME) process itself | IME blocked = all Win32 app installs fail during ESP | Always add IME processes to allow list: `Microsoft.Management.Services.IntuneWindowsAgent.exe`, `IntuneWindowsAgent.exe` |
  | Reboot Policies (forced restart, maintenance window) | Immediate reboot setting conflicts with ESP flow | ESP interrupted by forced reboot — restarts from scratch or fails | Defer reboots until post-ESP using maintenance windows. Set reboot grace period to at least 120 minutes. |

- ## KB5065848 Known Issue (2025):
  "Windows devices may get stuck on 'Identifying' screen during Autopilot and BitLocker may not apply correctly. This is a known issue tracked in KB5065848."
  - Symptoms: Device stuck at "Identifying your device" during OOBE; BitLocker not applied post-enrollment
  - Mitigation: Ensure the KB fix is installed in the Windows image or apply via Windows Update during ESP (if enabled)
  - Status: Documented in Microsoft Learn Known Issues for Autopilot

- ## General Guidance:
  - "Apply security baselines AFTER successful enrollment, not as part of the initial ESP blocking configuration."
  - "Use Intune configuration profiles for individual security settings (BitLocker, Firewall, etc.) rather than monolithic security baselines during initial deployment."
  - "Test every security baseline change with a pilot Autopilot deployment before fleet-wide rollout."

  > **What breaks if misconfigured:** Security baseline applied during ESP enables a setting that requires reboot or blocks IME. ESP either fails or loops. Admin sees "Some settings could not be applied" in ESP but cannot determine which baseline setting caused it.

- ## Configuration-Caused Failures table:
  | Misconfiguration | Symptom | Runbook |
  |------------------|---------|---------|
  | WDAC policy blocks IME | Win32 apps fail to install during ESP; timeout | Add IME to WDAC allow list |
  | Forced reboot in baseline | ESP interrupted; deployment restarts or fails | Defer reboots 120+ minutes |
  | BitLocker via baseline only | BitLocker not applied; device non-compliant post-enrollment | Use separate BitLocker profile |

- ## See Also: ca-enrollment-timing.md, compliance-timing.md, ../lifecycle/04-esp.md, ../admin-setup-apv1/10-config-failures.md

**compliance-timing.md (WSEC-03):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "Compliance Policy Timing and State Transitions"
- Version gate: "This guide expands on the compliance verification in [Post-Enrollment Verification](../lifecycle/05-post-enrollment.md) with detailed timing and state transition information."

- ## Compliance State Timeline Post-Enrollment:
  | Time After Enrollment | Expected State | What Is Happening | Admin Action |
  |----------------------|----------------|-------------------|--------------|
  | 0-15 minutes | Not evaluated | Compliance check not yet run; Intune collecting device inventory | Wait — this is normal. Do NOT escalate. |
  | 15-30 minutes | Compliant or Non-compliant | First compliance evaluation completes | If Non-compliant, investigate which policy fails |
  | Up to 8 hours | Final state | Full Intune inventory sync (app status, hardware inventory) | Portal now shows complete device state |

- ## The Three Compliance States (critical distinction):
  1. **Not evaluated** — Expected waiting state post-enrollment. Device has enrolled but compliance policies have not yet been checked. This is NOT a failure. Appears as gray or "unknown" in portal.
  2. **Non-compliant** — Policy evaluated and device FAILS one or more compliance requirements. Investigate. Check which policy and which setting fails in Devices > [device] > Device compliance.
  3. **Compliant** — Policy evaluated and device passes all requirements. Healthy state.
  > **Admin Note:** "Not evaluated" and "Non-compliant" both appear as non-green states in the portal. New admins frequently confuse them. Always check the timestamp — if less than 30 minutes since enrollment, "Not evaluated" is expected.

- ## Grace Period Configuration:
  - Minimum grace period: 0.25 days (6 hours)
  - Grace period = time after first Non-compliant evaluation before enforcement actions trigger
  - Actions that trigger after grace period: email notification, push notification, mark non-compliant (for CA enforcement), remotely lock, retire
  - Recommendation: Set grace period to at least 1 day for new deployments. This gives admins time to investigate and remediate before users are affected.
  > **What breaks if misconfigured:** Grace period set to 0.25 days with "Retire" as enforcement action. Device enrolls, evaluates as non-compliant due to pending Windows update, and is auto-retired 6 hours later — before admin can intervene. User loses access to all corporate resources.

- ## Conditional Access Integration:
  - Once "Non-compliant" state is set (after grace period), CA policies with "Require compliant device" immediately block access for that device
  - During the "Not evaluated" window (0-30 min post-enrollment), CA behavior depends on the "Mark devices with no compliance policy assigned as" setting (Compliant or Not compliant). Default: Not compliant.
  - Cross-reference: [Conditional Access Enrollment Timing](ca-enrollment-timing.md) for the enrollment-specific chicken-and-egg problem

- ## Compliance Evaluation Schedule (ongoing):
  - Every 8 hours: automatic Intune check-in
  - Manual sync: user can trigger from Company Portal or Settings > Accounts > Access work or school > Info > Sync
  - After policy change: re-evaluation within 15 minutes of next check-in (not immediate)
  - After device restart: triggers check-in and re-evaluation

- ## Configuration-Caused Failures table:
  | Misconfiguration | Symptom | Runbook |
  |------------------|---------|---------|
  | Grace period 0 days with retire action | Device auto-retired before admin can investigate | Set grace period >= 1 day |
  | "No policy" marked as non-compliant | All devices without explicit policy are blocked by CA | Change default or assign compliance policy to all device groups |
  | Conflicting compliance policies | Device flips between compliant and non-compliant | Review policy assignment scope overlap |

- ## See Also: ca-enrollment-timing.md, security-baseline-conflicts.md, ../lifecycle/05-post-enrollment.md
