- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows` (per D-15: applies_to both since doc covers migration FROM APv1 TO APv2)
- Title: "APv1-to-APv2 Migration: Parallel Deployment Playbook"
- Version gate: "This guide covers migration from Windows Autopilot (classic/APv1) to Autopilot Device Preparation (APv2). This is a coexistence model, not a one-shot cutover. For feature differences, see [APv1 vs APv2](../apv1-vs-apv2.md)."
- This doc models ongoing coexistence per D-18 — both frameworks running in same tenant for weeks/months with different device populations.

- ## Migration Prerequisites (per D-14):
  - Source environment requirements:
    - Existing APv1 deployment with known profile assignments
    - Inventory of which device groups use which APv1 profiles
    - Documented list of features in use (pre-provisioning, self-deploying, hybrid join) for blocker assessment
  - Target environment requirements:
    - Windows 11 22H2 or later on target devices
    - Enrollment Time Grouping security group created
    - APv2 deployment policy created and tested with pilot devices

- ## APv1-to-APv2 Readiness Checklist:
  - [ ] All target devices running Windows 11 22H2 or later
  - [ ] No requirement for hybrid Entra join on target devices (APv2 does not support hybrid join)
  - [ ] No requirement for pre-provisioning / white glove on target devices (APv2 does not support)
  - [ ] No requirement for self-deploying mode for kiosks on target devices (APv2 does not support)
  - [ ] Maximum 25 apps during deployment acceptable (APv2 limit vs APv1's 100 app limit)
  - [ ] Hardware hash pre-staging NOT required for target devices (APv2 benefit — uses corporate identifiers instead)
  - [ ] Enrollment Time Grouping security group created and configured in APv2 policy
  - [ ] Existing APv1 hardware hash registration will be removed OR APv1 profile deleted from device group before APv2 policy assigned

  Each unchecked item is a **blocker** — if the answer is "No" or "Not sure", that device category stays on APv1.

- ## Feature Gap Matrix:
  "For the full feature comparison, see [APv1 vs APv2 Feature Comparison](../apv1-vs-apv2.md#feature-comparison). The table below adds migration blocker assessment." (per D-16: link to existing table, add blocker column)
  | Feature | APv1 | APv2 | Migration Blocker? |
  |---------|------|------|--------------------|
  | Hybrid Entra join | Yes | No | YES — devices requiring hybrid join MUST stay on APv1 |
  | Pre-provisioning | Yes | No | YES — if technician pre-staging workflow is required |
  | Self-deploying mode | Yes | No | YES — kiosk/shared devices using self-deploying must stay on APv1 |
  | Max apps during ESP | 100 | 25 | MAYBE — only a blocker if >25 apps are Required during deployment |
  | Hardware hash pre-staging | Required | Not required | NO — APv2 is actually easier here |
  | Windows 10 support | Yes | No | YES — Windows 10 devices must stay on APv1 |
  | Near real-time monitoring | No | Yes | NO — APv2 is better |
  | Enrollment Time Grouping | No | Yes | NO — APv2 benefit |

- ## Steps (Coexistence Model per D-18):
  ### Step 1: Identify device categories for migration
  Categorize existing fleet by APv2 compatibility. Example categories: "New purchases" (migrate first), "Existing Win11 22H2+ cloud-only" (migrate second), "Hybrid join / Win10 / kiosk" (stay APv1).
  > **What breaks if sequenced incorrectly:** Migrating hybrid-join devices to APv2 fails silently. Device enrolls but never completes hybrid join. Admin discovers weeks later when Conditional Access blocks access. (per D-13)

  ### Step 2: Create APv2 deployment policy for pilot group
  1. Navigate to **Intune admin center** > Devices > Enrollment > Windows > Autopilot device preparation
  2. Create policy targeting the Enrollment Time Grouping security group
  3. Configure apps (max 25), compliance settings, and user affinity

  ### Step 3: Remove APv1 registration for pilot devices
  "When BOTH an APv1 profile AND an APv2 policy apply to a device, APv1 silently wins. You MUST remove the APv1 registration before APv2 can take effect."
  > **What breaks if sequenced incorrectly:** Admin assigns APv2 policy but forgets to remove APv1 registration. Device uses APv1 silently. Admin believes APv2 is working but it never activates. Discovery happens when APv2-specific features (monitoring, Enrollment Time Grouping) don't appear.
  - Option A: Delete APv1 hardware hash from Intune > Devices > Windows enrollment > Devices
  - Option B: Remove device from the APv1 profile's assigned group (profile no longer targets it)

  ### Step 4: Wipe and re-provision pilot devices with APv2
  ### Step 5: Validate pilot results (30-day soak)
  ### Step 6: Expand to next device category
  Repeat Steps 3-5 for each category. No required timeline — migrate by category when ready.

- ## Rollback/Recovery (per D-12):
  "APv1-to-APv2 rollback is fleet-level, not device-level."
  - To stop migration: stop deregistering devices from APv1. New devices continue on APv1.
  - For devices already on APv2: Wipe → re-register hardware hash → re-assign APv1 profile → user re-provisions
  - There is no "switch back to APv1" button per-device. Rollback requires wipe + re-registration.

- ## Verification:
  - [ ] Pilot devices complete APv2 enrollment successfully
  - [ ] Enrollment Time Grouping populates with pilot devices
  - [ ] APv2 monitoring dashboard shows deployment status
  - [ ] Remaining APv1 devices unaffected by APv2 policy

**imaging-to-autopilot.md (WMIG-02):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "On-Premises Imaging to Autopilot Migration Guide"
- Version gate: "This guide covers migration from MDT and SCCM/ConfigMgr imaging to Windows Autopilot (APv1 or APv2)."

- ## MDT Retirement Notice (important context):
  "Microsoft Deployment Toolkit (MDT) receives no further updates, fixes, or support after the first ConfigMgr release following October 2025. Downloads have been removed from official channels. MDT-based task sequences must be replaced."
  > **Admin Note:** If your organization still uses MDT, plan the transition now. There is no extended support timeline.

- ## Migration Prerequisites (per D-14):
  - Source environment requirements:
    - Complete inventory of task sequence applications (name, version, installer type: MSI/EXE/custom)
    - Documented task sequence customizations (scripts, drivers, OS settings)
    - SCCM client version documented (if SCCM OSD is source, for "Autopilot for Existing Devices" scenario)
    - GPO baseline exported as XML (feeds WMIG-03 GPO-to-Intune migration)
  - Target environment requirements:
    - Intune tenant configured with Autopilot per [Admin Setup Overview](../admin-setup-apv1/00-overview.md)
    - Win32 Content Prep Tool downloaded (Microsoft/Microsoft-Win32-Content-Prep-Tool on GitHub)

- ## Migration Paths (table):
  | Source | Target | Method | Notes |
  |--------|--------|--------|-------|
  | MDT task sequence | Autopilot user-driven (APv1/APv2) | Replace TS apps with Intune Win32 apps; re-image devices to clean Windows | Most common migration path |
  | MDT task sequence | ConfigMgr OSD | Direct evolution of MDT; requires existing SCCM site | Only if organization already has ConfigMgr investment |
  | SCCM OSD | Autopilot for Existing Devices | Create JSON profile; embed in task sequence; runs Autopilot on existing hardware | Limitation: JSON supports user-driven Entra join and hybrid join ONLY. Self-deploying and pre-provisioning NOT supported via JSON. |
  | Manual imaging (USB/WDS) | Autopilot | Collect hardware hash during imaging; wipe to clean state; register in Intune | For small environments without MDT/SCCM |

- ## Steps: App Packaging Migration
  For each task sequence application:
  ### Step 1: Identify installer format (MSI, EXE, custom script)
  ### Step 2: Package as Win32 (.intunewin) using Microsoft Win32 Content Prep Tool
  1. Download and extract the Win32 Content Prep Tool
  2. Run: `IntuneWinAppUtil.exe -c <source_folder> -s <setup_file> -o <output_folder>`
  3. Upload .intunewin file to Intune admin center > Apps > Windows > Add > Windows app (Win32)
  ### Step 3: Set detection rule
  - MSI: use product code (auto-detected)
  - EXE: use file version check (preferred) or registry key
  - See [Win32 App Packaging Best Practices](win32-app-packaging.md) for detection rule priority
  ### Step 4: Configure assignment
  - Assign as Required to device group for ESP tracking
  - Assign as Available for optional apps (not tracked by ESP)
  ### Step 5: Declare dependencies if install order matters
  > **What breaks if sequenced incorrectly:** App requires .NET Framework runtime. Runtime not declared as dependency. App installs before runtime. App fails. ESP retries until timeout.

- ## Autopilot for Existing Devices (SCCM path):
  - JSON profile location: `%windir%\provisioning\autopilot\AutoPilotConfigurationFile.json`
  - JSON only supports: user-driven Entra join, user-driven hybrid Entra join
  - JSON does NOT support: self-deploying, pre-provisioning
  - Requires: full device wipe as part of task sequence (not upgrade-in-place)

- ## Rollback/Recovery (per D-12):
  "Maintain imaging infrastructure in parallel during migration. Do not decommission MDT/SCCM until Autopilot is validated for all device categories."
  - If Autopilot fails for a device category: fall back to imaging for that category
  - Keep at least one task sequence operational as emergency fallback for 90 days post-migration

**gpo-to-intune.md (WMIG-03):**
- Frontmatter: `last_verified: 2026-04-13`, `review_by: 2026-07-12`, `applies_to: both`, `audience: admin`, `platform: Windows`
- Title: "GPO-to-Intune Policy Migration Guide"
- Version gate: "This guide covers migrating Group Policy Objects to Intune using Group Policy Analytics. Applies to both APv1 and APv2 managed devices."
- CRITICAL: This doc is organized by OUTCOME, not by GPO setting name (per D-17).

- ## Migration Prerequisites (per D-14):
  - GPO baseline exported as XML from Group Policy Management Console (GPMC.msc)
  - Unicode encoding for XML files (required for GPA import)
  - Max 4 MB per XML file for GPA import
  - Inventory of which GPOs apply to which OUs (for targeting in Intune)

- ## Steps: Using Group Policy Analytics
  ### Step 1: Export GPOs as XML
  1. Open GPMC.msc (Group Policy Management Console)
  2. Right-click each GPO > Save report > XML format
  3. Ensure file encoding is Unicode (not ANSI)

  ### Step 2: Import into Group Policy Analytics
  1. Navigate to **Intune admin center** > Devices > Manage devices > Group Policy analytics
  2. Click **Import**
  3. Select XML file (max 4 MB)
  4. Wait for analysis (typically 30-60 seconds)

  ### Step 3: Review MDM Support Report
  GPA generates a per-setting report with three categories:
  - **Ready for migration** — Matching Intune setting exists in Settings Catalog. Migrate directly.
  - **Not supported** — No MDM equivalent exists. Evaluate if the setting is still needed.
  - **Deprecated** — Applies to old Windows versions or deprecated Edge. Remove from migration scope.
  The MDM Support percentage is the ratio of "Ready" settings to total settings.

  ### Step 4: Migrate supported settings
  For "Ready" settings: GPA offers a "Migrate" button that creates a Settings Catalog profile with the matching Intune settings pre-populated.
  > **What breaks if sequenced incorrectly:** Admin migrates GPO settings to Intune but leaves GPO active in AD. Both GPO and Intune apply the same settings. If values differ, the "last write wins" behavior is unpredictable. Always remove the GPO from the OU scope AFTER confirming the Intune policy applies correctly.

  ### Step 5: Handle unsupported settings
  For "Not supported" settings, evaluate by outcome:
  - If the outcome is still needed: use Custom OMA-URI profile in Intune (advanced, requires CSP knowledge)
  - If the outcome is obsolete: add to "Do not migrate" list

- ## Outcome-Based Policy Organization (per D-17):
  "Migrate by business outcome, not by GPO setting name."

  | Business Outcome | GPO Equivalent | Intune Implementation | Notes |
  |-----------------|----------------|----------------------|-------|
  | Require device encryption | BitLocker GPO | Endpoint security > Disk encryption > BitLocker profile | Or Settings Catalog > BitLocker |
  | Enforce password complexity | Computer Config > Policies > Windows Settings > Security > Account Policies | Settings Catalog > DeviceLock settings | Or Endpoint security > Account protection |
  | Control USB access | Administrative Templates > System > Removable Storage Access | Settings Catalog > Removable Storage settings | Block read/write separately |
  | Restrict Control Panel | User Config > Administrative Templates > Control Panel | Settings Catalog > ControlPanel settings | User-targeted in Intune |
  | Configure Windows Update | Computer Config > Administrative Templates > Windows Update | Update rings for Windows 10/11 | Native Intune feature; no GPO needed |
  | Restrict local logon | Security Settings > Local Policies > User Rights | Custom OMA-URI (not in Settings Catalog) | Advanced — may need WDAC instead |

- ## "Do Not Migrate" List (per D-17):
  | GPO Category | Why Not Migrate | Modern Alternative |
  |-------------|-----------------|-------------------|
  | Login scripts | Scripts run at domain logon; no domain in cloud-only | Intune PowerShell scripts or Remediation scripts |
  | Folder redirection | Server-based paths; not applicable to cloud | OneDrive Known Folder Move |
  | IE Enhanced Security | IE deprecated | N/A — remove |
  | WMI filter-based GPOs | No Intune equivalent for WMI-based targeting | Intune filters (device property based) or assignment groups |
  | Environment-specific GPOs | Domain paths, printer mappings to on-prem servers | Intune-native printer deployment, Universal Print |
  | Drive mapping GPOs | Mapped drives to on-prem file servers | OneDrive / SharePoint / Intune scripts |

- ## Rollback/Recovery (per D-12):
  "Keep GPOs active in AD alongside Intune policies during transition. Remove GPOs only after confirming Intune policies apply correctly and after devices are no longer domain-joined (for cloud-only migration)."
