**Section 1: IntuneMacODC Comprehensive Collection** (primary tool per D-11)

Explain that IntuneMacODC is a Microsoft-provided bash script (not a Windows tool adapted for Mac -- per D-12) that collects all Intune agent logs, system profiler output, installed profiles, process lists, and network configuration in a single zip archive.

**Steps:**
1. Open Terminal on the macOS device (Applications > Utilities > Terminal, or Spotlight search "Terminal").
2. Download the diagnostic script:
   ```bash
   curl -L https://aka.ms/IntuneMacODC -o IntuneMacODC.sh
   ```
3. Make the script executable:
   ```bash
   chmod u+x ./IntuneMacODC.sh
   ```
4. Run with elevated privileges:
   ```bash
   sudo ./IntuneMacODC.sh
   ```
5. When prompted, enter the local administrator password.
6. Wait for collection to complete. Output: `IntuneMacODC.zip` in an `ODC/` subfolder, opened automatically in Finder.
7. Transfer the zip file to your investigation workstation.

**What IntuneMacODC collects** (bulleted list):
- Intune agent logs (`IntuneMDMDaemon*.log`, `IntuneMDMAgent*.log`)
- Company Portal logs
- System Profiler configuration profile dump
- `profiles show` output
- Network configuration
- Process list
- System log excerpts

Include a `> **Note:**` callout: "For full command reference including parameters and expected output, see [macOS Terminal Commands Reference](../reference/macos-commands.md#intunemacODC)."

**Runbook 11: `11-macos-profile-delivery.md`** -- Profile Delivery Investigation

**H1:** `# macOS Profile Delivery Investigation`

**Context:** Explain that configuration profiles on macOS are delivered via the Apple MDM channel (APNs push + device check-in). This is fundamentally different from Windows (CSP + SyncML). Do NOT apply Windows ESP or CSP diagnostic patterns.

**Investigation steps (after Step 1 log collection):**

Step 2: Verify MDM enrollment status
```bash
profiles status -type enrollment
```
Check: "Enrolled via DEP: Yes" and "MDM enrollment: Yes". If MDM enrollment shows "No" or missing, the device lost its management relationship -- escalate.

Step 3: List all installed profiles
```bash
sudo profiles show
```
Look for the expected profile by name. If missing, the profile was never delivered or was removed.

Step 4: Check profile delivery in unified log
```bash
log show --predicate 'subsystem == "com.apple.ManagedClient"' --info --last 1h
```
Look for: "Profile install request received", "Profile installed successfully", or error messages indicating payload rejection.

Step 5: Check for profile conflicts
```bash
system_profiler SPConfigurationProfileDataType
```
Compare installed profile list against expected profiles in Intune admin center (Devices > macOS > [device] > Configuration profiles). Identify any profile with "Error" or "Conflict" status.

Step 6: Verify Intune-side assignment and delivery status
- Open Intune admin center > Devices > macOS > [device] > Configuration profiles
- Check each profile's status: Succeeded, Pending, Error, Conflict, Not applicable
- If "Pending": device has not checked in. Trigger Sync from Intune portal.
- If "Error": note the error detail for resolution.

Step 7: Check specific failure scenarios (reference `06-config-failures.md` Configuration Profile Failures):
- SSID case mismatch (Wi-Fi profiles): verify SSID in profile matches actual network name exactly including case
- Firewall blocking MDM: if firewall profile is deployed, verify MDM exceptions are included
- FileVault without escrow: check if recovery key escrow is configured
- Certificate profile missing: 802.1X Wi-Fi requires a certificate profile deployed before or alongside Wi-Fi profile
- Deprecated Endpoint protection template: check if profile uses Settings Catalog instead

**Resolution Scenarios:**
1. Profile pending + device not checking in: Trigger Sync from Intune, wait 15 minutes, recheck
2. Profile conflict: Identify conflicting profiles, remove duplicate, redeploy
3. SSID mismatch: Correct SSID case in Intune profile, redeploy
4. Firewall blocking MDM: Add MDM server exceptions to firewall profile
5. Certificate chain incomplete: Deploy root CA and intermediate CA profiles before 802.1X

**Escalation Ceiling:** Open Microsoft support case if: profile shows "Error" with no actionable detail, MDM enrollment lost with no user action, profile installs but immediately removes itself. Include IntuneMacODC zip, `sudo profiles show` output, and unified log excerpt.

Target: 130-170 lines.

**Runbook 12: `12-macos-app-install.md`** -- App Install Failure Diagnosis

**H1:** `# macOS App Install Failure Diagnosis`

**Context:** Explain the two delivery channels: Apps deployed as DMG/PKG use the Intune Management Extension (IME) channel via the IntuneMDMDaemon process. VPP/Apps and Books apps use the Apple MDM command channel. Different channels = different diagnostic paths.

**IMPORTANT distinction per Research anti-pattern:** Do NOT conflate IME channel (DMG/PKG) with MDM channel (VPP). The investigation branches based on app type.

**Investigation steps (after Step 1 log collection):**

Step 2: Identify the app type and deployment method
- Check Intune admin center > Apps > macOS > [app name] > Properties
- Note: Type (DMG, PKG, or VPP/Apps and Books), Assignment type (Required/Available), Target group

Step 3: Check Intune-side install status
- Open Intune admin center > Apps > macOS > [app name] > Device install status
- Note the status per device: Installed, Failed, Pending, Not applicable

Step 4: Verify Intune agent is running (for DMG/PKG apps)
```bash
pgrep -il "^IntuneMdm"
```
Expected: `IntuneMDMDaemon` process running. If absent, the Intune management extension is not active -- device may have lost MDM relationship.

Step 5: Check IntuneMDMDaemon log (for DMG/PKG apps)
- Log path: `/Library/Logs/Microsoft/Intune/IntuneMDMDaemon*.log` (see [macOS Log Paths](../reference/macos-log-paths.md))
- Log format: `timestamp | process | level | PID | task | detail`
- Search for app name or "install" keyword
- Common error patterns:
  - "Download failed" -- network issue or CDN unreachable
  - "Detection script returned" -- detection rule mismatch
  - "Installation failed with exit code" -- installer error (note exit code)
  - "Disk space insufficient" -- device needs free space

Step 6: Check VPP/Apps and Books status (for VPP apps)
- In Intune admin center > Tenant administration > Connectors and tokens > Apple VPP tokens
- Verify token status: Active, Expired, or Expiring
- Check license count: Total licenses vs. assigned vs. consumed
- If "Available" assignment to device group: app will NOT appear in Company Portal (D-08 failure scenario -- must be user group)

Step 7: Check specific failure scenarios (reference `06-config-failures.md` App Deployment Failures):
- Non-app file in DMG `Included apps` list: verify DMG contains a .app bundle at expected path
- PKG > 2 GB: upload fails in Intune -- must split or repackage
- PKG without payload: continuous reinstall loop -- detection rule returns "not installed" after each install
- VPP license revoked: 30-day grace period before removal
- Expired VPP token: VPP apps stop syncing

**Resolution Scenarios:**
1. DMG/PKG download failure: verify network connectivity to CDN endpoints (see [endpoints reference](../reference/endpoints.md)), trigger Sync
2. Detection rule mismatch: update detection rule in Intune to match actual install path
3. VPP token expired: admin must renew in ABM > Settings > Apps and Books
4. VPP Available assigned to device group: change assignment to user group
5. Insufficient disk space: free space on device, trigger reinstall via Sync

**Escalation Ceiling:** Open Microsoft support case if: IntuneMDMDaemon log shows internal error with no remediation path, VPP license sync stuck for >24 hours, app installs then immediately uninstalls repeatedly. Include IntuneMacODC zip, IntuneMDMDaemon log excerpt, and Intune app install status screenshot.

Target: 130-170 lines.

**Runbook 13: `13-macos-compliance.md`** -- Compliance Evaluation Investigation

**H1:** `# macOS Compliance Evaluation Investigation`

**Context:** Explain macOS compliance evaluation in Intune: device checks in, Intune evaluates device state against compliance policy, sets compliant/non-compliant status, Conditional Access uses this status for resource access decisions. Unlike Windows, macOS has no Intune security baselines -- compliance policies are the only enforcement mechanism.

**IMPORTANT:** This runbook covers L2 investigation of WHY a device is non-compliant or why compliance status is not updating. The L1 runbook (14) handles the portal-observable scenarios; this L2 runbook investigates the underlying technical causes using Terminal diagnostics.

**Investigation steps (after Step 1 log collection):**

Step 2: Verify enrollment and MDM status
```bash
profiles status -type enrollment
```
Check: DEP enrollment = Yes, MDM enrollment = Yes. Compliance cannot evaluate if MDM enrollment is incomplete.

Step 3: List installed compliance-relevant profiles
```bash
sudo profiles show
```
Cross-reference against Intune admin center > Devices > macOS > [device] > Compliance. Check which specific compliance setting is failing.

Step 4: Check compliance evaluation in Intune
- Open Intune admin center > Devices > macOS > [device] > Compliance
- Note: Per-policy status, Per-setting status, Last evaluation timestamp
- If "Not evaluated": device has not checked in since policy was assigned. Check last check-in time.

Step 5: Investigate specific compliance failures:

**SIP (System Integrity Protection) disabled:**
```bash
csrutil status
```
Expected: "System Integrity Protection status: enabled." If disabled, no MDM remediation exists -- user must boot to Recovery Mode (Cmd+R at startup) and run `csrutil enable` from Terminal Utilities.

**FileVault not enabled:**
```bash
fdesetup status
```
Expected: "FileVault is On." If Off, check if FileVault configuration profile is deployed and if recovery key escrow is configured in Intune.

**Firewall not enabled:**
```bash
/usr/libexec/ApplicationFirewall/socketfilterfw --getglobalstate
```
Expected: "Firewall is enabled." If disabled, check if firewall configuration profile is deployed.

**OS version non-compliant:**
```bash
sw_vers -productVersion
```
Compare against minimum version in compliance policy. If update not available from Apple, entire fleet may be non-compliant until Apple releases update.

Step 6: Check Conditional Access interaction
- If device shows "Compliant" in Intune but user still cannot access resources:
  - Check Entra ID > Devices for Entra device registration status
  - Check Entra ID > Security > Conditional Access > [policy] for scope and conditions
  - Verify Company Portal registration completed (Entra device object exists with compliant status)

Step 7: Check Company Portal logs for registration issues
- Log path: `/Library/Logs/Microsoft/Intune/CompanyPortal*.log` (see [macOS Log Paths](../reference/macos-log-paths.md))
- Search for "registration" or "compliance" keywords
- Common patterns: "Device registration failed" (Entra issue), "Compliance check returned: non-compliant" (policy evaluation)

**Resolution Scenarios:**
1. Compliance not evaluated: Trigger Sync from Intune portal, wait for next check-in cycle (8 hours default)
2. SIP disabled: Guide user through Recovery Mode re-enable (this is a user action, not remotely fixable)
