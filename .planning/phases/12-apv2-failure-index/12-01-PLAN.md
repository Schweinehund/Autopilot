**Section structure** (per D-01 — 5 sections matching deployment phases):

1. `## Enrollment Failures (Steps 2-3)` — contains scenarios:
   - `### Deployment experience never launched` — Symptom: APv2 progress screen never appeared during OOBE; standard OOBE or ESP showed instead. Probable Cause: 6 causes listed in priority order from RESEARCH.md Scenario 1 (OS version below minimum, MDM auto-enrollment not configured, user not in policy user group, no device group selected, corporate identifiers missing, users not allowed to Entra join). Quick Check: Intune admin center > Devices > Windows > Enrollment > Device preparation policies path. Runbook: `**Runbook:** L1 runbook: APv2 deployment experience never launched (Phase 13)`
   - `### APv1 profile took precedence` — Symptom: ESP screen appeared during OOBE; the device ran the Enrollment Status Page instead of the APv2 Device Preparation progress screen. Add a bold callout: `> **Silent failure:** This is not an error. The device silently ran APv1 instead of APv2. No error message is displayed.` Probable Cause: Device is registered as a Windows Autopilot device (APv1) or has an APv1 profile assigned. Quick Check: Intune admin center > Devices > Windows > Windows enrollment > Devices path. Runbook: `**Runbook:** L1 runbook: APv1 profile took precedence over APv2 (Phase 13)`
   - `### Entra join failed` — Probable causes from RESEARCH.md Scenario 3 (join permissions, MDM scope, ETG group ownership). Quick Check: deployment monitor + Entra device settings. Runbook forward ref to Phase 13.
   - `### Intune enrollment failed` — Probable causes from RESEARCH.md Scenario 4 (MDM scope None, missing license). Quick Check: Entra Mobility + M365 admin licenses. Runbook forward ref to Phase 13.

2. `## IME Failures (Step 4)` — contains:
   - `### IME install failed` — Probable cause: network connectivity or transient Intune service issue. Key fact: IME failure blocks all subsequent app/script installation. Quick Check: deployment details Phase column. Runbook forward ref to Phase 14 (L2 — requires log analysis).

3. `## App Installation Failures (Steps 7-9)` — contains:
   - `### LOB or M365 app install failed` — Probable causes from RESEARCH.md Scenario 6 (packaging error, User vs System context, Skipped due to ETG assignment). Deployment report signals. Quick Check: deployment details Apps tab + app assignments. Runbook forward ref to Phase 13.
   - `### Win32, Store, or Enterprise App Catalog app install failed` — Probable causes from RESEARCH.md Scenario 8. MUST document both causes of "Skipped" status: (a) app not assigned to ETG device group, (b) Managed Installer policy was active (resolved April 2026 but still relevant for diagnosis). Quick Check: Apps tab + App control for business. Runbook forward ref to Phase 13.
   - `### Deployment timed out` — Probable causes from RESEARCH.md Scenario 9 (timeout too low, Windows 365 hardcoded timeout resolved Feb 2026, too many apps). Quick Check: Device preparation policy timeout setting. Runbook forward ref to Phase 13.

4. `## Script Failures (Step 8)` — contains:
   - `### PowerShell script execution failed` — Probable causes from RESEARCH.md Scenario 7 (non-zero exit, unhandled exception, timeout, User context, Skipped). Key fact: script failures cause entire deployment to fail. Quick Check: Scripts tab. Runbook forward ref to Phase 14 (L2 — script debugging requires logs).

5. `## Post-Deployment Issues (Step 10+)` — contains:
   - `### Wrong apps or policies applied` — Probable causes from RESEARCH.md Scenario 10 (local admin settings conflict, background sync vs OOBE install distinction, MDM policy conflicts, priority order). Quick Check: policy priority + Entra local admin settings + deployment details. Runbook forward ref to Phase 14 (L2 — policy conflict analysis).
