### L1 actor boundary on Linux (Gray Area 3 — winner: pragmatic per-runbook-per-cause hybrid built on the GA-3B read-vs-write apt sub-rule)

- **D-13:** **Read-vs-write apt distinction is the foundational sub-rule** (Adversary disprove 3B-5 ADOPTED). On Ubuntu/Debian: read-only inspection commands (`apt list`, `apt list --installed`, `apt-cache policy`, `dpkg -l`, `dpkg -s <pkg>`) are user-runnable WITHOUT sudo — only state-changing commands (`apt install`, `apt update`, `apt upgrade`, `apt remove`, `dpkg -i`) require root. L1 runbooks MUST NOT prepend sudo to read-only queries — doing so creates a false elevation requirement and conflicts with Phase 30 D-10 L1-readonly scope discipline. Validator (D-19) regex-asserts `sudo apt list` and `sudo dpkg -l` (and similar read-only-with-sudo patterns) do NOT appear in any of the 4 runbooks. Per-cause exceptions only when the underlying systemd unit / log path requires elevation (e.g., `journalctl --system` reads of root-only journals; `cryptsetup status` for LUKS state).
- **D-14:** **Reject Option 3A (strict portal-only across all 4 runbooks).** Defects 3A-1 (Runbook 33 has no portal-visible diagnostic surface) and 3A-2 (Runbook 30 package-install cause requires terminal apt/dpkg) were both CONFIRMED CRITICAL. Strict portal-only would force Runbook 33 to "escalate immediately" (failing LIN-11's "L1 responder can RESOLVE" mandate) and would force Runbook 30 package-install cause to "escalate immediately" (failing SC#2's per-cause observable-error-state mandate at ROADMAP line 210). Phase 30 D-10 Android L1 strict-portal worked because Android portal had diagnostic parity (admin center + Managed Google Play console); Linux portal lacks parity for systemctl/journalctl/dpkg surfaces (3A-3 MED confirmed).
- **D-15:** **Per-runbook-per-cause actor-boundary assignment:**
  - **Runbook 30 (LIN-08):** TERMINAL walkthrough across ALL 3 causes. Package-install cause needs `apt list --installed | grep intune-portal` + `dpkg -l intune-portal` + `cat /var/log/dpkg.log | grep intune-portal`; sign-in failure cause needs `journalctl --user -u microsoft-identity-broker` (`microsoft-identity-broker` is the systemd unit per Phase 49 D-14 glossary anchor); enrollment-timeout cause needs `journalctl --user -u intune-agent.timer`. Defects 3A-2 CRIT + 3D-2 MED both confirmed: portal cannot see deb-install state, cannot see Identity Broker errors, cannot see timer-poll status.
  - **Runbook 31 (LIN-09):** PORTAL-FIRST for distro/version + encryption + password causes (admin portal shows compliance failing-setting label per P-09-equivalent navigation); TERMINAL walkthrough for custom-compliance Bash cause only — `cat /var/log/intune-update.log` + `journalctl --user | grep intune-update` to read Bash exit codes (3C-2 MED confirmed: portal cannot see Bash script exit codes).
  - **Runbook 32 (LIN-10):** PORTAL-ONLY across all 3 causes — device-not-enrolled is visible in admin portal device inventory; device-not-compliant has compliance state in P-09-equivalent; Edge-not-signed-in is the user-side Edge browser sign-in dialog (no terminal needed). 3C-1 CRIT defect ("Runbook 30 package-install cause cannot be diagnosed via portal-only") was confirmed but applies to Runbook 30, not Runbook 32 — Runbook 32's 3 causes are all portal-observable.
  - **Runbook 33 (LIN-11):** TERMINAL walkthrough — single-cause path through `systemctl --user status intune-agent.timer` + `systemctl --user is-active intune-agent.timer` + `journalctl --user -u intune-agent.timer --since "1 hour ago"` + `apt list --installed | grep intune-portal` (read-only diagnostic only; remediation `systemctl --user restart` requires admin action per Phase 30 D-10 L1-readonly).
- **D-16:** **L1 actor boundary anatomy within terminal-walkthrough sections** preserves Phase 30 D-10 L1-readonly discipline: L1 reads commands aloud to user; user runs the command on their device; L1 interprets output; L1 does NOT execute remediation (state-changing commands route to `## Admin Action Required` per Phase 30 D-12). Each numbered L1 Triage Step in a terminal-walkthrough cause carries the format: `> **Say to the user:** "[explanation]. Please open Terminal and type: \`<command>\`. Read me the output."` mirroring Runbook 22 line 31 `> **Say to the user:**` precedent. Defects 3B-3 (Linux-literate L1 hiring constraint) and 3B-4 (auditability gap from L1-directed user-typed commands) acknowledged but ACCEPTED as the cost of bridging Linux portal-parity gap; alternative (3A strict-portal) carries higher CRIT cost.
- **D-17:** **Mixed actor-boundary shapes across Runbooks 30/31/32/33 are accepted** as the structurally-correct response to Linux's portal-parity gap (defects 3D-1 + 3C-3 + 3D-4 confirmed but unavoidable). Validator (D-19) carries per-runbook actor-boundary regex assertions (terminal-walkthrough presence in Runbook 30 + Runbook 33; portal-only structural assertion in Runbook 32; mixed assertion in Runbook 31).
- **D-18:** **L1 sudo-prohibition regex.** Validator asserts that read-only inspection commands documented in L1 Triage Steps do NOT carry sudo prefixes (e.g., `sudo apt list`, `sudo dpkg -l`, `sudo systemctl --user status` — `--user` scope never needs sudo per systemd user-mode model). Allows sudo prefixes only on commands whose underlying surface requires elevation (e.g., `sudo journalctl --system`, `sudo cryptsetup status` for LUKS). Per CDI-Phase51-05.
