   [Tenant-to-Tenant Device Migration](../../device-operations/04-tenant-migration.md) (v1.2)
   for the Windows hardware-hash deregistration step-by-step procedure with batch-size +
   timing constraints. This guide covers the BitLocker re-key options and post-migration
   escrow validation that are scope-net-new in Phase 56.
3. **BitLocker re-key** — choose ONE of three approaches enumerated below (neutral; no
   Microsoft preference). Choose based on tenant-side data-risk tolerance, third-party
   budget, and PowerShell scripting capacity.
4. **Autopilot re-registration** in the target tenant (hash import + profile assignment).
5. **Post-migration BitLocker recovery key escrow validation** — verify keys are present in
   the target Microsoft Entra ID via the Entra portal (Devices → BitLocker keys; per-device
   inspection or bulk export).

### BitLocker re-key — three approaches (neutral enumeration)

The three BitLocker re-key approaches are enumerated below WITHOUT Microsoft preference. REQ
DRIFT-04 explicitly requires neutrality on tooling/approach choice.

#### Option (a): Source-tenant escrow → target Entra via PowerShell scheduled task

Run a PowerShell scheduled task post-migration that calls `BackupToAAD-BitLockerKeyProtector`
to escrow the existing recovery key from the workstation to the target Microsoft Entra ID.
This approach preserves device encryption continuity (no decrypt window). It is the canonical
PowerShell-driven path used by community tooling and by Quest On Demand Migration's own
BitLocker handling.

**Prerequisite caveat (CRITICAL):** `BackupToAAD-BitLockerKeyProtector` only escrows the
existing RecoveryPassword IF a new key protector is created and encryption is re-triggered,
OR if the existing protector has not yet been escrowed during the device's lifetime. Some
workflows therefore require:

```powershell
Add-BitLockerKeyProtector -MountPoint "C:" -RecoveryPasswordProtector
# then escrow the NEW protector
BackupToAAD-BitLockerKeyProtector -MountPoint "C:" -KeyProtectorId $newProtectorId
```

Without this prerequisite, the script appears to succeed (no error returned) but no escrow
occurs in the target Entra tenant — a silent-failure pattern. Document and test this caveat
verbatim in any custom implementation.

#### Option (b): Decrypt source-tenant BitLocker → re-encrypt under target tenant

Decrypt each device using `manage-bde -off` (or GPO/MDM control), complete tenant migration,
then re-encrypt and escrow the new recovery key to the target Entra ID.

> ⚠️ **Data-risk window:** Option (b) leaves devices unencrypted between decrypt and
> re-encrypt. The duration depends on disk size + tenant migration cadence. For regulated
> workloads or remote / unsupervised devices, prefer option (a) or (c).

#### Option (c): Third-party tool

A third-party tool (e.g., Quest On Demand Migration) may provide an integrated BitLocker
handling workflow as part of the broader tenant-migration product. Quest On Demand
Migration includes optional `BitlockerBackupToEntraID` tasks that install a PowerShell
script + scheduled task on the workstation to escrow the recovery key to the target
Entra ID post-migration login.

The choice of third-party tool is a procurement decision outside the scope of this runbook;
this guide names Quest only as a documented example of a tool that includes BitLocker
re-key tasks. Tool selection criteria (licensing, scope of features beyond BitLocker, support
SLAs) are outside the documentation scope.

### Post-migration escrow validation

After completing the BitLocker re-key approach of choice and the Autopilot re-registration
in the target tenant, perform escrow validation to confirm BitLocker recovery keys are
present in the target Entra:

1. In the Microsoft Entra admin center, navigate to **Devices** → select the device →
   **BitLocker keys** tab.
2. Confirm that one or more recovery key entries are listed with the expected
   `KeyProtectorId` (matches the protector escrowed by the chosen approach).
3. For fleet-level validation, export the BitLocker keys table from Entra and reconcile
   against the device serial-number inventory from step 1.

If the recovery key is absent or stale, re-run the chosen approach (option a/b/c) and
re-validate. Devices missing recovery keys post-migration are a known-and-recoverable
condition; re-run is idempotent.
```

```markdown
## macOS / iOS tenant migration

> 📌 **OS 26+ device migration:** Apple has introduced new ABM device-migration APIs in OS 26
> (iOS 26 / iPadOS 26 / macOS 26) that materially change the cross-MDM migration surface (no
> wipe; preserves data; deadline enforcement). Devices on OS 26+ may use the new path; older
> devices use the legacy path documented below. Apple's documentation owns the OS 26
> procedure detail — see
> [Apple's Migrate managed devices guide](https://support.apple.com/guide/deployment/migrate-managed-devices-dep4acb2aa44/web)
> for current 2026 OS 26 mechanics. The legacy path remains the BASELINE for fleets on iOS <26
> / iPadOS <26 / macOS <26.

### Legacy path (BASELINE for iOS <26 / iPadOS <26 / macOS <26)

macOS and iOS tenant migration follows a four-step sequence:

1. **ABM token re-issue** — In Apple Business Manager (or Apple School Manager), `release`
   each device from the MDM server in the source tenant ABM. Then `re-assign` each device
   to the MDM server in the target tenant ABM. The `ABM token` for the target tenant must
   be uploaded to the target Intune tenant's enrollment configuration ahead of re-assignment.
   Note: the MDM server assignment **cannot** be transferred between tenants — devices must
   be released from Tenant A's MDM server first, then assigned to Tenant B's MDM server.
2. **In-use device handling** — Devices that are currently enrolled and in use require a
   `wipe` followed by `re-enrollment` against the target tenant. Cross-tenant enrollment-
   state preservation is not supported on the legacy path; the wipe + re-enrollment sequence
   is the only Apple-blessed path.
3. **Await-Configuration behavior** — Devices in DEP/ADE flows wait at the Setup Assistant
   `Await-Configuration` step for the new MDM server check-in. The target tenant's ADE
   profile must be assigned and ready before the device powers on; otherwise the device
   stalls at the Setup Assistant prompt waiting for the new MDM server.
4. **Re-enrollment validation** — After re-enrollment, validate device compliance state in
   the target tenant's Intune compliance reports. FileVault recovery key escrow (a separate
   concern from configuration; see the cross-platform encryption-drift section below) does
   NOT auto-update post-re-enrollment — admins should plan a follow-up escrow workflow.

### OS 26+ alternative path

For fleets on iOS 26 / iPadOS 26 / macOS 26 OR newer, Apple's cross-MDM migration APIs allow
device transfer between MDM servers without `wipe` + `re-enrollment`:

- In ABM/ASM portal, transfer device assignment between MDM servers using the OS 26
  device-migration API.
- Set an enrollment deadline; devices auto-lock if not re-enrolled by deadline.
- Apple's documentation owns the procedure detail; this runbook references but does not
  replicate Apple's OS 26 procedure.

The OS 26 path is operationally simpler for fleets that meet the version requirement; the
legacy path remains canonical for mixed fleets and as a fallback.
```

**Section 6 — `## Android tenant migration` H2 (DRIFT-06 FOLD; V-56-20 + V-56-21):**

```markdown
## Android tenant migration

Android tenant migration follows a four-step sequence covering Managed Google Play re-binding
and per-ownership-mode device re-provisioning.

### MGP re-binding sequence

1. **`disconnect` from source MGP** — In the Intune admin center → Settings → Managed Google
   Play → **Disconnect**. Note: disconnect retires + deletes all enrolled Android devices
   in the source tenant per Microsoft Q&A 2026 — this is a fleet-affecting operation that
   must be planned around device-recovery readiness.
2. **`bind new` MGP account to target Intune tenant** — Use the Entra-linked Managed Google
   Domain account (current 2026 recommended path; the "new way" introduced August 2024
   replaces personal-Gmail-based bindings). Personal-Gmail-based bindings still work for
   existing tenants but Entra-linked is recommended for new bindings.
3. **`re-approve` all apps** in the target Managed Google Play console — the existing app
   catalog does NOT survive the disconnect/rebind. This is a known gotcha that catches admins
   off-guard; budget time to re-approve every app in the source catalog.
4. **`re-provision` devices** per ownership mode (next sub-section).

### Per-ownership-mode re-provisioning

Android device re-provisioning depends on the ownership mode:

- **Corporate-owned (fully managed / dedicated / COBO):** Perform `factory reset` on each
  device, then `re-enrollment` via the target tenant's DPC. This is the only supported
  cross-tenant migration path for corporate-owned Android.
- **BYOD (work profile):** Perform `work profile re-creation` — the user removes the existing
  work profile via Settings → Accounts → Work account → Remove, then re-enrolls via the
  target tenant Company Portal. Personal data on the device is unaffected.
- **Zero-Touch enrolled corporate-owned:** Perform `ZT portal re-upload` with target tenant
  credentials. Disconnect the ZT account from the source Intune tenant (Settings → Android
  Enterprise → Zero Touch enrollment), then link the new ZT account to the target Intune
  tenant; push the new EMM DPC config to the ZT portal. Devices then receive the new
  EMM DPC JSON config at first boot after factory reset.

### Validation

After re-provisioning:

- Validate device enrollment state in the target tenant's Intune Devices view.
- Verify the work profile (BYOD) or fully-managed surface (corporate-owned) is bound to the
  target tenant via the device's Settings → Accounts → Work account → Account info.
- Re-validate Play Integrity verdict if Play-Integrity-based compliance is enabled in the
  target tenant.
```

## Cross-platform encryption drift

Tenant migration interacts with platform-level encryption surfaces in different ways across
Windows, macOS, iOS, and Android. This section is **pure educational comparison content** —
it documents the encryption surfaces, drift risks, and MDM management boundaries WITHOUT
recommending specific encryption-key-recovery tools (per the MEDIUM-confidence framing of
this runbook and to maintain neutrality on commercial recovery products).

### Per-platform encryption surfaces

| Platform | Encryption surface | Drift risk during tenant migration | MDM management |
|----------|-------------------|-----------------------------------|----------------|
| **Windows** | `BitLocker` (full-disk encryption; Entra ID escrow target) | Key escrow targeting risk: BitLocker recovery key escrowed to source Entra; tenant migration leaves the key orphaned unless re-escrow workflow runs | YES — Intune disk encryption profile + `BackupToAAD-BitLockerKeyProtector` cmdlet (see Windows tenant migration section above) |
| **macOS** | `FileVault` (full-disk encryption; recovery key escrow) | Escrow not updated post-re-enrollment behavior: re-enrolling the device in the target tenant does NOT automatically re-escrow FileVault recovery key; admin-side action required | YES — Intune FileVault policy with personal/institutional recovery key options |
| **iOS** | `iOS device-level` encryption (hardware-backed; passcode-derived) | None — encryption is automatic on all iOS devices and is not tenant-bound; no recovery key escrow surface to manage | NO — encryption is automatic on iOS; MDM cannot manage keys; only compliance check (passcode complexity policy + encryption-required compliance signal) |
| **Android** | `dm-crypt` (or file-based encryption on Android 10+; per-OEM variance) | None — encryption is automatic on Android 10+ and is not tenant-bound; no Intune-supported recovery-key escrow surface | NO — encryption automatic; per-OEM variance noted (Pixel / Samsung / Zebra implementations differ; AOSP per-OEM precedent applies). Note: LUKS (Linux-only) is **not** an Android implementation. |

### Editorial boundaries

This section intentionally:

- **Does NOT recommend specific encryption-key-recovery tools** for any platform — the
  4-platform encryption surface comparison is educational, not prescriptive.
- **Does NOT name** commercial recovery products beyond the single mention of Quest On
  Demand Migration in the Windows tenant migration section (per the single-mention contract
  scoped to the BitLocker option (c) narrative).
- **Frames iOS as "no MDM management beyond compliance check"** — admins migrating iOS
  fleets should not expect any iOS-side encryption-key handling beyond what the platform
  does automatically.
- **Frames Android as "per-OEM variance"** — admins should validate per-OEM encryption
  behavior on a per-fleet basis; AOSP per-OEM precedent applies.

For BitLocker re-key approaches (Windows) and FileVault re-escrow planning (macOS), see
the Windows and macOS / iOS tenant migration sections above. The cross-platform encryption-
drift content here is comparative, not procedural.
```
