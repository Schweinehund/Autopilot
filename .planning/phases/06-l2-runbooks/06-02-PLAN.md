**Title:** `# ESP Deep-Dive Investigation`

**Context section:** ESP failures are the most common Autopilot escalation. This guide covers registry-level investigation for both device phase and user phase failures, including LOB and Win32 app conflict identification. Link first-mention glossary terms (ESP, OOBE, MDM, etc.) to `../_glossary.md#term`.

**Triage block (per D-07/D-08/D-09):**

```markdown
## Triage

**From L1 escalation ([ESE1](../decision-trees/01-esp-failure.md), [ESE2](../decision-trees/01-esp-failure.md), [ESE4](../decision-trees/01-esp-failure.md), or [ESE5](../decision-trees/01-esp-failure.md))?**
L1 collected: serial, deployment mode, timestamp, ESP phase, time spent on ESP before/after reboot, app name shown on ESP screen.
Skip to Step 2.

**Starting fresh?** Begin at Step 1.
```

**Investigation section with numbered steps:**

**Step 1: Collect diagnostic package**
"Before starting: collect a diagnostic package per [Log Collection Guide](01-log-collection.md)."

**Step 2: Identify ESP phase (device vs user)**
PowerShell to read the FirstSync registry key:
```powershell
$enrollments = Get-ChildItem "HKLM:\SOFTWARE\Microsoft\Enrollments" -ErrorAction SilentlyContinue
foreach ($enrollment in $enrollments) {
    $firstSync = Get-ItemProperty -Path "$($enrollment.PSPath)\FirstSync" -ErrorAction SilentlyContinue
    if ($firstSync) {
        [PSCustomObject]@{
            EnrollmentGUID = $enrollment.PSChildName
            IsServerProvisioningDone = $firstSync.IsServerProvisioningDone
        }
    }
}
```
Explain: `IsServerProvisioningDone = 0` means device phase incomplete; `= 1` means device phase done and issue is in user phase. Link to [registry-paths.md](../reference/registry-paths.md) for full path documentation.

**Step 3: Check expected policies (device phase stuck)**
```powershell
$espPolicies = "HKLM:\SOFTWARE\Microsoft\Windows\Autopilot\EnrollmentStatusTracking\ESPTrackingInfo\Diagnostics\ExpectedPolicies"
Get-Item -Path $espPolicies -ErrorAction SilentlyContinue
```
Explain what policies are expected vs applied. Link to [registry-paths.md](../reference/registry-paths.md).

**Step 4: Check Win32 app install status (Sidecar)**
```powershell
$sidecar = "HKLM:\SOFTWARE\Microsoft\Windows\Autopilot\EnrollmentStatusTracking\ESPTrackingInfo\Diagnostics\Sidecar"
Get-ChildItem -Path $sidecar -ErrorAction SilentlyContinue | Select-Object Name, PSChildName
```
Explain per-app subkeys and how to identify which app is blocking. Distinguish LOB vs Win32 app behavior.

**Step 5: Check app install log**
CRITICAL: Use `AppWorkload.log` (NOT `IntuneManagementExtension.log` — the log path changed August 2024):
```powershell
$appLog = "C:\ProgramData\Microsoft\IntuneManagementExtension\Logs\AppWorkload.log"
Get-Content $appLog -Tail 100 | Select-String -Pattern "error|fail|0x8" -CaseSensitive:$false
```
Note: `IntuneManagementExtension.log` still exists but is no longer the primary source.

**Step 6: Check for policy conflicts**
If investigation reveals policy-related ESP failure, cross-reference to [Policy Conflict Analysis](05-policy-conflicts.md).

**Resolution section with multiple scenarios:**

**Scenario A: ESP stuck due to stale FirstSync keys**
```powershell
Restart-EnrollmentStatusPage -WhatIf   # dry run first
Restart-EnrollmentStatusPage           # execute after confirming output
```
Per D-01, include inline copy-pasteable PowerShell.

**Scenario B: Win32 app install failure blocking ESP**
Steps to identify the blocking app from Sidecar subkeys, check AppWorkload.log for the specific app error, and resolve (retry install, remove app from ESP-required list).

**Scenario C: ESP timeout (no specific error)**
Check timeout thresholds (30 min device phase, 60 min user phase per Phase 4 convention), review if ESP skip is configured, check if policy is causing indefinite wait.

**Escalation Ceiling (per D-02):**
"Escalate when root cause is a Microsoft-side service issue or Intune backend bug. Indicators: all policies appear correct, no app install failures, but ESP never completes; or Intune service health dashboard shows degradation."

**Title:** `# Policy Conflict Analysis`

**Context section:** Policy conflicts are a common cause of ESP failures, OOBE reboot loops, and autologon failures during Autopilot provisioning. This guide covers systematic identification and resolution of conflicting policies. Link first-mention glossary terms.

**Triage block (per D-07/D-08/D-09):**
Reference relevant Phase 4 escalation node IDs. Since policy conflicts typically surface during ESP, reference ESE nodes. Dual-path: L1 escalation data expected vs starting fresh.

**Investigation section with numbered steps:**

**Step 1: Collect diagnostic package**
Link to [Log Collection Guide](01-log-collection.md).

**Step 2: Check AutoAdminLogon setting**
This is the most common policy conflict for Autopilot:
```powershell
$winlogon = "HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon"
Get-ItemProperty -Path $winlogon -Name AutoAdminLogon -ErrorAction SilentlyContinue
```
Explain: If AutoAdminLogon is set to `0`, Autopilot autologon during provisioning breaks. Security Baselines and DeviceLock policies commonly set this to `0`.

**Step 3: Check for AppLocker CSP conflicts**
Explain that AppLocker CSP configured via Intune triggers reboot loops during OOBE/ESP. Investigation: check if AppLocker CSP policy is targeted at the Autopilot device group.

**Step 4: Check for DeviceLock policy conflicts**
Investigate DeviceLock/Password policies that may conflict with Autopilot's zero-touch provisioning flow. Check if DeviceLock policies are deployed to Autopilot device groups.

**Step 5: Check PreferredAadTenantDomainName conflicts**
```powershell
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Provisioning\Diagnostics\Autopilot" -ErrorAction SilentlyContinue
```
Explain the PreferredAadTenantDomainName scenario from error-codes/03-esp-enrollment.md.

**Step 6: Review Security Baseline policy assignments**
Check if Microsoft 365 Security Baseline is applied to Autopilot device groups. Note: specific policy names should be validated against the current baseline version (STATE.md blocker — flag as requiring verification at authoring time). Common conflict areas: VBS settings, UAC settings, DeviceLock settings.

**Resolution section with scenarios from error-codes/03-esp-enrollment.md policy conflicts table:**

**Scenario A: AutoAdminLogon set to 0**
Resolution: Exclude Autopilot device group from the policy setting AutoAdminLogon to 0. Per D-01, include inline PowerShell to verify the fix:
```powershell
# Verify AutoAdminLogon is restored after policy exclusion
Get-ItemProperty -Path "HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon" -Name AutoAdminLogon
# Expected: AutoAdminLogon = 1 (or not present)
```

**Scenario B: AppLocker CSP causing reboot loop**
Remove AppLocker CSP from Autopilot device group during provisioning. Re-apply after provisioning is complete.

**Scenario C: DeviceLock/Password policy during OOBE**
Exclude or defer DeviceLock policies from Autopilot provisioning groups.

**Scenario D: Security Baseline conflicts**
Create an Autopilot-specific Security Baseline profile that excludes conflicting VBS and autologon settings during provisioning.

**Scenario E: PreferredAadTenantDomainName mismatch**
Verify tenant domain name configuration matches the target Azure AD tenant.

**Escalation Ceiling (per D-02):**
"Escalate when root cause is a Microsoft-side service issue or Intune backend bug. Indicators: policy assignments appear correct, no conflicting policies identified, but provisioning still fails."

**Tool References (per D-03):**
- [Get-AutopilotDeviceStatus](../reference/powershell-ref.md#get-autopilotdevicestatus) — comprehensive device state snapshot
- [Registry Paths](../reference/registry-paths.md) — Autopilot and provisioning registry paths
- [ESP Error Codes](../error-codes/03-esp-enrollment.md) — policy conflicts table
- [ESP Deep-Dive](02-esp-deep-dive.md) — for ESP-specific registry investigation when policy conflict is confirmed
