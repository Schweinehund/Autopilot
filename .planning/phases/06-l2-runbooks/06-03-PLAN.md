
**Step 3: Identify TPM manufacturer and version**
```powershell
Get-CimInstance -Namespace "root\cimv2\security\microsofttpm" -ClassName Win32_Tpm |
    Select-Object IsEnabled_InitialValue, IsActivated_InitialValue, ManufacturerId, ManufacturerVersion, SpecVersion
```
Explain how to match `ManufacturerId` to chipset manufacturer. This identifies which hardware-specific section applies.

**Step 4: Check Autopilot registration state**
```powershell
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Provisioning\Diagnostics\Autopilot" -ErrorAction SilentlyContinue
```
Verify device is registered and profile downloaded before TPM attestation runs.

**Step 5: Cross-reference error code**
If error code was captured, look up in [TPM Attestation Error Codes](../error-codes/02-tpm-attestation.md). Do NOT reproduce the error code table — link to it.

**Resolution section — per hardware-specific chipset:**

CRITICAL per D-02 and Research Pitfall 6: L2 investigation identifies the chipset and error code. Resolution sections direct to OEM support for firmware updates — they do NOT document the firmware update procedure itself.

**Scenario A: AMD fTPM (0x80070490)**
Investigation identifies AMD fTPM via ManufacturerId. Resolution: contact OEM support for AMD firmware update. Escalate per D-02.

**Scenario B: ST Micro RSA cert expiry**

**Scenario C: Nuvoton RSA-3072 key length**

**Scenario D: Infineon SLB9670**
Investigation identifies Infineon chipset. Resolution: check for known Infineon firmware advisories; contact OEM. Escalate.

**Scenario E: Intel Tiger Lake platform**
Investigation identifies Intel Tiger Lake via platform info. Resolution: contact OEM for platform firmware update. Escalate.

**Scenario F: TPM clear and retry (generic)**
For non-hardware-specific cases where TPM is in a bad state:
```powershell
Reset-TPMForAutopilot -WhatIf   # dry run first — device WILL reboot
Reset-TPMForAutopilot           # execute after confirming
```
Per D-01, inline copy-pasteable PowerShell with -WhatIf dry run shown first.

**Scenario G: TPM version 1.2 (TPE2)**
TPM supports 2.0 but running 1.2 firmware. Resolution: BIOS update required. Contact OEM.

**Scenario H: Retry limit exceeded (0x81039001)**
Intermittent failure. Retry provisioning (max 2 attempts, 15 min between). If persistent, escalate.

**Version gate banner** (exact text from template).

**Title:** `# Hybrid Join Failure Investigation`

**Context section:** Hybrid Azure AD join failures occur when domain-joined devices fail to complete the hybrid identity registration flow. Common causes include ODJ Connector issues, domain controller reachability, and OU permissions. Link first-mention glossary terms.

**Triage block (per D-07/D-08/D-09):**
Reference appropriate Phase 4 escalation nodes. Hybrid join issues may come via TRE3-TRE6 from initial triage or from L1 device registration runbook escalation. Dual-path routing.

**Investigation section:**

**Step 1: Collect diagnostic package**
Link to [Log Collection Guide](01-log-collection.md).

**Step 2: Verify ODJ Connector version and status**
CRITICAL (STATE.md blocker resolved): The log path changed with connector version 6.2501.2000.5 (January 2025).
```powershell
# Check connector version (LOW confidence registry path — verify existence)
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Intune Connector for AD" -ErrorAction SilentlyContinue |
    Select-Object DisplayVersion, InstallDate
```
Note: flag this registry path as LOW confidence per Research. The Event Viewer log existence itself confirms the version.

Minimum version required: **6.2501.2000.5**. If older, upgrade connector before investigating further.

**Step 3: Check ODJ Connector event log**
CRITICAL: Use the CURRENT log path, not the legacy path.
```powershell
# Current path (connector v6.2501.2000.5+):
# Event Viewer: Applications and Services Logs > Microsoft > Intune > ODJConnectorService
Get-WinEvent -LogName "Microsoft-Intune-ODJConnectorService/Operational" -MaxEvents 50 -ErrorAction SilentlyContinue
```

> **Do not use** the legacy path `Applications and Services Logs > ODJ Connector Service` — it is stale on connectors v6.2501.2000.5 and later.

Look for Event IDs from error-codes/05-hybrid-join.md: 807, 809, 815, 908, 171, 172. Link to [Hybrid Join Error Codes](../error-codes/05-hybrid-join.md) for event ID meaning — do NOT reproduce the table.

**Step 4: Check domain controller reachability**
```powershell
# Verify domain controller connectivity from the connector server
nltest /dsgetdc:contoso.com  # replace with actual domain
Test-ComputerSecureChannel -Verbose
```

**Step 5: Check OU permissions for connector service account**
Verify the connector service account has permissions to create computer objects in the target OU. This is the third 0x80070774 cause.

**Resolution section — CRITICAL: 0x80070774 has THREE distinct causes (Research Pitfall 3):**

**Scenario A: 0x80070774 — "Assign user" feature active in hybrid profile**
Profile misconfiguration. Resolution: remove the "Assign user" setting from the hybrid Autopilot profile. This is a profile fix, not an infrastructure issue.

**Scenario B: 0x80070774 — ODJ Connector installed in wrong domain**
Connector is joined to a domain that doesn't match the target hybrid profile domain. Resolution: reinstall connector in the correct domain.

**Scenario C: 0x80070774 — Connector service account lacks OU permissions**
The service account running the ODJ Connector doesn't have Create Computer Objects permissions in the target OU. Resolution: grant permissions in AD.

**Scenario D: ODJ Connector version outdated**
Connector version is below 6.2501.2000.5. Resolution: update the connector to the latest version.

**Scenario E: Domain controller unreachable**
Network or DNS issue between connector server and DC. This is an infrastructure issue — escalate.

**Escalation Ceiling (per D-02):**
"Escalate when domain controller or AD Connect issues are confirmed. These are infrastructure team responsibilities, not desktop engineering. Indicators: connector can reach DC but join fails due to replication lag, AD Connect sync issues, or ADFS federation problems."
