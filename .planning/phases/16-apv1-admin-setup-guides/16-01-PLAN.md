    Mermaid decision tree at top of file (per D-04) helping admin choose which path:
    ```mermaid
    flowchart TD
      START([How were devices acquired?]) --> OEM{Purchased from<br/>authorized OEM with<br/>Autopilot registration?}

      BATCH -->|Few devices or<br/>one-at-a-time| PSPATH[PowerShell Path: Get-WindowsAutopilotInfo]
    ```

    ## Prerequisites (shared across all paths):
    - Intune Administrator or Autopilot Administrator role
    - Devices powered on and network-connected (for PowerShell direct upload path)
    - For CSV path: access to existing CSV export or physical device to generate hash

    **## Path 1: OEM Delivery** (per D-05 — minimal depth, 3-5 verification steps only):
    Steps: Navigate to Intune admin center > Devices > Windows > Enrollment > Windows Autopilot > Devices. Search by serial number. Confirm device appears with ZTDID. Confirm Profile Status. If missing, contact procurement to confirm OEM registration was included.

    One what-breaks callout: "What breaks if device not found: Admin sees device not in Autopilot list. End user sees standard Windows OOBE instead of Autopilot. Runbook: [Device Not Registered](../l1-runbooks/01-device-not-registered.md)"

    **## Path 2: CSV Bulk Import** (per D-05 — moderate depth):
    Prerequisites: CSV file in ANSI encoding (NOT UTF-8, NOT Unicode — this is the #1 import failure cause).

    CSV format — required columns (case-sensitive): `Device Serial Number,Windows Product ID,Hardware Hash` — optional: `Group Tag,Assigned User`. No extra columns. No quotation marks. Max 500 devices per CSV. Do NOT open in Excel (Excel reformats and breaks the file).

    Steps: Navigate to Intune admin center > Devices > Windows > Enrollment > Windows Autopilot > Devices > Import. Select CSV. Wait for import completion (up to 15 min for large batches). Verify devices appear.

    Import error reference table:
    | Error | Meaning | Resolution |
    |-------|---------|------------|
    | ZtdDeviceAssignedToAnotherTenant | Hash already in different tenant | Remove from original tenant first |
    | ZtdDeviceAlreadyAssigned | Device already in this tenant | Skip — already registered |
    | ZtdDeviceDuplicated | Duplicate rows in CSV | Remove duplicates, re-import |
    | InvalidZtdHardwareHash | Missing manufacturer or serial in hash | Re-capture hash from device |

    What-breaks callouts for: ANSI encoding (wrong encoding causes silent failure or header error), duplicate serial numbers, Excel editing.

    **## Path 3: PowerShell Script** (per D-05 — full depth, most error-prone):

    Prerequisites: Physical access to device, PowerShell 5.1+, internet connectivity, Intune Administrator credentials (for -Online path).

    **Option A: Save hash locally as CSV then upload:**
    ```powershell
    [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
    New-Item -Type Directory -Path "C:\HWID"
    Set-Location -Path "C:\HWID"
    $env:Path += ";C:\Program Files\WindowsPowerShell\Scripts"
    Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned
    Install-Script -Name Get-WindowsAutopilotInfo
    Get-WindowsAutopilotInfo -OutputFile AutopilotHWID.csv
    ```
    Output: `C:\HWID\AutopilotHWID.csv` — upload via CSV Import path above.

    **Option B: Direct upload to Intune:**
    ```powershell
    [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
    Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned
    Install-Script -Name Get-WindowsAutopilotInfo -Force
    Get-WindowsAutopilotInfo -Online
    ```
    Prompts for Intune Administrator sign-in. Agree to NuGet/PSGallery installation prompt.

    **Common errors section** (dense what-breaks callouts):
    1. Execution policy block: fix with `-Scope Process -ExecutionPolicy RemoteSigned` (process-scoped, no machine-wide impact). What-breaks: Admin sees red error text about running scripts being disabled.
    2. NuGet "no match found": TLS 1.2 not set. Fix: `[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12` must be first command. What-breaks: Install-Script fails with no matching package.
    3. Graph auth errors: Script uses Microsoft Graph PowerShell modules (updated July 2023, not deprecated AzureAD). May require approving enterprise app permissions on first run. What-breaks: `-Online` flag fails with authentication popup that errors.
    4. Stale hash from reimaged device: Hash must be captured from final hardware state. Hardware changes, BIOS updates, or driver updates after capture invalidate the hash. What-breaks: Device registered but OOBE shows standard setup instead of Autopilot. Runbook: [Device Not Registered](../l1-runbooks/01-device-not-registered.md)

    Title: `# Deployment Profile Configuration`

    ## Prerequisites:
    - Intune Administrator role
    - At least one device registered in Windows Autopilot (see [Hardware Hash Upload](01-hardware-hash-upload.md))
    - For hybrid join: Intune Connector for AD configured (see [Intune Connector](09-intune-connector-ad.md))
    - Company branding configured in Entra ID (required if hiding change account options)

    ## Steps:

    ### Step 1: Create deployment profile
    Navigate to **Intune admin center** > **Devices** > **Windows** > **Enrollment** > **Windows enrollment** > **Windows Autopilot** > **Deployment Profiles** > **Create Profile** > **Windows PC**.

    ### Step 2: Configure OOBE settings
    Document EVERY setting from the research table with a what-breaks callout for each configurable option:

    1. **Deployment mode**: User-Driven or Self-Deploying. What-breaks: Self-Deploying greys out most other settings; choosing wrong mode causes OOBE failure. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    2. **Join to Microsoft Entra ID as**: Microsoft Entra joined or Hybrid Microsoft Entra joined. What-breaks: Hybrid requires Intune Connector — without it, join fails silently during OOBE. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    3. **Microsoft Software License Terms**: Show or Hide. What-breaks: Show adds a user-facing screen; no functional impact but affects user experience timing.

    4. **Privacy settings**: Show or Hide. What-breaks: Hiding disables location services by default — apps relying on location will fail.

    5. **Hide change account options**: Show or Hide. What-breaks: Requires company branding configured in Entra ID. Without branding, this setting is ignored. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    6. **User account type**: Administrator or Standard User. What-breaks: Administrator grants local admin rights to the joining user — may violate security policy.

    7. **Allow pre-provisioned deployment**: Yes or No. What-breaks: Setting to No still allows Win+F12 keypress but deployment fails with error 0x80180005. Must be Yes for pre-provisioning flow. Runbook: [OOBE Failure](../l1-runbooks/05-oobe-failure.md)

    8. **Language (Region)**: selection from list. What-breaks: If not set, user must select during OOBE — adds friction but no failure.

    9. **Automatically configure keyboard**: Yes or No. What-breaks: Requires ethernet (Wi-Fi not supported at this stage). If no ethernet and set to Yes, keyboard config fails silently.

    10. **Apply device name template**: Yes or No. What-breaks: Requires Entra join (not hybrid). Names must be <=15 characters. Supports `%SERIAL%` and `%RAND:x%` macros. Names exceeding 15 chars cause enrollment failure. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    11. **Convert all targeted devices to Autopilot**: Yes or No. What-breaks: Auto-registers non-Autopilot corporate devices with 48-hour processing delay. Unexpected device registrations if enabled broadly.

    ### Step 3: Assign profile to device group
    Assign to the dynamic device group created in [Dynamic Groups](04-dynamic-groups.md). Avoid "All Devices" assignment — exclusions not supported and causes assignment conflicts.

    **Profile conflict resolution callout:** Multiple profiles targeting same device: oldest created profile wins (not most specific, not highest priority). Audit creation dates when troubleshooting unexpected profile assignment. Runbook: [Profile Not Assigned](../l1-runbooks/03-profile-not-assigned.md)

    ### Step 4: Verify profile assignment
    Check Profile Status progression: Unassigned -> Assigning -> Assigned. Verify "Date assigned" field is populated before imaging the device.

    - [ ] Profile appears in Deployment Profiles list

    Title: `# Enrollment Status Page (ESP) Policy`

    ## Prerequisites:
    - Intune Administrator role
    - Deployment profile already created and assigned (see [Deployment Profile](02-deployment-profile.md))
    - App assignments planned (blocking app list should be finalized before configuring ESP)
    - Max 51 ESP profiles per tenant (1 default + 50 custom)

    ## Steps:

    ### Step 1: Create ESP profile
    Navigate to **Intune admin center** > **Devices** > **Enrollment** (Device onboarding) > **Windows** (tab) > **Windows Autopilot** > **Enrollment Status Page** > **Create**.

    ### Step 2: Configure ESP settings
    Document EVERY setting from the research table with what-breaks callouts:

    1. **Show app and profile configuration progress**: Yes or No. What-breaks: No = ESP doesn't appear during setup at all — user goes straight to desktop before apps finish installing. Runbook: [ESP Stuck or Failed](../l1-runbooks/02-esp-stuck-or-failed.md)

    2. **Show error when installation exceeds N minutes**: Default 60. What-breaks: Too low = ESP times out before large apps install; too high = user waits excessively on failed deployments. Recommend 60 minutes standard; increase for large app sets. **For hybrid join: add 40 minutes buffer** (connector creates AD object before ESP proceeds — a 60-min timeout means ~100-min actual wait). Runbook: [ESP Stuck or Failed](../l1-runbooks/02-esp-stuck-or-failed.md)

    3. **Show custom message on timeout/error**: Yes or No. What-breaks: No = generic "Setup could not be completed" message with no guidance.

    4. **Turn on log collection and diagnostics page**: Yes or No. Recommended: Yes. What-breaks: No = no collect-logs button shown and no Autopilot diagnostics page (Windows 11). Disabling makes L2 troubleshooting significantly harder.

    5. **Only show page to devices provisioned by OOBE**: Yes or No. What-breaks: No = ESP shown to every new user signing into the device, not just the first user. Causes confusion on shared devices.

    6. **Install Windows quality updates (might restart)**: Yes or No.

       **CRITICAL DEFAULT CHANGE callout (prominent, boxed):**
       - New ESP profiles default to **Yes** (installs monthly security updates during OOBE)
       - Existing ESP profiles default to **No**
       - This adds 20-40 minutes to provisioning and may cause restarts
       - Windows 11 only (not supported on Windows 10)
       - NOT supported in pre-provisioning Technician Flow (only applies during User Flow)
       - **Dependency:** `Block device use until all apps/profiles installed` must be **Yes** for this setting to take effect; if Block = No, device may exit ESP before updates apply

       What-breaks: Admin sees provisioning take 20-40 minutes longer than expected. End user sees unexpected restart during setup. Runbook: [ESP Stuck or Failed](../l1-runbooks/02-esp-stuck-or-failed.md)

    7. **Block device use until all apps/profiles installed**: Yes or No. What-breaks: No = users can exit ESP early, potentially reaching desktop before required apps are installed. Yes = users cannot proceed until all tracked apps install (or timeout).

    8. **Allow users to reset device if error occurs**: Yes or No (only visible when Block = Yes). What-breaks: No = user must contact IT on ESP failure. Yes = user can self-service reset but loses all progress.

    9. **Allow users to use device if error occurs**: Yes or No. What-breaks: Yes adds bypass button on error screen — user reaches desktop with incomplete setup.

    10. **Block device use until these required apps installed**: All or Selected. What-breaks: All = every assigned app blocks desktop access. Selected = up to 100 apps in blocking list. Choose Selected for large app catalogs to avoid excessive wait times.

    11. **Only fail selected blocking apps in technician phase**: Yes or No (pre-provisioning only). What-breaks: Yes = nonblocking app failures ignored during technician flow (reseal succeeds even if non-critical apps fail). No = any app failure causes technician flow to fail.

    ### Step 3: Configure app tracking
    **App type tracking reference table:**
    | App Type | Tracked (blocks desktop) | Notes |
    |----------|------------------------|-------|
    | Win32 (required) | Yes | Preferred app type for ESP |
    | LOB/MSI (required) | Yes | Do NOT mix with Win32 — causes TrustedInstaller conflict |
    | Microsoft Store (required) | Yes | |
    | PowerShell scripts (device) | Yes | |
    | Certificates (device) | Yes | |
    | Available assignment apps | No | Never blocks ESP |

    **CRITICAL WARNING callout:** Do NOT assign both LOB (MSI) and Win32 apps as required in the same ESP deployment. TrustedInstaller conflict causes ESP hang. Convert all apps to Win32 format, or use APv2 which supports mixed app types. What-breaks: ESP stuck at app installation; sidecar logs show "Another installation is in progress." Runbook: [ESP Stuck or Failed](../l1-runbooks/02-esp-stuck-or-failed.md)

    **Known issue callout:** Microsoft 365 Apps deployed as MSI type + Win32 apps simultaneously causes ESP hang. Deploy Microsoft 365 Apps as Win32 type instead.

    ### Step 4: Assign ESP profile
    Assign to the same device group as the deployment profile. Profile priority: if multiple ESP profiles apply, Intune uses the most specific (unlike deployment profiles which use oldest-wins).
