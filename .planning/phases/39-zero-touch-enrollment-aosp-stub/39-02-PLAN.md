## Scope and Status

> ⚠️ **This is a stub in v1.4. Full AOSP admin coverage is planned for v1.4.1.** See [Deferred Content](#deferred-content) below for what this stub does NOT cover. Per-OEM enrollment mechanics, AOSP user-associated vs userless distinctions, and AOSP-specific L1/L2 runbooks are deferred (see [deferred content table](#deferred-content)).

This guide establishes the scope, supported device classes, and enrollment constraints for AOSP in Intune at v1.4. It enumerates the OEM list from Microsoft's AOSP supported-devices page with RealWear highlighted as the confirmed GA case, and routes admins with non-RealWear hardware to the Android Enterprise fully managed path where Google Mobile Services is available.

## What AOSP Is

AOSP (Android Open Source Project) is the base Android operating system without Google Mobile Services (GMS). Devices running AOSP do NOT have:

- Google Mobile Services / Firebase Cloud Messaging (FCM) push notifications — Intune uses direct polling for device-management signals instead of push delivery.
- Managed Google Play (MGP) — AOSP devices cannot install Play Store apps; app deployment uses Intune-provided APK delivery mechanisms.
- Google account sign-in as a management handshake — AOSP corporate modes mark device ownership automatically during enrollment without Google identity.

In Intune, AOSP enrollment targets a distinct management surface from Android Enterprise (COBO/BYOD/Dedicated/ZTE). AOSP devices report via `intunecdnpeasd.manage.microsoft.com` rather than the GMS-path endpoints.

## When to Use AOSP

Use AOSP in Intune when the device hardware is in one of the supported classes documented on MS Learn:

- **AR/VR headsets** (DigiLens, HTC, Lenovo, Meta, PICO, RealWear, Vuzix)
- **Wearable scanners** (Zebra)

Do NOT use AOSP for mainstream Android phones or tablets. If the device has Google Mobile Services (GMS) — which includes virtually all consumer and enterprise Android phones/tablets — use an Android Enterprise mode instead (Fully Managed COBO, BYOD Work Profile, Dedicated, or Zero-Touch Enrollment). See [Android Enrollment Overview](../android-lifecycle/00-enrollment-overview.md) for the mode decision framework.

## Supported OEMs

The authoritative list of AOSP-supported devices is maintained by Microsoft at [MS Learn — AOSP Supported Devices](https://learn.microsoft.com/en-us/intune/fundamentals/aosp-supported-devices). Re-verify at plan time; the list is updated periodically.

## Enrollment Constraints

AOSP enrollment has three load-bearing constraints that differ from GMS-based Android Enterprise modes:

- **QR-only enrollment.** The only supported provisioning method for AOSP is QR code. NFC, DPC identifier (afw#setup), and Zero-Touch are NOT supported for AOSP enrollment. (Android 10+ per MS Learn aosp-supported-devices.)

  > **What breaks if misconfigured:** Attempting NFC, DPC identifier, or Zero-Touch on an AOSP device → enrollment fails silently or device falls through to consumer setup. Recovery: re-provision using QR code after factory reset.

- **One device at a time.** Bulk enrollment is not supported for AOSP. Each device scans its own QR code during its own enrollment session.

  > **What breaks if misconfigured:** Attempting bulk or queue-based enrollment → only the first device completes enrollment; subsequent devices require separate QR-scan sessions. Recovery: enroll each device individually with its own session.

- **Wi-Fi credential embedding (RealWear-specific).** RealWear devices cannot join Wi-Fi networks interactively during enrollment. The QR payload MUST include Wi-Fi credentials (SSID + password/EAP config) so the device comes up already network-connected. See [RealWear (confirmed GA)](#realwear-confirmed-ga) above for device-model specifics.

  > **What breaks if misconfigured:** Wi-Fi credentials NOT in QR payload on RealWear → device cannot reach Intune enrollment server after first boot; enrollment hangs or fails. Recovery: generate new QR payload with Wi-Fi SSID/password embedded; factory-reset and re-scan.
