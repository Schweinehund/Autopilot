    **1. `## Per-Cause Deep-Dive`** — Exact H2 heading (V-31-14 second match).

       `### Cause A: First-Evaluation Window (timing axis)`

       **Scope:** The 0-30 minute first-evaluation window post-enrollment. Compliance state is expected to transition from Not evaluated → Compliant/Non-compliant within 30 min of the device's first check-in after a policy is assigned.

       **Indicator:**
       - User completes ADE enrollment; home screen appears.
       - User opens Outlook/Teams within 0-30 minutes; sign-in fails with "Your device does not meet your organization's policy."
       - Intune admin center > Devices > [device] > Compliance shows **Not evaluated**.

       **Investigation steps:**
       1. Check tenant Default posture: Endpoint security > Device compliance > Compliance policy settings > "Mark devices with no compliance policy assigned as" — is it **Not compliant**? If so, devices without an assigned policy will be blocked by CA during the window.
       2. Check per-device compliance: Devices > [device] > Compliance — state Not evaluated? Timestamp of enrollment completion (Overview) vs now: < 30 min?
       3. Trigger Company Portal > Sync on device (or Intune remote Sync).
       4. Wait up to 30 minutes; recheck.

       **Resolution:**
       - State transitions to Compliant: timing gap was cause; no action needed except user communication.
       - State remains Not evaluated > 30 min with Default posture = Not compliant: proceed to §Cause C below.
       - State transitions to Non-compliant: proceed to §Cause B below.

       **Cross-link:** [Compliance Evaluation Timing and Conditional Access](../admin-setup-ios/06-compliance-policy.md#compliance-evaluation-timing-and-conditional-access) — Phase 28 D-11 anchor (verified line 149).

       `### Cause B: Policy Mismatch — OS Version / Jailbreak / Passcode / Restricted Apps (config axis + defects)`

       **Scope:** Device does not meet a specific compliance setting. Pareto coverage per D-15.

       **Investigation step (common to all B sub-settings):**
       1. Intune admin center > Devices > [device] > Compliance > [policy name] > expand to see PER-SETTING pass/fail.
       2. Identify which setting fails. Match against sub-sections below.

       **[TIMING sub-bullet]** If Intune admin center shows a policy modification in the last hour AND device hasn't checked in post-modification, this is [TIMING]: wait for next check-in.

       Followed by four compact sub-sections:

       `#### OS Version [CONFIG]`
       - Indicator: Device runs iOS < policy minimum.
       - Resolution: User updates via Settings > General > Software Update. If hardware cannot reach minimum, adjust policy OR move device to lower-tier policy group.
       - Cross-link: [OS version gates](../admin-setup-ios/06-compliance-policy.md#os-version-gates).

       `#### Jailbreak [CONFIG] + [DEFECT] fallback`
       - Indicator: Compliant on other settings; jailbreak setting fails. Device NOT actually jailbroken.
       - Typical cause: compromised APNs push delays OR outdated iOS version missing jailbreak-detection signal.
       - Resolution: verify actual jailbreak state (Settings > General > About — no unauthorized kernel info). Update iOS to current version.
       - **[DEFECT]** If updated OS still reports jailbreak → Microsoft Support (false-positives are real and tracked).
       - Cross-link: [Jailbreak detection](../admin-setup-ios/06-compliance-policy.md#jailbreak-detection).

       `#### Passcode [CONFIG]`
       - Indicator: Device has no passcode OR complexity < policy.
       - Resolution: User sets passcode via Settings > Face ID & Passcode (or Touch ID & Passcode) meeting complexity. Compliance updates on next check-in.
       - Cross-link: [Passcode](../admin-setup-ios/06-compliance-policy.md#passcode).

       `#### Restricted Apps by Bundle ID [CONFIG]`
       - Indicator: Policy lists restricted app by Bundle ID; device has it installed OR Intune reports device non-compliant despite app not present.
       - Typical cause: Bundle ID mismatch in policy (App Store Bundle ID vs LOB Bundle ID variant).
       - Resolution: Admin verifies exact Bundle ID via ABM or App Store listing → update policy → wait for check-in.
       - Cross-link: [Restricted apps](../admin-setup-ios/06-compliance-policy.md#restricted-apps).

       `### Cause C: Default Compliance Posture Stuck (config axis + genuine-defect APNs-blocked)`

       **Scope:** Device remains Not evaluated > 30 minutes (beyond Cause A window).

       **Indicator:**
       - Enrollment completed > 30 min ago (possibly hours or days).
       - Device compliance state still Not evaluated.
       - User sign-in to CA-protected resource blocked.

       **Investigation steps:**
       1. Verify device checked in: Devices > [device] > Overview > "Last check-in" — if > 8 hours, proceed to Sync.
       2. Trigger Intune Sync; wait 30 min; recheck.
       3. If still Not evaluated: verify APNs reachability from device network. On device: Settings > General > VPN & Device Management > Management Profile > More Details — does "Last management action" timestamp update after trigger?
       4. Verify tenant-wide APNs cert: Tenant administration > Connectors and tokens > Apple push notification certificate — not expired, not revoked.
       5. If all above pass: collect sysdiagnose via runbook 14 Tier 3 → examine kernel.log for `apsd`/`identityservicesd` failures to `*.push.apple.com` / `courier.push.apple.com`.

       **Resolution:**
       - Network path fix: whitelist Apple APNs endpoints at firewall/proxy. Apple reference: `https://support.apple.com/HT203609`.
       - If network confirmed open → proceed to §"Not evaluated" terminal state below (Microsoft Support escalation).

       `### "Not evaluated" Terminal State (cross-cutting [DEFECT] — D-16)`

       **Scope:** APNs-blocked at network edge leads to indefinite Not-evaluated. Content overlap with Cause C §Default posture stuck is intentional — "stuck" is a timing/config investigation; "terminal" is a network/cert investigation with Microsoft Support escalation.

       **Cross-platform APNs blast radius (Phase 27 D-11 reference):** A single expired Apple Push Notification certificate breaks MDM communication for **all Apple platforms simultaneously** (iOS, iPadOS, macOS). APNs-blocked-at-edge on one device is often a symptom of a broader tenant-level cert issue — verify the APNs cert state FIRST before per-device troubleshooting.

       **Microsoft Support escalation data (D-16 required data collection):**
       - APNs tenant ID / Apple ID from APNs cert pane (Intune admin center > Tenant administration > Connectors and tokens > Apple push notification certificate > "Apple ID" field)
       - APNs cert expiry timestamp
       - Network-path evidence: sysdiagnose `kernel.log` excerpt showing `apsd` connection failures to `*.push.apple.com` / `courier.push.apple.com`
       - Date range of Not-evaluated persistence (first observed → latest verified)
       - Device serial number + iOS version
